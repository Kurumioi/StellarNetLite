# 07. StellarNet Lite 实体同步与动画同步深度解析

本文档不再只讲 `ObjectSync` 是什么，而是直接回答下面几个新手最关心的问题：

1. 怎么从 0 做出一个能在房间里生成的网络同步实体。
2. 这个实体必须挂什么组件、放到哪里、怎么被扫描到。
3. 动画同步要怎么配，`Animator` 里哪些状态和参数需要手动对应。
4. 生成失败、不同步、动画不播时，应该先查哪里。

如果你希望“照着文档一步一步做，最后真的能在房间里看到同步实体和同步动画”，这篇就是给你准备的。

---

## 1. 先建立正确认知

`ObjectSync` 不是完整玩法系统，它只负责“网络实体状态同步基础设施”。

它主要负责：

- 服务端创建实体。
- 服务端销毁实体。
- 把实体的 Transform 状态同步给客户端。
- 把实体的动画状态和最多 3 个浮点动画参数同步给客户端。
- 在重连和回放时导出 / 恢复这些实体状态。

它不负责：

- 你的角色输入逻辑。
- 你的战斗判定逻辑。
- 你的 AI 决策逻辑。
- 你的 UI 展示逻辑。

你应该把它理解为：

- 服务端维护权威实体状态。
- 客户端根据同步状态生成视图对象并做平滑表现。

---

## 2. 先看完整链路

当前仓库里，一个同步实体从服务端到客户端，大致会经过下面这条链：

1. 你在 Unity 里做一个 Prefab。
2. 这个 Prefab 根节点挂上 `NetIdentity`。
3. 运行 `StellarNetLite/生成网络预制体常量表`。
4. `NetPrefabScanner` 扫描所有挂了 `NetIdentity` 的 Prefab，生成 `Assets/StellarNetLiteGenerated/NetPrefabConsts.cs`。
5. 服务端房间组件拿到 `ServerObjectSyncComponent`，调用 `SpawnObject(...)` 创建实体。
6. `ServerObjectSyncComponent` 广播 `S2C_ObjectSpawn`。
7. 客户端 `ClientObjectSyncComponent` 收到后先缓存实体状态。
8. 场景里的 `ObjectSpawnerView` 监听到 `S2C_ObjectSpawn`，根据 `PrefabHash` 去解析 Prefab。
9. `ObjectSyncClientRuntime.DefaultNetViewBinder` 给实例绑定：
   - `NetIdentity`
   - `NetTransformView`
   - `NetAnimatorView`
10. 后续服务端周期广播 `S2C_ObjectSync`。
11. 客户端 `ClientObjectSyncComponent` 更新缓存。
12. `NetTransformView` 和 `NetAnimatorView` 每帧从缓存里取预测后的同步结果，驱动实际表现。

如果这条链上任何一环没接好，就会出现：

- 对象生成不出来。
- 能生成但不动。
- 会动但动画不播。
- 动画状态切换错乱。
- 重连后对象不恢复。

所以真正排查问题时，一定要沿这条链逐段看，而不是只盯着某一个脚本。

## 4. 从 0 制作一个网络同步实体

这一节是最重要的实操部分。你如果只想先做出来，再回头理解原理，可以直接照下面做。

### 4.1 第一步：创建一个网络实体 Prefab

先做一个普通 Unity Prefab。

建议：

- 根节点就是你的实体主对象。
- 模型、Animator、CharacterController 等作为它的子节点或同级组件。

当前默认扫描规则要求：

- 只有“Prefab 根节点挂了 `NetIdentity`”的预制体，才会被 `NetPrefabScanner` 识别为网络预制体。

这意味着：

- 如果你把 `NetIdentity` 只挂在子节点上，默认扫描不到。
- 扫描不到就不会生成 `PrefabHash` 常量。
- 没有常量就不能稳定地 `SpawnObject(...)`。

所以最稳妥的做法是：

- 直接在 Prefab 根节点挂 `NetIdentity`。

### 4.2 第二步：把 Prefab 放到推荐路径

当前默认客户端预制体解析器是：

- `ObjectSyncClientRuntime.ResourcesNetPrefabResolver`

它依赖：

- `NetPrefabConsts.HashToPathMap`

而 `HashToPathMap` 中只有位于 `Resources` 目录下的 Prefab，才会有可直接 `Resources.Load` 的路径映射。

推荐放法：

- `Client/Resources/NetPrefabs/你的Prefab.prefab`

例如样例里：

- `Samples/SocialDemo/Client/Resources/NetPrefabs/SocialPlayer.prefab`

这样扫描后会生成类似路径：

- `NetPrefabs/SocialPlayer`

注意：

- 不在 `Resources` 下也能被扫描出 hash。
- 但默认 `ResourcesNetPrefabResolver` 无法通过 `Resources.LoadAsync` 加载它。
- 这会导致客户端收到生成消息后，`ObjectSpawnerView` 报 “默认 Resources 解析器未找到 PrefabHash 对应路径映射”。

所以如果你不打算自定义 `INetPrefabResolver`，就把网络实体 Prefab 放进 `Resources`。

### 4.3 第三步：给实体决定同步内容

`SpawnObject(...)` 需要传一个 `EntitySyncMask`。

常见选择：

- `EntitySyncMask.Transform`
- `EntitySyncMask.Animator`
- `EntitySyncMask.All`

推荐理解方式：

- 只同步位移旋转缩放：用 `Transform`
- 只同步动画：理论可行，但大多数情况下没有意义
- 同步位移 + 动画：用 `All`

如果是一个玩家、NPC、怪物、载具，通常用：

- `EntitySyncMask.All`

如果只是一个移动平台、投射物、机关，不需要动画，通常用：

- `EntitySyncMask.Transform`

### 4.4 第四步：给需要同步位移的实体准备 `NetTransformView`

如果实体会同步 Transform，建议在 Prefab 上直接挂：

- `NetTransformView`

虽然默认绑定器会在运行时自动补挂，但我仍然建议你预先挂上，原因有两个：

1. Inspector 中可以直接调参数。
2. 新人更容易一眼看出“这个 Prefab 本来就是网络同步实体”。

这个组件负责：

- 读取 `ClientObjectSyncComponent` 中缓存的预测位置。
- 做平滑插值、追赶和瞬移校正。

关键参数：

- `PosSmoothTime`
  远端位置平滑时间。
- `RotSmoothSpeed`
  远端旋转平滑速度。
- `SnapThreshold`
  当前位置和目标位置相差太大时直接瞬移。
- `CatchUpThreshold`
  偏差较大时加快追赶。
- `StopThreshold`
  接近目标且速度接近 0 时直接落点。

如果对象还挂了 `CharacterController`，`NetTransformView` 内部会在回写位置时临时关闭再开启 `CharacterController`，避免直接赋值时报冲突。

### 4.5 第五步：给需要同步动画的实体准备 `NetAnimatorView`

如果实体需要同步动画，建议在 Prefab 上直接挂：

- `NetAnimatorView`

并且至少完成这些配置：

1. `TargetAnimator`
   指向你要驱动的 `Animator`。
2. `SyncedStateNames`
   填入服务端可能发过来的逻辑状态名。
3. `FloatParam1Name / FloatParam2Name / FloatParam3Name`
   如果你确实使用了对应浮点参数，就把参数名填上。

这一步非常重要，因为当前动画同步不是“把 Unity Animator 整个复制到网络上”，而是：

- 服务端发送逻辑动画状态 hash。
- 客户端 `NetAnimatorView` 再把逻辑状态 hash 映射为 Unity Animator 状态 hash。

也就是说，服务端和客户端之间依赖的是“状态名约定”，不是 Animator Controller 文件本身。

例如：

- 服务端逻辑认为状态名叫 `Idle`
- 客户端 `SyncedStateNames` 里也必须包含 `Idle`
- Animator Controller 中也必须存在名为 `Idle` 的状态

如果这三者对不上，就会出现：

- 对象生成了。
- 位置同步正常。
- 动画就是不播，或者总回到默认状态。

### 4.6 第六步：理解 `SyncedStateNames` 到底怎么填

`NetAnimatorView` 里有一个特别关键的字段：

- `SyncedStateNames`

它的作用不是给你展示提示文字，而是构建：

- 服务端逻辑状态 hash -> Unity Animator 状态 hash

当前映射规则是：

1. 对每个状态名，例如 `Idle`
2. 客户端用稳定字符串 hash 算出服务端逻辑 hash
3. 再用 `Animator.StringToHash("Idle")` 算出 Unity 状态 hash
4. 建立两者映射

所以你必须保证：

- 服务端使用的状态字符串，和客户端 `SyncedStateNames` 中的字符串完全一致。

例如 `SocialDemo` 里服务端使用的是：

- `Idle`
- `Walk`
- `Wave`
- `Dance`

那客户端对应实体的 `SyncedStateNames` 里也必须有：

- `Idle`
- `Walk`
- `Wave`
- `Dance`

大小写、空格、拼写都不能乱改。

### 4.7 第七步：如果动画用到 BlendTree，配置浮点参数映射

`NetAnimatorView` 最多支持 3 个浮点参数同步：

- `FloatParam1`
- `FloatParam2`
- `FloatParam3`

客户端不会知道这 3 个参数在 Animator 里具体叫什么，所以你必须在 Inspector 填：

- `FloatParam1Name`
- `FloatParam2Name`
- `FloatParam3Name`

例如：

- `FloatParam1Name = MoveSpeed`
- `FloatParam2Name = Turn`
- `FloatParam3Name = Emotion`

如果你没填：

- 服务端即使同步了参数值，
- 客户端也不会把它写进 Animator。

如果你的动画只靠状态切换，不靠 BlendTree 参数：

- 可以不填这 3 个名字。

### 4.8 第八步：生成网络预制体常量表

做完 Prefab 后，执行菜单：

- `StellarNetLite/生成网络预制体常量表`

它会扫描所有挂了 `NetIdentity` 的 Prefab，并输出：

- `Assets/StellarNetLiteGenerated/NetPrefabConsts.cs`

你后面服务端生成对象时，通常就直接引用这里的常量，例如：

```csharp
private const int PlayerPrefabHash = NetPrefabConsts.NetPrefabs_SocialPlayer;
```

如果这一步没做，你最常见会遇到的现象是：

- `PrefabHash` 还是 0
- 服务端 `SpawnObject(...)` 直接失败或返回 null
- 客户端没法解析资源

### 4.9 第九步：确保房间真的装配了 `ObjectSync`

`ObjectSync` 不是全局默认永远存在的，它是一个房间组件：

- 组件 Id 是 `200`
- 组件名是 `ObjectSync`

你创建房间时，必须把它放进房间组件列表，否则下面所有对象同步逻辑都不会存在。

当前样例 `RoomTypeTemplateRegistry` 的模板写法就是：

```csharp
ComponentIds = new[]
{
    ComponentIdConst.SocialRoom,
    ComponentIdConst.RoomSettings,
    ComponentIdConst.ObjectSync
}
```

所以你自己的房间模板里，如果希望使用同步实体，必须包含：

- `ComponentIdConst.ObjectSync`

否则：

- 服务端拿不到 `ServerObjectSyncComponent`
- 客户端拿不到 `ClientObjectSyncComponent`
- `ObjectSpawnerView.Init(room)` 也会失败

### 4.10 第十步：服务端实际生成实体

服务端生成实体的标准入口是：

- `ServerObjectSyncComponent.SpawnObject(...)`

最小示例：

```csharp
ServerObjectSyncComponent syncService = Room.GetComponent<ServerObjectSyncComponent>();
if (syncService == null)
{
    return;
}

ServerSyncEntity entity = syncService.SpawnObject(
    NetPrefabConsts.NetPrefabs_SocialPlayer,
    EntitySyncMask.All,
    new Vector3(0f, 0f, 0f),
    Vector3.zero,
    Vector3.zero,
    ownerSessionId: session.SessionId);

if (entity == null)
{
    return;
}

entity.AnimStateHash = GetStableStringHash("Idle");
entity.AnimNormalizedTime = 0f;
```

生成后你还要继续维护这个实体的状态，例如：

- `entity.Position`
- `entity.Rotation`
- `entity.Velocity`
- `entity.Scale`
- `entity.AnimStateHash`
- `entity.AnimNormalizedTime`
- `entity.FloatParam1~3`

服务端不会替你推导玩法状态，它只会把你填进去的状态同步出去。

### 4.11 第十一步：客户端场景里必须有 `ObjectSpawnerView`

很多人只做了服务端生成，忘了客户端视图生成入口，结果会出现：

- 报文收到了。
- `ClientObjectSyncComponent` 状态也更新了。
- 场景里还是没有任何对象。

原因通常是：

- 当前房间没有 `ObjectSpawnerView`
- 或者 `ObjectSpawnerView` 没有对当前房间执行 `Init(room)`

样例 `RoomViewManager` 的做法是：

1. 进房成功后创建一个房间表现层根节点。
2. 检查房间是否挂了 `ClientObjectSyncComponent`。
3. 如果有，就在表现层根节点上 `AddComponent<ObjectSpawnerView>()`。
4. 调用 `spawner.Init(room)`。

也就是说，如果你自己做新项目：

- 你必须有一处类似 `RoomViewManager` 的初始化逻辑。

### 4.12 第十二步：验证最小可运行结果

跑起来以后，最小验证应该至少看到下面几个结果：

1. 进入房间后客户端场景里出现该实体。
2. 服务端修改 `entity.Position` 后，客户端对象位置会变化。
3. 服务端切 `entity.AnimStateHash` 后，客户端动画会切状态。
4. 服务端改 `entity.FloatParam1~3` 后，对应 BlendTree 参数会更新。

如果这 4 步都成立，说明你的同步实体链路已经通了。

## 5. 一个最小可工作的同步实体清单

如果你只想先做一个最简单的“能跑”的版本，可以照着这个最小清单来：

### 5.1 Prefab 侧

- 一个放在 `Resources` 下的 Prefab
- 根节点挂 `NetIdentity`
- 根节点挂 `NetTransformView`
- 如果要同步动画，再挂 `NetAnimatorView`
- `NetAnimatorView.TargetAnimator` 指向正确 Animator
- `NetAnimatorView.SyncedStateNames` 填好所有逻辑状态名

### 5.2 服务端侧

- 房间组件列表中包含 `ComponentIdConst.ObjectSync`
- 能在房间组件里拿到 `ServerObjectSyncComponent`
- 调用 `SpawnObject(...)`
- Tick 或业务消息处理中持续更新 `ServerSyncEntity`

### 5.3 客户端侧

- 房间组件列表中包含 `ObjectSync`
- 当前房间创建后能拿到 `ClientObjectSyncComponent`
- 有 `ObjectSpawnerView`
- `ObjectSpawnerView.Init(room)` 成功执行

### 5.4 编辑器侧

- 跑过 `StellarNetLite/生成网络预制体常量表`
- 必要时也跑 `StellarNetLite/强制重新生成协议与组件常量表`

## 6. 服务端同步原理

理解原理后，你在设计同步策略时就不容易乱。

### 6.1 服务端实体模型

服务端用 `ServerSyncEntity` 表示一个同步实体。

核心字段包括：

- `NetId`
- `PrefabHash`
- `Mask`
- `OwnerSessionId`
- `Position`
- `Rotation`
- `Velocity`
- `Scale`
- `AnimStateHash`
- `AnimNormalizedTime`
- `FloatParam1`
- `FloatParam2`
- `FloatParam3`

可以把它理解成：

- “服务端权威实体状态”

### 6.2 生成时做了什么

`SpawnObject(...)` 调用后会：

1. 分配新的 `NetId`
2. 创建 `ServerSyncEntity`
3. 放进 `_entities`
4. 记录一份 `_lastSentSnapshots`
5. 广播 `S2C_ObjectSpawn`

也就是说，客户端第一次看到对象，不是靠 `S2C_ObjectSync`，而是靠：

- `S2C_ObjectSpawn`

### 6.3 周期同步时做了什么

`ServerObjectSyncComponent.OnTick()` 会周期性检查所有实体。

它不会每 Tick 无脑全量广播，而是带了三层控制：

1. 广播间隔控制
2. 脏位检测
3. 周期全量校正

#### 6.3.1 广播间隔

受这些配置影响：

- `ObjectSyncOnlineIntervalTicks`
- `ReplayObjectSyncRecordIntervalTicks`
- `ObjectSyncFullResyncIntervalTicks`

#### 6.3.2 自适应降频

如果 `EnableAdaptiveObjectSync = true`：

- 人少时按基础频率同步
- 人多时自动降低同步频率，减轻广播压力

#### 6.3.3 脏位同步

服务端会对比当前实体状态与上次发送快照：

- 没变化就不发
- 只发变了的字段

脏位包括：

- Position
- Rotation
- Velocity
- Scale
- AnimState
- AnimNormalizedTime
- FloatParam1
- FloatParam2
- FloatParam3

#### 6.3.4 周期全量重同步

即使平时只发脏位，系统也会按配置做周期全量校正，避免长时间漂移。

## 7. 客户端同步原理

### 7.1 客户端先缓存，不直接改场景对象

客户端收到：

- `S2C_ObjectSpawn`
- `S2C_ObjectDestroy`
- `S2C_ObjectSync`

以后，不会立刻直接去改场景对象，而是先由：

- `ClientObjectSyncComponent`

更新内部缓存。

这样做的好处是：

- 逻辑层和表现层解耦
- 视图层可以做预测和平滑
- 回放和在线房间可以复用一套数据结构

### 7.2 `ObjectSpawnerView` 只负责“把缓存实体变成场景对象”

`ObjectSpawnerView` 做的事是：

1. 监听 `S2C_ObjectSpawn`
2. 根据 `PrefabHash` 找 Prefab
3. 异步加载资源
4. Instantiate
5. 通过 `DefaultNetViewBinder` 把网络视图组件绑定上去

它还会在初始化时调用：

- `BootstrapExistingEntities()`

也就是：

- 如果房间里在你初始化之前已经有对象，它也会把已有缓存实体补建出来

### 7.3 `DefaultNetViewBinder` 自动补哪些组件

默认绑定器会：

1. 确保对象上有 `NetIdentity`
2. 如果 `Mask` 包含 `Transform`，确保有 `NetTransformView`
3. 如果 `Mask` 包含 `Animator`，确保有 `NetAnimatorView`
4. 调用对应 `HardSetInitialState(...)`
5. 根据 `OwnerSessionId == 当前 SessionId` 设置 `IsLocalPlayer`

这就是为什么即使 Prefab 上没预挂 `NetTransformView` / `NetAnimatorView`，运行时也可能“看起来能工作”。

但注意：

- 能自动补挂，不等于能自动配好参数。
- 特别是 `NetAnimatorView` 里的状态名和参数名，仍然需要你自己在 Prefab 上配置。

## 8. 动画同步原理

### 8.1 当前同步的不是完整 Animator

当前系统同步的是：

- `AnimStateHash`
- `AnimNormalizedTime`
- `FloatParam1`
- `FloatParam2`
- `FloatParam3`

而不是：

- Unity Animator Controller 的全部内部状态

好处是：

- 更可控
- 更省带宽
- 更容易回放
- 方便服务端完全用纯 C# 驱动逻辑动画状态

### 8.2 为什么服务端用“稳定字符串 hash”

服务端并不依赖 Unity `Animator.StringToHash`，而是用自己的稳定字符串 hash 规则。

这样做的原因是：

- 服务端可以完全脱离 Unity Animator 运行
- 状态同步逻辑更稳定
- 回放和服务端逻辑更好统一

客户端收到这个 hash 后，再通过 `SyncedStateNames` 建好的映射表，转成真正的 Animator 状态 hash。

### 8.3 `NetAnimatorView` 是怎么切动画的

`NetAnimatorView` 每帧会：

1. 从 `ClientObjectSyncComponent` 读取预测后的动画数据
2. 把服务端状态 hash 转换成 Animator 状态 hash
3. 必要时执行 `CrossFadeInFixedTime(...)`
4. 把浮点参数平滑写回 Animator
5. 根据回放倍率或视觉追赶需求调整 `Animator.speed`

### 8.4 防滑冰补偿是干什么的

`NetAnimatorView` 里有：

- `EnableAntiIceSkating`
- `AntiIceTargetStates`
- `AntiIceSourceStates`

它的主要作用是：

- 避免网络延迟下对象还在位移，但动画已经提前切到 `Idle`

例如默认常见配置：

- 来源状态：`Walk`
- 目标状态：`Idle`

如果目标想切 `Idle`，但当前位置和同步目标仍然显示对象在移动，就暂时不切，视觉上会更自然。

## 9. 本地拥有者与远端对象的差异

### 9.1 Transform 表现差异

`NetTransformView` 对本地拥有对象和远端对象处理不完全一样。

对于远端对象：

- 会按预测结果做平滑追赶和校正。

对于本地拥有对象：

- 在线房间状态下直接跳过常规网络和解。

原因是当前样例采用的是：

- 本地先真实驱动移动
- 服务端做轻量校验
- ObjectSync 做周期校正与回放支撑

如果本地玩家也每帧被网络同步结果强行回写，就容易和 `CharacterController` / 输入控制同时驱动位置，出现拉扯。

### 9.2 Animator 表现差异

`NetAnimatorView` 对本地拥有对象在在线模式下同样会跳过网络回写。

意思是：

- 本地玩家在线时，动画更偏向本地输入驱动
- 远端玩家才主要依赖网络同步结果表现

## 10. `SocialDemo` 当前采用的同步策略

`SocialDemo` 并不是“完全只靠 ObjectSync 自动同步一切”。

它采用的是：

- 业务即时同步 + ObjectSync 周期兜底

链路如下：

1. 客户端本地真实移动和触发动作。
2. 客户端把最新结果发给服务端。
3. 服务端做轻量合法性校验。
4. 服务端立即广播业务同步消息：
   - `S2C_SocialStateSync`
5. 同时 `ServerSyncEntity` 的状态也会持续被 `ObjectSync` 周期同步和录制。

这样做的目的：

- 保留即时手感
- 不放弃服务端收敛
- 让回放和重连仍然能复用同一套对象同步状态

所以你做业务时要想清楚：

- 是只靠 `ObjectSync` 周期同步就够
- 还是需要像 `SocialDemo` 一样再加一层“即时业务广播”

## 11. 从零制作同步实体时最容易漏掉的点

### 11.1 漏挂 `NetIdentity`

后果：

- `NetPrefabScanner` 不会收录
- `NetPrefabConsts` 不会生成对应常量

### 11.2 Prefab 不在 `Resources`

后果：

- 默认 `ResourcesNetPrefabResolver` 找不到实际资源

### 11.3 做了 Prefab 但没重新生成 `NetPrefabConsts`

后果：

- `PrefabHash` 常量不存在
- 服务端拿到的 hash 还是旧值或 0

### 11.4 房间模板没加 `ObjectSync`

后果：

- 服务端没有 `ServerObjectSyncComponent`
- 客户端没有 `ClientObjectSyncComponent`

### 11.5 场景里没有 `ObjectSpawnerView`

后果：

- 收到生成消息也不会真正实例化对象

### 11.6 `NetAnimatorView.TargetAnimator` 没指对

后果：

- 状态和参数同步都不会真正驱动目标 Animator

### 11.7 `SyncedStateNames` 没填全

后果：

- 某些动画状态服务端明明发了
- 客户端解析不到，日志里会看到未知状态 hash 警告

### 11.8 服务端状态名和客户端状态名不一致

例如服务端写的是：

- `Walk`

但客户端 `SyncedStateNames` 填的是：

- `Run`

后果：

- 客户端根本映射不到这个状态

## 12. 调试与排错顺序

如果同步实体工作异常，建议严格按这个顺序查。

### 12.1 第一步：先看对象是否真的被服务端生成

确认：

- `SpawnObject(...)` 是否真的执行到
- 返回的 `ServerSyncEntity` 是否不为 null
- `PrefabHash` 是否非 0

### 12.2 第二步：看 `NetPrefabConsts` 是否包含该预制体

打开：

- `Assets/StellarNetLiteGenerated/NetPrefabConsts.cs`

确认：

- 有这个 Prefab 的常量
- 如果走默认加载器，`HashToPathMap` 里也有对应资源路径

### 12.3 第三步：看房间是否真的装了 `ObjectSync`

确认：

- 服务端 `Room.GetComponent<ServerObjectSyncComponent>()` 不为 null
- 客户端 `room.GetComponent<ClientObjectSyncComponent>()` 不为 null

### 12.4 第四步：看 `ObjectSpawnerView.Init(room)` 是否执行成功

如果失败，最常见原因是：

- `room == null`
- 缺 `ClientObjectSyncComponent`

### 12.5 第五步：看客户端是否收到 `S2C_ObjectSpawn`

如果 `ClientObjectSyncComponent.OnS2C_ObjectSpawn(...)` 都没命中：

- 先查服务端是否广播成功
- 再查房间消息分发链路

### 12.6 第六步：看 `PrefabHash` 是否能解析成 Prefab

如果 `ObjectSpawnerView` 报：

- 未配置 `PrefabResolver`
- 找不到 `PrefabHash` 对应路径
- `Resources/...` 不存在或不是 `GameObject`

那就是资源解析链路有问题。

### 12.7 第七步：看 Transform 同步是否被正确写入

确认：

- `Mask` 包含 `EntitySyncMask.Transform`
- `TryGetTransformData(...)` 能返回 true
- 对象不是本地拥有者却期望它被远端同步驱动

### 12.8 第八步：看动画状态映射是否正确

确认：

- `Mask` 包含 `EntitySyncMask.Animator`
- `TargetAnimator` 不为空
- `SyncedStateNames` 包含所有服务端状态名
- `FloatParam1Name~3Name` 与 Animator 参数名一致

## 13. 高负载与生产环境下的理解建议

这套对象同步系统在设计上已经考虑了生产环境常见压力点：

- 不是每 Tick 全量广播
- 有脏位控制
- 有按人数自适应降频
- 有周期全量校正
- 客户端不直接硬回写，而是做预测和平滑

但你在真正项目里仍然要注意：

1. 不要把所有业务状态都塞进 `ObjectSync`
2. `ObjectSync` 更适合承载“需要跨端一致的基础表现状态”
3. 需要更强实时性的玩法，最好像 `SocialDemo` 一样叠加业务即时同步
4. 动画状态命名要稳定，不要频繁改名
5. Prefab 扫描与生成流程要纳入日常开发习惯
