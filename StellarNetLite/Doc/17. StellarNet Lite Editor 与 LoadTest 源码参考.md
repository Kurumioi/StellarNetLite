# 17. StellarNet Lite Editor 与 LoadTest 源码参考

本文档覆盖两块：

- `StellarNetLite/Editor`
- `StreamingAssets/StellarNetLiteLoadTest`

这两块虽然不参与正式运行时主链，但它们决定了：

- 框架声明如何变成静态注册代码
- 配置如何被维护
- 新模块如何快速落地
- 服务端如何在编辑器里观测
- 高负载下如何做压测与复现

---

## 1. Editor 总览

`Editor` 层可以按职责拆成四类：

- 生成器
  `LiteProtocolScanner`、`NetPrefabScanner`
- 配置器
  `NetConfigEditorWindow`
- 脚手架与辅助工具
  `StellarNetScaffoldWindow`、`FolderContentCopyTool`
- 运行期运维工具
  `ServerMonitorWindow`、`LoadTestWindow`

## 2. 协议与绑定生成器

### 2.1 `Editor/LiteProtocolScanner.cs`

文件职责：

- 编辑器里最核心的扫描器。
- 把源码中声明的 `[NetMsg]`、`[RoomComponent]`、`[ServerModule]`、`[ClientModule]`、`[NetHandler]` 转成 Runtime 真正依赖的静态注册代码。

它生成的核心文件包括：

- `Assets/StellarNetLiteGenerated/MsgIdConst.cs`
- `Assets/StellarNetLiteGenerated/ComponentIdConst.cs`
- `Assets/StellarNetLiteGenerated/AutoMessageMetaRegistry.cs`
- `Assets/StellarNetLiteGenerated/AutoUnauthenticatedProtocolRegistry.cs`
- `Assets/StellarNetLiteGenerated/AutoRegistry.cs`
- `Assets/StellarNetLiteGenerated/AutoRuntimeFeatureRegistry.cs`

核心常量路径：

- `GeneratedRootPath`
- `ProtocolIdsFolderPath`
- `ProtocolMetaFolderPath`
- `ComponentConstFolderPath`
- `BinderRoomFolderPath`
- `BinderModuleFolderPath`
- `MsgIdAggregateOutputPath`
- `ComponentAggregateOutputPath`
- `MessageMetaAggregateOutputPath`
- `UnauthenticatedProtocolRegistryOutputPath`
- `RegistryAggregateOutputPath`
- `RuntimeFeatureRegistryOutputPath`

内部类型：

- `ScanResultState`
  - `NoChange`
  - `Changed`
  - `Failed`
- `MethodMeta`
  单个处理函数扫描结果：
  - `MethodName`
  - `MsgFullName`
  - `MsgId`
  - `FeatureKey`
- `ClassMeta`
  单个模块/组件扫描结果：
  - `Id`
  - `Name`
  - `SafeVarName`
  - `FullName`
  - `DisplayName`
  - `FeatureKey`
  - `Methods`
- `ProtocolMeta`
  单个协议扫描结果：
  - `Id`
  - `SafeName`
  - `FullTypeName`
  - `Scope`
  - `Dir`
  - `AllowUnauthenticated`
  - `FeatureKey`
- `GeneratedFileRecord`
  记录待写入文件：
  - `Path`
  - `Content`

关键入口：

- 静态构造函数
  编辑器加载时自动触发一次扫描。
- `ManualRun()`
  手动重新生成。
- `ForceRun()`
  菜单强制重新生成。
- `RunScanAndGenerate(bool forceWrite)`
  总调度入口：
  - 生成协议常量与元数据
  - 生成组件/模块绑定器
  - 生成运行时扩展桥注册表

这份脚本的意义：

- 没有它，`Runtime` 根本不知道有哪些协议、模块、组件和处理器。
- 这是整个框架从“反射声明”转向“静态注册”的关键。

### 2.2 `Editor/NetPrefabScanner.cs`

文件职责：

- 扫描挂有 `NetIdentity` 的 Prefab。
- 生成稳定的 `PrefabHash` 常量表，供 `ObjectSync` 使用。

关键常量：

- `OutputConstFilePath`
  输出到 `Assets/StellarNetLiteGenerated/NetPrefabConsts.cs`。

关键函数：

- `ManualRun`
  菜单触发手动扫描。
- `RunScanAndGenerate`
  遍历项目 Prefab，筛选符合约定的网络预制体。
- `GetResourcesRelativePath`
  解析 `Resources` 相对路径。
- `GetHashSourceKey`
  生成 hash 输入源，保证稳定性。
- `GetStableHash`
  计算稳定 hash。
- `GenerateValidVariableName`
  把路径/名称转成可用常量名。
- `GenerateConstFile`
  写出常量表。

扫描时重点检查两类冲突：

- `Hash` 冲突
- 变量名冲突

## 3. 配置编辑器

### 3.1 `Editor/NetConfigEditorWindow.cs`

文件职责：

- 统一编辑三套配置：
  - `NetConfig`
  - `ReplayGlobalConfig`
  - `ObjectSyncGlobalConfig`
- 但保存时分别落到独立文件。

核心字段：

- `_runtimeConfig`
  运行时网络配置。
- `_replayConfig`
  回放配置。
- `_objectSyncConfig`
  对象同步配置。
- `_targetRoot`
  当前编辑的是 `StreamingAssets` 还是 `PersistentDataPath` 下的配置。
- `_loadedSnapshotJson`
  加载时的快照，用于判断是否脏。
- `_runtimePath` / `_replayPath` / `_objectSyncPath`
  当前解析出来的三个目标文件路径。
- `_scrollPos`
  滚动位置。
- `_headerStyle` / `_pathStyle`
  UI 样式缓存。
- `_stylesInitialized`
  样式初始化标记。

内部枚举：

- `SaveResultState`
  - `NoChange`
  - `Saved`
  - `Failed`

关键函数：

- `ShowWindow`
  打开配置窗口。
- `OnEnable`
  根据当前根目录加载配置。
- `OnGUI`
  绘制整个配置窗口。
- `InitializeStyles`
  初始化样式。
- `DrawToolbar`
  切换运行时根目录、确认脏数据切换。
- `DrawResolvedPathInfo`
  显示当前文件路径。
- `DrawRuntimeFields`
  绘制 `NetConfig` 字段。
- `DrawReplayFields`
  绘制回放配置字段。
- `DrawObjectSyncFields`
  绘制对象同步配置字段。
- `DrawActionButtons`
  保存、重载、切换根目录等操作。
- `LoadFromCurrentRoot`
  从当前根目录同步加载三套配置。

这个窗口的重要性：

- 它把原本散落的多份 JSON 配置统一到一个入口。
- 也是当前框架“配置分拆保存”之后的主要维护面板。

## 7. 维护建议

如果你改了协议声明、组件声明、模块声明或运行时扩展桥：

1. 先看 `LiteProtocolScanner.cs`
2. 再重新生成 `StellarNetLiteGenerated`
3. 最后检查 `Runtime` 是否已经通过 `GeneratedRuntimeAccess` / `AutoRegistry` 消费到新内容

如果你要排查“压测和真实客户端表现不一致”：

1. 先看 `NetworkPrimitives.cs` 的协议 ID 与 DTO 是否和正式协议同步
2. 再看 `Transports.cs` 是否只保留了控制面包
3. 再看 `Program.cs` 中机器人是否模拟了与你当前玩法一致的业务顺序
