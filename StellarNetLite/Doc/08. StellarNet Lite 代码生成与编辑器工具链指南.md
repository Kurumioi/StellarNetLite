# 08. StellarNet Lite 代码生成与编辑器工具链指南

本文说明当前仓库里的 Editor 工具链，重点回答 3 个问题：

1. 哪些代码是手写业务代码，哪些代码是 Editor 扫描后生成的。
2. 生成物在哪里，什么时候要重新生成。
3. 每个 Editor 工具分别解决什么问题。

---

## 1. 这套工具链在解决什么问题

StellarNet Lite 当前不是“运行时反射扫描一切”的结构。

当前做法是：

1. 业务层手写协议、模块、房间组件、预制体。
2. Editor 扫描这些声明式代码和资源。
3. 生成静态常量、静态注册表、静态绑定器。
4. Runtime 启动时直接使用生成结果。

这样做的意义是：

- 启动链更稳定。
- 协议和组件注册更集中。
- 运行时不依赖大量反射扫描。
- 新业务只要遵守标记规则，就能接入自动装配。

---

## 2. 当前有哪些生成目录

当前生成根目录是：

- `Assets/StellarNetLiteGenerated`

主要生成物包括：

- `Assets/StellarNetLiteGenerated/MsgIdConst.cs`
- `Assets/StellarNetLiteGenerated/ComponentIdConst.cs`
- `Assets/StellarNetLiteGenerated/AutoMessageMetaRegistry.cs`
- `Assets/StellarNetLiteGenerated/AutoRegistry.cs`
- `Assets/StellarNetLiteGenerated/NetPrefabConsts.cs`

分片目录包括：

- `Assets/StellarNetLiteGenerated/MsgIds`
- `Assets/StellarNetLiteGenerated/Meta`
- `Assets/StellarNetLiteGenerated/Components`
- `Assets/StellarNetLiteGenerated/RoomBinders`
- `Assets/StellarNetLiteGenerated/ModuleBinders`

要点：

- 当前仓库默认不提交这些生成物。
- 新机器拉下项目后，需要在 Unity Editor 里手动重新生成。
- 如果你只看 Runtime，看不到这些文件，不代表框架缺文件。

## 3. LiteProtocolScanner 做什么

文件：

- `Editor/LiteProtocolScanner.cs`

菜单：

- `StellarNetLite/重新生成协议与组件常量表`
- `StellarNetLite/强制重新生成协议与组件常量表`

特点：

- 带 `[InitializeOnLoad]`，Editor 启动后会自动尝试扫描一次。
- 也支持手动触发普通生成和强制生成。

### 3.1 它会扫描什么

当前会扫描这些标记：

- `[NetMsg]`
- `[RoomComponent]`
- `[ServerModule]`
- `[ClientModule]`
- `[NetHandler]`

### 3.2 它会生成什么

1. 协议 Id 常量。
2. 协议元数据注册表。
3. 房间组件 Id 常量。
4. 房间组件绑定器。
5. 全局模块绑定器。
6. 聚合注册入口 `AutoRegistry`。

### 3.3 FeatureKey 是怎么来的

`LiteProtocolScanner` 会优先从源文件头部读取：

```csharp
// [AutoGeneratedFeatureKey] FeatureName
```

如果没写这个标记，就回退到命名空间推导。

这会影响：

- 分片文件名
- 分片常量类名
- 分片绑定器类名

所以脚手架生成出来的文件头不要随便删。

### 3.4 什么时候必须重新生成

出现以下改动后，必须重新跑一次：

1. 新增协议类。
2. 修改协议 MsgId。
3. 新增房间组件。
4. 新增全局模块。
5. 新增或修改 `[NetHandler]` 方法。
6. 改了 FeatureKey、命名空间或标记特性。

## 4. NetPrefabScanner 做什么

文件：

- `Editor/NetPrefabScanner.cs`

菜单：

- `StellarNetLite/生成网络预制体常量表 (Net Prefabs)`

职责：

- 扫描项目中的网络预制体。
- 生成 `NetPrefabConsts.cs`。

当前识别条件：

1. Prefab 位于 `Resources` 目录下。
2. Prefab 根节点挂有 `NetIdentity`。

生成结果主要包含：

1. `PrefabHash` 常量。
2. `Hash -> Resources 路径` 映射。

这份映射会被客户端对象生成视图使用，例如：

- `Runtime/Client/Views/ObjectSpawnerView.cs`

常见坑：

- 预制体不在 `Resources` 下。
- `NetIdentity` 没挂在根节点。
- 改了预制体后没有重新生成。

---

## 5. NetConfigEditorWindow 做什么

文件：

- `Editor/NetConfigEditorWindow.cs`

菜单：

- `StellarNetLite/网络配置 (NetConfig)`

职责：

- 可视化编辑 `NetConfig`。
- 保存到配置文件 `NetConfig/netconfig.json`。

当前对应的核心配置类型：

- `Runtime/Shared/Infrastructure/NetConfig.cs`

当前核心字段包括：

- `Ip`
- `Port`
- `MaxConnections`
- `TickRate`
- `MaxRoomLifetimeHours`
- `MaxReplayFiles`
- `OfflineTimeoutLobbyMinutes`
- `OfflineTimeoutRoomMinutes`
- `EmptyRoomTimeoutMinutes`
- `MinClientVersion`

保存根目录支持：

- `StreamingAssets`
- `PersistentDataPath`

## 6. StellarNetScaffoldWindow 做什么

文件：

- `Editor/StellarNetScaffoldWindow.cs`

菜单：

- `StellarNetLite/业务脚手架生成器`

职责：

- 通过模板快速生成一套业务骨架。
- 支持两类脚手架：
  - 房间组件
  - 全局模块

当前模板输出会覆盖这些区域：

- `Shared/Protocol/...Protocols.cs`
- `Server/Components/...` 或 `Server/Modules/...`
- `Client/Components/...` 或 `Client/Modules/...`

生成完成后会自动调用：

- `LiteProtocolScanner.ManualRun()`

也就是说：

- 脚手架生成完，不只是文件落盘。
- 协议与绑定代码也会同步重新生成。

---

## 7. ServerMonitorWindow 做什么

文件：

- `Editor/ServerMonitorWindow.cs`

菜单：

- `StellarNetLite/服务端运行时监控 (Server Monitor)`

职责：

- 在 Unity 运行时查看当前服务端状态。

适合检查：

- 房间数量。
- 会话数量。
- 房间成员。
- 房间状态。
- 已装配组件。

这个窗口不参与 Runtime 逻辑，只负责观察当前运行态。

---

## 8. FolderContentCopyTool 做什么

文件：

- `Editor/FolderContentCopyTool.cs`

菜单：

- `StellarNetLite/Folder Content Copy Tool`

职责：

- 扫描指定目录。
- 按文件类型筛选。
- 拼接内容后复制到剪贴板。

它不是框架运行必需工具，主要用于：

- 代码审阅
- 内容整理
- 外部分析输入准备

## 9. 推荐开发顺序

如果你要新增一套业务功能，推荐顺序如下：

1. 用 `StellarNetScaffoldWindow` 生成基础骨架。
2. 补协议、客户端组件、服务端组件或模块逻辑。
3. 运行 `重新生成协议与组件常量表`。
4. 如果新增了网络预制体，运行 `生成网络预制体常量表`。
5. 检查 `Assets/StellarNetLiteGenerated` 是否完整生成。
6. 进入运行模式，用 `Server Monitor` 检查注册结果是否正确。

## 10. 最常见的 4 个问题

### 10.1 写了协议，但运行时报未注册

优先检查：

1. 是否加了 `[NetMsg]`。
2. 是否重新运行了 `LiteProtocolScanner`。
3. MsgId 是否冲突。

### 10.2 写了组件或模块，但收不到消息

优先检查：

1. 是否加了 `[RoomComponent]` / `[ServerModule]` / `[ClientModule]`。
2. 方法是否加了 `[NetHandler]`。
3. 方法签名是否符合扫描规则。
4. 是否重新生成了绑定器。

### 10.3 预制体明明存在，但对象就是生成不出来

优先检查：

1. 是否在 `Resources` 下。
2. 根节点是否挂了 `NetIdentity`。
3. 是否重新运行了 `NetPrefabScanner`。

### 10.4 配置改了，但运行结果没变

优先检查：

1. 改的是 `StreamingAssets` 还是 `PersistentDataPath`。
2. 当前运行代码到底从哪个根目录加载配置。
3. `NetConfig/netconfig.json` 是否真的写到目标目录。

---

## 11. 结论

当前 StellarNet Lite 的自动装配链可以概括成一句话：

- Runtime 写业务事实。
- Editor 负责扫描与生成静态注册代码。

理解这条边界后，定位问题会快很多：

- 业务逻辑问题，先查 Runtime。
- 注册缺失问题，先查 Editor 生成链。
