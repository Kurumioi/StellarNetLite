# 02. StellarNet Lite 底层通讯与传输层指南

> 本文补齐当前仓库缺失的传输层教学。  
> 对应代码主要在：
> - `Runtime/Shared/Core/INetworkTransport.cs`
> - `Runtime/Transports/KCP/KcpTransportProvider.cs`
> - `Runtime/Transports/TCP/TcpTransportProvider.cs`
> - `Runtime/Transports/UDP/UdpTransportProvider.cs`
> - `Runtime/StellarNetAppManager.cs`

---

## 1. 传输层在框架里的位置

StellarNet Lite 的业务层并不直接依赖某个具体网络库。

当前 Runtime 通过接口隔离传输层：

```csharp
INetworkTransport
```

这意味着：

- 业务层不直接耦合 KCP
- 业务层不直接耦合 TCP Socket
- 业务层不直接耦合 UDP Socket

框架真正依赖的是：

- 启动 / 停止
- 收发 `Packet`
- 连接事件

---

## 2. `INetworkTransport` 规定了什么

当前接口主要规定了三类东西：

### 2.1 服务端事件

- `OnServerStartedEvent`
- `OnServerStoppedEvent`
- `OnServerClientConnectedEvent`
- `OnServerClientDisconnectedEvent`
- `OnServerReceivePacketEvent`

### 2.2 客户端事件

- `OnClientStartedEvent`
- `OnClientStoppedEvent`
- `OnClientConnectedEvent`
- `OnClientDisconnectedEvent`
- `OnClientReceivePacketEvent`

### 2.3 主动行为

- `StartServer()`
- `StartClient()`
- `StartHost()`
- `StopServer()`
- `StopClient()`
- `SendToServer(Packet packet)`
- `SendToClient(int connectionId, Packet packet)`
- `DisconnectClient(int connectionId)`
- `ApplyConfig(NetConfig config)`

这就是当前传输层和上层框架之间的契约边界。

---

## 3. `StellarNetAppManager` 怎么使用传输层

当前启动管理器要求同物体上必须挂一个传输层组件：

```csharp
[RequireComponent(typeof(INetworkTransport))]
```

在 `Awake()` 里，它会：

1. 找到 `INetworkTransport`
2. 加载 `NetConfig`
3. 调用 `Transport.ApplyConfig(...)`
4. 订阅服务端和客户端事件

也就是说，当前传输层不是“运行时动态选择字符串配置”，而是：

- 你在场景里挂哪个 Provider
- 当前项目就跑哪个物理层

---

## 4. 当前仓库内置了哪三种传输层

### 4.1 `KcpTransportProvider`

位置：

- `Runtime/Transports/KCP/KcpTransportProvider.cs`

特点：

- 基于 `kcp2k`
- 默认走可靠通道
- 当前配置最完整
- 当前最适合作为默认在线业务传输层

### 4.2 `TcpTransportProvider`

位置：

- `Runtime/Transports/TCP/TcpTransportProvider.cs`

特点：

- 纯 C# Socket 实现
- 逻辑清晰
- 适合学习和调试
- 当前也能承载完整主链

### 4.3 `UdpTransportProvider`

位置：

- `Runtime/Transports/UDP/UdpTransportProvider.cs`

特点：

- 纯 UDP
- 不保证可靠性
- 当前代码里已经明确打了警告日志
- **不建议用于生产业务链**

---

## 5. 三种传输层的共同点

虽然底层实现不同，但当前三种 Provider 对上层都暴露同一种行为：

### 5.1 都接收 `NetConfig`

通过：

```csharp
ApplyConfig(NetConfig config)
```

当前 `NetConfig` 里和传输层最直接相关的字段有：

- `Ip`
- `Port`
- `MaxConnections`

### 5.2 都收发统一的 `Packet`

上层不会直接处理 Socket 数据，而是统一处理：

```csharp
Packet
```

实际打包和拆包由：

- `LitePacketFormatter`

负责。

### 5.3 都通过事件把数据交给 `StellarNetAppManager`

所以业务主链不关心底层到底是：

- KCP
- TCP
- UDP

---

## 6. KCP 当前实现特点

### 6.1 依赖 `kcp2k`

当前 KCP Provider 使用：

- `KcpServer`
- `KcpClient`

### 6.2 当前默认配置偏向“可用的在线业务链”

代码里当前已经写了一套默认 `KcpConfig`，包括：

- `NoDelay = true`
- `Interval = 10`
- `FastResend = 2`
- `CongestionWindow = false`

这说明当前 KCP 方案不是最保守的配置，而是偏低延迟的可靠传输配置。

### 6.3 当前主机模式怎么启动

当前主机模式本质上就是：

```csharp
StartServer();
StartClient();
```

### 6.4 当前建议

如果你要跑默认 Demo 或真实在线业务，当前最优先推荐：

- KCP

---

## 7. TCP 当前实现特点

### 7.1 纯 Socket 方案

当前 TCP Provider 基于：

- `TcpListener`
- `TcpClient`
- `NetworkStream`

### 7.2 自己维护连接和主线程回调

为了避免在后台线程直接触碰 Unity 侧逻辑，当前实现用了：

```csharp
ConcurrentQueue<Action> _mainThreadActions
```

后台线程负责收包，主线程 `Update()` 再把事件吐给上层。

### 7.3 包格式处理

当前 TCP 实现不是裸写 `Packet`，而是：

1. 先写入 4 字节长度头
2. 再写序列化后的包体

所以它是：

- 长度头 + 包体

的经典 TCP 流式拆包方案。

### 7.4 当前建议

如果你想学习底层收发流程，TCP 版本更直观。

如果你想直接拿来跑更接近实际在线游戏的链路，优先还是 KCP。

---

## 8. UDP 当前实现特点

### 8.1 当前代码已经明确标警告

`UdpTransportProvider` 在启动时会直接打出警告日志，说明：

- 纯 UDP 会丢包
- 当前实现不适合作为生产环境业务链

### 8.2 它仍然保留的意义

UDP 版本当前更像：

- 传输层实验实现
- 最小物理层演示
- 学习用参考

### 8.3 当前限制

因为它没有可靠重传、顺序保证、拥塞控制等完整治理能力，所以对于：

- 登录
- 建房
- 房间状态同步
- 回放下载

这类业务链都不够稳。

---

## 9. 传输层和弱网 / 重连是什么关系

当前重连主链在：

- `StellarNetAppManager`
- `ClientNetworkMonitor`

要注意两件事：

### 9.1 物理断线由传输层触发

比如：

- `OnClientDisconnectedEvent`

这个事件会被 `StellarNetAppManager` 接住，并决定是：

- 进入软挂起自动重连
- 还是直接硬清理

### 9.2 当前弱网监控主要靠 Ping 协议，不是靠传输层 `GetRTT()`

虽然接口里有：

```csharp
float GetRTT();
```

但当前弱网监控真正依赖的是：

- `C2S_Ping`
- `S2C_Pong`
- `Local_PingResult`

也就是说，当前 `GetRTT()` 更多是接口占位，不是整个弱网系统的唯一事实源。

---

## 10. 现在该怎么选

如果你是初学者，当前建议这样选：

### 10.1 想直接跑默认在线链路

选：

- `KcpTransportProvider`

### 10.2 想学习底层收发和长度头拆包

选：

- `TcpTransportProvider`

### 10.3 想实验不可靠传输或研究极简物理层

选：

- `UdpTransportProvider`

但不要把 UDP 版当成默认生产方案。

---

## 11. 初学者最容易误解的点

### 11.1 传输层和协议层不是一回事

传输层管的是：

- 连上没有
- 包有没有送到
- 物理连接什么时候断

协议层管的是：

- 这包是什么业务
- 应该进全局还是进房间
- 应该交给哪个模块或组件

### 11.2 换了传输层，不代表业务代码要重写

只要你的 Provider 继续满足 `INetworkTransport`，上层主链基本不用改。

### 11.3 UDP 版本存在，不等于它适合直接上线

当前代码已经明确把它定义成：

- 不推荐生产业务链

不要因为它“能跑”就默认它“适合跑正式逻辑”。

---

## 12. 一句话总结

当前框架的传输层设计核心是：

- 用 `INetworkTransport` 把业务层和物理层解耦
- KCP 作为当前最完整的在线实现
- TCP 作为可读性更高的教学实现
- UDP 作为实验性实现
