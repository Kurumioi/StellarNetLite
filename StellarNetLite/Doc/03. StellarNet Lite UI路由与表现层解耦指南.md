# 03. StellarNet Lite UI路由与表现层解耦指南

> **保姆级教学**：在联机游戏中，UI 的管理往往是最容易失控的环节。为什么回放录像时，UI 总是会弹出莫名其妙的在线交互按钮？本篇将向你**提倡并推荐**一种 UI 路由模式，帮助你优雅地解耦表现层，告别 UI 状态管理的噩梦。

---

## 1. 为什么推荐将网络逻辑与 UI 彻底解耦？

很多新手在早期开发时，习惯这样写代码：
```csharp
// ❌ 容易导致耦合的写法
public void OnS2C_LoginResult(S2C_LoginResult msg) {
    if (msg.Success) {
        GameObject.Find("LoginPanel").SetActive(false);
        GameObject.Find("LobbyPanel").SetActive(true);
    }
}
```
**这种写法的潜在隐患：**
1. **强耦合**：网络模块直接操作了具体的 UI 对象。如果 UI 预制体改名了、删除了，网络代码就会报错。
2. **状态跌落难以处理**：如果玩家在房间里突然断网了，谁来负责把房间 UI 关掉，把登录 UI 重新打开？
3. **回放灾难**：看录像时，底层同样会收到网络包，如果网络模块直接控制 UI，看录像时就会弹出“开始游戏”等在线交互按钮！

**StellarNet Lite 提倡的解法：MSV 架构与事件直抛。**
网络模块只负责抛出数据事件，由专门的 **UIRouter（UI 路由）** 来决定开什么面板、关什么面板。虽然框架不强制你必须这么做，但强烈建议你采用这种模式。

---

## 2. 第一层：全局 UI 路由 (GlobalUIRouter)

`GlobalUIRouter` 负责跨房间、跨状态的“大导航”。
比如：登录、大厅、断线提示、录像播放面板的切换。

它监听的是 `GlobalTypeNetEvent`。

### 实战：监听全服广播
打开你的 UI 脚本（假设叫 `Panel_Lobby`）：

```csharp
using StellarNet.UI;
using StellarNet.Lite.Client.Core.Events;
using MineGame.Shared.Protocol;

public class Panel_Lobby : UIPanelBase
{
    public override void OnInit()
    {
        base.OnInit();
        // ✅ 推荐写法：注册事件，并绑定生命周期！
        GlobalTypeNetEvent.Register<S2C_BroadcastSync>(OnReceiveBroadcast)
            .UnRegisterWhenGameObjectDestroyed(gameObject);
    }

    private void OnReceiveBroadcast(S2C_BroadcastSync msg)
    {
        Debug.Log($"收到广播：{msg.SenderName} 说 {msg.Content}");
        // 这里可以实例化一个跑马灯 UI 等等
    }
}
```

### 🚨 致命避坑：防内存泄漏神器
注意上面的 `.UnRegisterWhenGameObjectDestroyed(gameObject)`！
如果你不加这句后缀，当这个 UI 面板被销毁后，事件总线里还存着它的引用。下次再收到广播时，就会报出可怕的 `MissingReferenceException`（试图访问已销毁的对象）。
- 如果 UI 是频繁隐藏/显示的，请用：`.UnRegisterWhenMonoDisable(this)`

## 3. 第二层：房间组件 UI 路由 (RoomUIRouterBase)

房间里的 UI 更加复杂。因为房间分为 **在线房间 (OnlineRoom)** 和 **回放房间 (ReplayRoom)**。
- 在线房间：需要显示“准备按钮”、“攻击按钮”、“发送表情按钮”。
- 回放房间：只需要显示只读的画面，**绝对不能**让玩家点“发送表情”发到服务器去！

### 推荐方案：一对 Router 伺候一个 Component
我们提倡在业务组件（如 `ClientEmojiComponent`）之上，建立两个 Router：
1. `EmojiOnlineUIRouter`
2. `EmojiReplayUIRouter`

#### 编写 Online Router
```csharp
using StellarNet.Lite.Game.Client.Infrastructure;
using MineGame.Client.Components;
using StellarNet.UI;

// 继承 ClientRoomUIRouterBase，泛型填入你的客户端组件
public class EmojiOnlineUIRouter : ClientRoomUIRouterBase<ClientEmojiComponent>
{
    protected override void OnBind(ClientEmojiComponent component)
    {
        // 当进入在线房间，且组件绑定成功时，打开表情交互面板
        UIKit.OpenPanel<Panel_EmojiInteract>();
    }

    protected override void OnUnbind()
    {
        // 当离开房间，或者组件解绑时，自动关闭面板
        UIKit.ClosePanel<Panel_EmojiInteract>();
    }
}
```

#### 编写 Replay Router
```csharp
public class EmojiReplayUIRouter : ClientRoomUIRouterBase<ClientEmojiComponent>
{
    protected override void OnBind(ClientEmojiComponent component)
    {
        // 回放时，我们什么交互面板都不打开！保持只读！
        // 或者你可以打开一个专属的 Panel_EmojiReplayView (只显示统计数据，没有发送按钮)
    }

    protected override void OnUnbind()
    {
    }
}
```

### 它是怎么被挂载的？
在监听进房事件的管理器中（如 `RoomViewManager.cs`）：
```csharp
bool isOnline = NetClient.State == ClientAppState.OnlineRoom;
var emojiComp = evt.Room.GetComponent<ClientEmojiComponent>();

if (emojiComp != null)
{
    if (isOnline) {
        var router = roomViewRoot.AddComponent<EmojiOnlineUIRouter>();
        router.Bind(emojiComp);
    } else {
        var router = roomViewRoot.AddComponent<EmojiReplayUIRouter>();
        router.Bind(emojiComp);
    }
}
```
**这就是 StellarNet Lite 提倡的回放系统不串线的核心思路！** 回放和在线，从路由层就彻底分道扬镳了。

---

## 4. 总结：UI 开发的推荐原则

在 StellarNet Lite 中写 UI，我们强烈建议你遵循以下原则：
1. **Panel 尽量不直接发网络包**：UI 最好只负责采集输入，调用 `NetClient.Send`，绝不自己组装底层 Packet。
2. **Panel 的生命周期交由 Router 管理**：Panel 尽量不要监听“游戏结束”来 `CloseSelf()`，而是由 `UIRouter` 来统一决定它的打开和关闭。
3. **Panel 监听事件必须防泄漏**：只要 `Register` 了网络事件，尾巴上务必加上 `.UnRegisterWhenMonoDisable(this)` 或 `.UnRegisterWhenGameObjectDestroyed(gameObject)`。
