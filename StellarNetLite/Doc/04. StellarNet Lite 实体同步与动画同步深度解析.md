# 04. StellarNet Lite 实体同步与动画同步深度解析

> 本文只讲当前仓库里已经落地的官方对象同步扩展方案。  
> 对应代码主要在：
> - `Extensions/ObjectSync/Server/Components/ServerObjectSyncComponent.cs`
> - `Extensions/ObjectSync/Client/Components/ClientObjectSyncComponent.cs`
> - `Extensions/ObjectSync/Client/Views/ObjectSpawnerView.cs`
> - `Extensions/ObjectSync/Client/Views/NetTransformView.cs`
> - `Extensions/ObjectSync/Client/Views/NetAnimatorView.cs`
> - `Samples/SocialDemo/Server/Components/ServerSocialRoomComponent.cs`
>
> 补充说明：
> - 当前客户端对象加载链已经支持 `INetPrefabResolver / INetSpawnStrategy / INetViewBinder` 扩展点。
> - 默认策略仍然是 `Resources` 加载。
> - 本文后半段已经直接归并了客户端加载解耦、`PrefabHash / NetId`、AB / Addressables / YooAsset 和对象池接入说明。

---

## 1. 先建立正确心智

StellarNet Lite 当前的内置实体同步不是“客户端权威同步”，而是：

- 服务端维护真实状态。
- 服务端按 Tick 主动广播。
- 客户端缓存最新状态。
- 客户端 View 只负责平滑表现。

所以要先记住两句话：

1. 实体状态事实在服务端。
2. 客户端表现层不负责定义事实，只负责消费事实。

---

## 2. 内置同步链里有哪些固定角色

当前内置同步链一共 5 个核心角色。

### 2.1 `ServerObjectSyncComponent`

职责：

- 在服务端保存所有同步实体。
- 负责 `SpawnObject` / `DestroyObject`。
- 负责周期广播 `S2C_ObjectSync`。
- 负责导出回放和重连所需快照。

### 2.2 `ClientObjectSyncComponent`

职责：

- 缓存服务端同步下来的实体状态。
- 接收 `S2C_ObjectSpawn` / `S2C_ObjectDestroy` / `S2C_ObjectSync`。
- 对 View 提供查询接口：
  - `TryGetTransformData`
  - `TryGetAnimatorData`
- 负责消费录像快照。

### 2.3 `ObjectSpawnerView`

职责：

- 监听房间内的本地生成/销毁事件。
- 根据 `PrefabHash` 解析预制体并生成场景实例。
- 自动挂接或初始化：
  - `NetIdentity`
  - `NetTransformView`
  - `NetAnimatorView`

### 2.4 `NetTransformView`

职责：

- 消费 `ClientObjectSyncComponent` 给出的预测位移数据。
- 对远端对象做平滑追赶。
- 对本地玩家做软和解。

### 2.5 `NetAnimatorView`

职责：

- 消费 `ClientObjectSyncComponent` 给出的动画状态。
- 处理状态切换、BlendTree 浮点参数和平滑播放速度。

## 3. 三种核心协议分别做什么

### 3.1 `S2C_ObjectSpawn`

作用：

- 通知客户端生成对象。
- 携带完整初始化状态。

核心字段来自：

- `ObjectSpawnState`

它适合“完整恢复”场景：

- 首次生成
- 断线重连补快照
- 回放 Seek 后恢复

补充：

- 当前回放/重连快照恢复不仅会处理“缺了要补、没了要删”。
- 对于快照里仍然存在的对象，`ClientObjectSyncComponent` 也会刷新其缓存状态。
- 这能避免回放 Seek 或重连恢复后，老对象停留在旧位置或旧动画参数上。

### 3.2 `S2C_ObjectDestroy`

作用：

- 通知客户端销毁指定 `NetId` 的对象。

### 3.3 `S2C_ObjectSync`

作用：

- 高频广播对象增量状态。
- 只传 Transform / Animator 这类高频字段。

---

## 4. 为什么有 `ObjectSpawnState` 和 `ObjectSyncState` 两套结构

这是当前实现里非常关键的一层拆分。

### 4.1 `ObjectSpawnState`

适合完整恢复。

它包含：

- `NetId`
- `PrefabHash`
- `Mask`
- 位置 / 旋转 / 方向 / 缩放
- 动画状态和浮点参数
- `OwnerSessionId`

### 4.2 `ObjectSyncState`

适合高频同步。

它只保留当前 Tick 需要刷新的字段：

- `NetId`
- `Mask`
- Transform 数据
- Animator 数据
- `ServerTime`

拆分原则很简单：

- 完整恢复走 `ObjectSpawnState`
- 高频同步走 `ObjectSyncState`

---

## 5. `EntitySyncMask` 的意义

当前内置同步掩码是：

```csharp
[Flags]
public enum EntitySyncMask : byte
{
    None = 0,
    Transform = 1 << 0,
    Animator = 1 << 1,
    All = Transform | Animator
}
```

使用建议：

- 只同步位移：`EntitySyncMask.Transform`
- 位移 + 动画一起同步：`EntitySyncMask.All`
- 纯静态对象：通常不需要走这套链

---

## 6. 服务端这边到底做了什么

### 6.1 `ServerObjectSyncComponent` 维护实体表

服务端内部维护：

```csharp
private readonly Dictionary<int, ServerSyncEntity> _entities;
```

含义：

- key: `NetId`
- value: 服务端当前实体状态

### 6.2 `SpawnObject()` 负责生成

调用：

```csharp
var entity = _sync.SpawnObject(
    prefabHash,
    EntitySyncMask.All,
    spawnPos,
    Vector3.zero,
    Vector3.zero,
    ownerSessionId);
```

它会完成：

1. 分配新 `NetId`
2. 写入 `_entities`
3. 广播 `S2C_ObjectSpawn`

### 6.3 `DestroyObject()` 负责销毁

调用：

```csharp
_sync.DestroyObject(netId);
```

它会完成：

1. 从 `_entities` 移除
2. 广播 `S2C_ObjectDestroy`

### 6.4 `OnTick()` 周期广播增量同步

当前服务端对象同步组件实现了：

```csharp
ITickableComponent
```

调用链是：

```text
ServerRuntimeHost.RunLoop
-> ServerApp.Tick()
-> Room.Tick()
-> ITickableComponent.OnTick()
```

也就是说：

- 同步是服务端主动推进的。
- 不是等客户端发包才同步。
- 当前这条链已经不依赖 Unity `FixedUpdate`
- 当前默认在线同步频率已经调整为每 Tick 一次广播，开发态在未生成配置时也能保持更顺的肉眼观感
- 回放录制频率需要和在线同步频率保持一致，否则录像观感会比局内更稀疏

## 7. 客户端这边到底做了什么

### 7.1 `ClientObjectSyncComponent` 不直接改场景对象

它只负责：

1. 缓存同步状态。
2. 提供查询接口。
3. 抛出本地生成/销毁事件。
4. 回放和重连时恢复快照。

这意味着客户端链路是两段式：

- `ClientObjectSyncComponent` 负责数据
- View 负责表现

### 7.2 `ObjectSpawnerView` 把状态变成场景对象

它监听：

- `Local_ObjectSpawned`
- `Local_ObjectDestroyed`

收到生成事件时会做：

1. 先建立该 `NetId` 的客户端生成记录
2. 调用 `ObjectSyncClientRuntime.PrefabResolver` 按 `PrefabHash` 异步解析预制体
3. 调用 `ObjectSyncClientRuntime.SpawnStrategy` 生成实例
4. 调用 `ObjectSyncClientRuntime.ViewBinder` 挂接 `NetIdentity / NetTransformView / NetAnimatorView`
5. 如果异步加载期间服务端状态继续推进，则在真正生成前读取该 `NetId` 的最新缓存状态

默认情况下：

1. `ResourcesNetPrefabResolver` 仍然会查 `NetPrefabConsts.HashToPathMap`
2. `DefaultNetSpawnStrategy` 仍然会 `Instantiate`
3. `DefaultNetViewBinder` 仍然会完成 `NetIdentity / NetTransformView / NetAnimatorView` 初始化

### 7.3 `NetTransformView` 和 `NetAnimatorView` 只负责表现

- `NetTransformView` 负责位置、旋转、缩放表现
- `NetAnimatorView` 负责动画状态和参数表现

不要把业务事实写进这两个 View。

### 7.4 当前版本一定要分清 `NetId` 和 `PrefabHash`

这是接入 AB / Addressables / YooAsset 时最容易混淆的点。

`NetId` 表示：

- 当前房间里某一个运行时实体实例
- 它解决的是“这个实例是谁”

`PrefabHash` 表示：

- 这个实例应该使用哪一个预制体类型
- 它解决的是“应该加载哪一类资源”

所以正确关系一定是：

```text
服务端下发 ObjectSpawnState
-> 客户端拿到 NetId + PrefabHash
-> NetId 用来维护运行时实体记录
-> PrefabHash 用来交给 PrefabResolver 找资源
```

不是：

```text
NetId -> 资源路径
```

AB / Addressables / YooAsset 都应该做成：

```text
PrefabHash -> 资源定位键
```

这个定位键可以分别是：

- Resources：`HashToPathMap` 里的相对路径
- AB：`bundleName + assetName`
- Addressables：`address`
- YooAsset：`location`

### 7.5 客户端加载策略现在有哪些扩展点

当前客户端对象加载链被拆成了 3 个扩展点：

- `INetPrefabResolver`
- `INetSpawnStrategy`
- `INetViewBinder`

统一入口是：

- `ObjectSyncClientRuntime`

默认行为仍然保持兼容：

- 默认 resolver：`ResourcesNetPrefabResolver`
- 默认 spawn strategy：`DefaultNetSpawnStrategy`
- 默认 view binder：`DefaultNetViewBinder`

也就是说，如果项目什么都不配，旧项目仍然按 `Resources` 路径映射加载和实例化。

### 7.6 加载策略案例：Resources / AB / Addressables / YooAsset

最简单的默认用法：

```csharp
ObjectSyncClientRuntime.Configure(
    prefabResolver: new ResourcesNetPrefabResolver());
```

如果你要接 AssetBundle，核心不是 `NetId -> AB`，而是维护一份：

```text
PrefabHash -> BundleName/AssetName
```

示例：

```csharp
public sealed class AssetBundleNetPrefabResolver : INetPrefabResolver
{
    private readonly Dictionary<int, BundleAssetRef> _hashToBundleRef;

    public AssetBundleNetPrefabResolver(Dictionary<int, BundleAssetRef> hashToBundleRef)
    {
        _hashToBundleRef = hashToBundleRef;
    }

    public async Task<NetPrefabResolveResult> ResolveAsync(int prefabHash, CancellationToken cancellationToken)
    {
        if (!_hashToBundleRef.TryGetValue(prefabHash, out BundleAssetRef bundleRef))
        {
            return NetPrefabResolveResult.CreateFailure($"未知 PrefabHash:{prefabHash}");
        }

        GameObject prefab = await LoadFromBundleAsync(bundleRef.BundleName, bundleRef.AssetName, cancellationToken);
        return prefab != null
            ? NetPrefabResolveResult.CreateSuccess(prefab)
            : NetPrefabResolveResult.CreateFailure($"AB 资源不存在: {bundleRef.BundleName}/{bundleRef.AssetName}");
    }
}

ObjectSyncClientRuntime.Configure(
    prefabResolver: new AssetBundleNetPrefabResolver(hashToBundleRef));
```

Addressables 的思路一样：

```csharp
public sealed class AddressablesNetPrefabResolver : INetPrefabResolver
{
    private readonly Dictionary<int, string> _hashToAddress;

    public AddressablesNetPrefabResolver(Dictionary<int, string> hashToAddress)
    {
        _hashToAddress = hashToAddress;
    }

    public async Task<NetPrefabResolveResult> ResolveAsync(int prefabHash, CancellationToken cancellationToken)
    {
        if (!_hashToAddress.TryGetValue(prefabHash, out string address))
        {
            return NetPrefabResolveResult.CreateFailure($"未知 PrefabHash:{prefabHash}");
        }

        GameObject prefab = await LoadAddressAsync(address, cancellationToken);
        return prefab != null
            ? NetPrefabResolveResult.CreateSuccess(prefab)
            : NetPrefabResolveResult.CreateFailure($"AA 资源不存在: {address}");
    }
}

ObjectSyncClientRuntime.Configure(
    prefabResolver: new AddressablesNetPrefabResolver(hashToAddress));
```

YooAsset 也是同一套模式：

```csharp
public sealed class YooNetPrefabResolver : INetPrefabResolver
{
    private readonly Dictionary<int, string> _hashToLocation;

    public YooNetPrefabResolver(Dictionary<int, string> hashToLocation)
    {
        _hashToLocation = hashToLocation;
    }

    public async Task<NetPrefabResolveResult> ResolveAsync(int prefabHash, CancellationToken cancellationToken)
    {
        if (!_hashToLocation.TryGetValue(prefabHash, out string location))
        {
            return NetPrefabResolveResult.CreateFailure($"未知 PrefabHash:{prefabHash}");
        }

        GameObject prefab = await LoadYooPrefabAsync(location, cancellationToken);
        return prefab != null
            ? NetPrefabResolveResult.CreateSuccess(prefab)
            : NetPrefabResolveResult.CreateFailure($"Yoo 资源不存在: {location}");
    }
}

ObjectSyncClientRuntime.Configure(
    prefabResolver: new YooNetPrefabResolver(hashToLocation));
```

### 7.7 对象池案例：给高频网络对象接 `INetSpawnStrategy`

如果是高频生成/销毁的网络对象，例如：

- 子弹
- 临时特效代理体
- 高频刷新的小型怪物

那么更适合把“资源加载”和“对象生成”继续拆开：

- `INetPrefabResolver` 负责拿到 prefab
- `INetSpawnStrategy` 负责从对象池取实例和归还实例

一个最小对象池策略可以写成：

```csharp
public sealed class PoolNetSpawnStrategy : INetSpawnStrategy
{
    private readonly INetObjectPool _pool;

    public PoolNetSpawnStrategy(INetObjectPool pool)
    {
        _pool = pool;
    }

    public GameObject Spawn(GameObject prefab, in NetEntitySpawnContext context)
    {
        ObjectSpawnState state = context.State;
        GameObject instance = _pool.Get(prefab);
        instance.transform.SetPositionAndRotation(
            new Vector3(state.PosX, state.PosY, state.PosZ),
            Quaternion.Euler(state.RotX, state.RotY, state.RotZ));
        instance.transform.localScale = new Vector3(
            Mathf.Approximately(state.ScaleX, 0f) ? 1f : state.ScaleX,
            Mathf.Approximately(state.ScaleY, 0f) ? 1f : state.ScaleY,
            Mathf.Approximately(state.ScaleZ, 0f) ? 1f : state.ScaleZ);
        return instance;
    }

    public void Despawn(GameObject instance, in NetEntitySpawnContext context)
    {
        if (instance == null)
        {
            return;
        }

        _pool.Release(instance);
    }
}
```

接入方式可以和任意 resolver 组合：

```csharp
ObjectSyncClientRuntime.Configure(
    prefabResolver: new ResourcesNetPrefabResolver(),
    spawnStrategy: new PoolNetSpawnStrategy(pool));
```

或者：

```csharp
ObjectSyncClientRuntime.Configure(
    prefabResolver: new AddressablesNetPrefabResolver(hashToAddress),
    spawnStrategy: new PoolNetSpawnStrategy(pool));
```

所以现在已经补齐两层策略组合案例：

- 不同资源来源：Resources / AB / AA / YooAsset
- 不同生成方式：普通 Instantiate / 对象池

### 7.8 异步加载期间为什么不会把状态搞乱

当前 `ObjectSpawnerView` 内部会追踪实体加载状态：

- `LoadingAsset`
- `SpawningView`
- `Ready`
- `Failed`
- `Destroyed`

关键行为是：

1. 收到 `spawn` 时，先建立该 `NetId` 的客户端记录
2. 异步加载资源期间，`ClientObjectSyncComponent` 继续接收服务端同步
3. 真正生成前，会重新读取该 `NetId` 的最新缓存状态
4. 如果资源还没回来，但服务端已经销毁了该实体，就取消或丢弃结果

这保证了：

- 服务端 AI / 战斗 / 动画可以继续推进
- 客户端不会因为资源晚到就丢失中间状态
- 视图生成出来时直接落到最新权威状态

## 8. 开发者如何使用框架自带的实体同步组件

这一节是给业务开发者看的实操说明。

### 8.1 第一步：让房间带上 `ObjectSync` 组件

你创建房间时，房间组件列表里必须包含：

```csharp
ComponentIdConst.ObjectSync
```

例如当前 Demo 的房型模板：

```csharp
ComponentIds = new[]
{
    ComponentIdConst.SocialRoom,
    ComponentIdConst.RoomSettings,
    ComponentIdConst.ObjectSync
};
```

如果房间里没有 `ObjectSync`，后面所有内置生成、同步、动画表现都不会工作。

### 8.2 第二步：在服务端业务组件里拿到 `ServerObjectSyncComponent`

典型写法：

```csharp
private ServerObjectSyncComponent _sync;

public override void OnInit()
{
    _sync = Room.GetComponent<ServerObjectSyncComponent>();
}
```

如果这里拿不到，说明房间组件列表没把 `ObjectSync` 加进去。

### 8.3 第三步：准备网络预制体

要让 `ObjectSpawnerView` 正常生成对象，预制体必须满足这些条件：

1. 预制体根节点挂有 `NetIdentity`。
2. 如果使用默认 `ResourcesNetPrefabResolver`，预制体放在 `Resources` 目录下。
3. 如果要播动画，预制体上要有 `Animator`。
4. 改完预制体后要重新生成 `NetPrefabConsts`。
5. 如果使用 AB / Addressables / YooAsset，自行维护 `PrefabHash -> 资源定位键` 映射。

注意：

- `NetIdentity` 不只是运行时标识，也是 `NetPrefabScanner` 识别网络预制体的条件。
- `NetPrefabScanner` 现在会为所有根节点挂了 `NetIdentity` 的 Prefab 生成静态 `PrefabHash`，不再强制要求它必须在 `Resources` 下。
- `NetTransformView`、`NetAnimatorView` 可以提前挂在预制体上做参数调校。
- 如果没提前挂，`ObjectSpawnerView` 会按 `Mask` 自动补挂。

### 8.4 第四步：生成网络预制体常量

预制体准备好后，运行：

- `StellarNetLite/生成网络预制体常量表 (Net Prefabs)`

然后你才能在服务端安全使用：

```csharp
NetPrefabConsts.NetPrefabs_YourPrefabName
```

### 8.5 第五步：在服务端生成对象

最小生成代码：

```csharp
var entity = _sync.SpawnObject(
    NetPrefabConsts.NetPrefabs_SocialPlayer,
    EntitySyncMask.All,
    spawnPos,
    Vector3.zero,
    Vector3.zero,
    session.SessionId);
```

如果你需要初始动画，可以立刻补：

```csharp
entity.AnimStateHash = Animator.StringToHash("Idle");
entity.AnimNormalizedTime = 0f;
```

### 8.6 第六步：只在服务端修改实体状态

后续所有位置、旋转、速度、动画状态，都应该改服务端实体：

```csharp
entity.Position = newPos;
entity.Rotation = new Vector3(0f, yaw, 0f);
entity.Velocity = newVel;
entity.AnimStateHash = walkHash;
```

不要在客户端直接把远端对象当事实源去改。

### 8.7 第七步：销毁对象

对象生命周期结束时：

```csharp
_sync.DestroyObject(entity.NetId);
```

客户端会自动收到销毁通知并移除场景对象。

## 9. 严谨案例 A：同步一个可移动角色

这一节给出一套最小可用流程。

### 9.1 目标

实现一个房间内可移动角色，并使用框架自带的：

- 实体生成
- Transform 同步
- Animator 同步

### 9.2 房间组件准备

房间组件列表至少包含：

```csharp
new[]
{
    ComponentIdConst.RoomSettings,
    ComponentIdConst.ObjectSync,
    ComponentIdConst.SocialRoom
}
```

### 9.3 服务端业务组件写法

下面这段就是当前 Demo 使用的模式，已经和仓库实现对齐：

```csharp
private ServerObjectSyncComponent _sync;
private readonly Dictionary<string, int> _sessionToNetId = new();
private static readonly int AnimHash_Idle = Animator.StringToHash("Idle");
private static readonly int AnimHash_Walk = Animator.StringToHash("Walk");

public override void OnInit()
{
    _sync = Room.GetComponent<ServerObjectSyncComponent>();
    _sessionToNetId.Clear();
}

public override void OnGameStart()
{
    foreach (var kvp in Room.Members)
    {
        SpawnPlayer(kvp.Value);
    }
}

private void SpawnPlayer(Session session)
{
    var entity = _sync.SpawnObject(
        NetPrefabConsts.NetPrefabs_SocialPlayer,
        EntitySyncMask.All,
        new Vector3(0f, 0f, 0f),
        Vector3.zero,
        Vector3.zero,
        session.SessionId);

    entity.AnimStateHash = AnimHash_Idle;
    entity.AnimNormalizedTime = 0f;
    _sessionToNetId[session.SessionId] = entity.NetId;
}
```

### 9.4 处理移动输入

当服务端收到业务移动消息时，只改服务端实体：

```csharp
[NetHandler]
public void OnMove(Session session, C2S_Move msg)
{
    if (!_sessionToNetId.TryGetValue(session.SessionId, out int netId))
    {
        return;
    }

    var entity = _sync.GetEntity(netId);
    if (entity == null)
    {
        return;
    }

    entity.Position = new Vector3(msg.PosX, msg.PosY, msg.PosZ);
    entity.Velocity = new Vector3(msg.VelX, msg.VelY, msg.VelZ);
    entity.Rotation = new Vector3(0f, msg.RotY, 0f);
    entity.AnimStateHash = entity.Velocity.sqrMagnitude > 0.01f ? AnimHash_Walk : AnimHash_Idle;
}
```

### 9.5 客户端需要额外写什么

如果你已经：

1. 把 `ObjectSync` 放进房间组件列表。
2. 让预制体满足网络预制体要求。
3. 让 `RoomViewManager` 能挂上 `ObjectSpawnerView`。

那么客户端通常不需要再写一套实体同步组件。

框架会自动完成：

1. `ClientObjectSyncComponent` 缓存状态。
2. `ObjectSpawnerView` 生成预制体。
3. `NetTransformView` 平滑位移。
4. `NetAnimatorView` 切换动画。

这就是“用框架自带同步组件”的正确方式。

## 10. 严谨案例 B：只同步一个抛射物

如果对象只需要位移，不需要 Animator，可以只用：

```csharp
EntitySyncMask.Transform
```

服务端示例：

```csharp
var bullet = _sync.SpawnObject(
    NetPrefabConsts.NetPrefabs_Bullet,
    EntitySyncMask.Transform,
    muzzlePos,
    Vector3.zero,
    fireDir * bulletSpeed);
```

后续在 `OnTick()` 中推进：

```csharp
bullet.Position += bullet.Velocity * deltaTime;
```

命中或超时后：

```csharp
_sync.DestroyObject(bullet.NetId);
```

这个场景下客户端只会挂：

- `NetIdentity`
- `NetTransformView`

不会走动画同步链。

## 11. 开发者如何正确使用 `NetAnimatorView`

如果你要同步动画，不只是改 `AnimStateHash`，还要注意预制体上的 `NetAnimatorView` 配置。

### 11.1 最少配置

需要：

1. 预制体上有 `Animator`
2. `Mask` 含 `Animator`
3. 服务端正确填写：
   - `AnimStateHash`
   - `AnimNormalizedTime`

### 11.2 如果你的动画依赖 BlendTree 参数

则要在预制体的 `NetAnimatorView` 上配置：

- `FloatParam1Name`
- `FloatParam2Name`
- `FloatParam3Name`

然后服务端写：

```csharp
entity.FloatParam1 = moveSpeed;
entity.FloatParam2 = turnSpeed;
```

否则即使同步包里有值，客户端也不知道这些值要喂给哪个 Animator 参数。

## 12. 为什么这套组件天然支持重连和回放

这是当前内置同步链最有价值的地方之一。

`ServerObjectSyncComponent` 实现了：

```csharp
IReplaySnapshotProvider
```

`ClientObjectSyncComponent` 实现了：

```csharp
IReplaySnapshotConsumer
```

所以同一套实体同步链同时能覆盖：

- 在线同步
- 断线重连快照恢复
- 录像关键帧恢复

对业务开发者来说，这意味着：

- 只要你基于内置 `ObjectSync` 做实体同步，
- 你就天然接入了当前框架的重连和回放主链。

## 13. 常见错误排查

### 13.1 对象根本生成不出来

优先检查：

1. 房间组件列表里是否包含 `ComponentIdConst.ObjectSync`
2. 如果使用默认 `Resources` 策略，预制体是否在 `Resources`
3. 预制体根节点是否挂了 `NetIdentity`
4. 是否重新生成了 `NetPrefabConsts`
5. 如果使用 AB / Addressables / YooAsset，是否正确配置了 `PrefabHash -> 资源定位键`

### 13.2 对象生成了，但不会动

优先检查：

1. `SpawnObject` 时的 `Mask` 是否包含 `Transform`
2. 服务端是否真的在更新 `entity.Position` / `entity.Velocity`
3. 是否只改了客户端对象，没改服务端实体

### 13.3 动画始终不播

优先检查：

1. `Mask` 是否包含 `Animator`
2. `AnimStateHash` 是否有效
3. 预制体上是否存在 `Animator`
4. 如果用了 BlendTree，`FloatParam1Name ~ 3Name` 是否配置正确

### 13.4 本地玩家抖动很大

优先检查：

1. 是否同时有别的脚本直接改同一个 Transform
2. 客户端是否把远端对象也当成本地玩家在驱动
3. 服务端是否长期不更新状态，导致客户端只在追旧包

## 14. 什么时候不要继续往 `ObjectSync` 里塞字段

以下字段通常不建议直接塞进高频同步结构：

- 血量
- Buff
- 技能冷却
- 背包
- 任务状态

更推荐：

1. 新建独立房间组件
2. 新定义业务协议
3. 按状态变化时同步，而不是按高频 Tick 同步

`ObjectSync` 更适合：

- 位置
- 旋转
- 速度
- 动画状态
- 少量高频表现参数

---

## 15. 给新手开发者的最短结论

如果你只是想快速用上框架自带实体同步，请记住下面 5 步：

1. 房间组件里加上 `ComponentIdConst.ObjectSync`
2. 服务端业务组件里拿到 `ServerObjectSyncComponent`
3. 给网络预制体根节点挂 `NetIdentity`，默认策略放进 `Resources`，自定义策略维护 `PrefabHash -> 资源定位键`
4. 用 `SpawnObject()` / `DestroyObject()` 管对象生命周期
5. 只在服务端改 `ServerSyncEntity` 的位置、速度和动画状态

按这 5 步做，内置的实体同步、动画同步、重连快照、回放快照都会一起接上。
