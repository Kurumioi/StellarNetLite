# 04. StellarNet Lite 实体同步与动画同步深度解析

> **核心导读**：在多人联机游戏中，网络延迟（Latency）和网络抖动（Jitter）是不可避免的物理限制。如果仅仅是把服务端的坐标直接赋值给客户端，玩家看到的画面将是极其卡顿和瞬移的。本篇将深度揭秘 StellarNet Lite 是如何通过一系列“欺骗视觉”的黑科技，实现丝滑的实体与动画同步的；同时，我们将讲解服务端的 `ITickableComponent` 心跳机制。

---

## 1. 核心心智：高频状态同步与组件化驱动

StellarNet Lite 采用的是**状态同步（State Synchronization）**而非帧同步。
服务端的 `ServerObjectSyncComponent` 会以固定的频率（受 `TickRate` 控制），将场景中所有实体的绝对坐标、速度、旋转和动画状态打包成 `S2C_ObjectSync` 协议，广播给所有客户端。

**绝对禁忌**：
客户端的 View 层（如 `NetTransformView`）**绝对不能**直接修改核心数据模型，它只能被动接收 `PredictedTransformData`，并通过插值算法去“追赶”服务端的真实状态。

---

## 2. 空间同步黑科技：NetTransformView 深度解析

打开客户端的 `NetTransformView.cs`，你会发现它针对“本地玩家”和“远端玩家”采用了两套完全不同的同步策略。

### 2.1 远端平滑预测 (Dead Reckoning)
当你在看其他玩家移动时，走的是 `ProcessRemoteTransformSync` 逻辑。
- **航位推测**：客户端不仅接收坐标，还接收**速度（Velocity）**。在两次网络包到达的间隙，客户端会利用速度乘以时间，自己推测玩家应该走到了哪里。
- **SmoothDamp 追赶**：当真实的包到达时，如果发现预测位置和真实位置有偏差，框架不会直接“瞬移”过去，而是使用 `Vector3.SmoothDamp` 像弹簧一样平滑地把模型拉回正确轨道。
- **动态阈值（Snap & CatchUp）**：
    - 如果误差极小（`< StopThreshold`），直接对齐，防止原地微小抖动。
    - 如果误差较大（`> CatchUpThreshold`），加快平滑速度（`effectiveSmoothTime *= 0.5f`）赶紧追上去。
    - 如果误差大得离谱（`> SnapThreshold`），说明发生了严重卡顿或传送，放弃平滑，直接**瞬移（Snap）**过去。

### 2.2 本地柔性和解 (Soft Reconciliation)
当你在控制自己的角色时，走的是 `ProcessLocalReconciliation` 逻辑。
- **客户端先行**：为了保证手感，你按下方向键时，本地的 `CharacterController` 已经开始移动了（客户端预测）。
- **服务端纠偏**：服务端收到你的移动请求后，会进行合法性校验，并把最终的权威坐标发回来。
- **柔性和解**：如果服务端的坐标和本地坐标误差很小（`< LocalIgnoreThreshold`），客户端会**忽略服务端的坐标**，相信自己（防止本地画面拉扯）。如果误差在可接受范围内，则使用 `Vector3.Lerp` 缓慢将本地坐标拉向服务端坐标。

## 3. 动画同步与防滑冰：NetAnimatorView 深度解析

动画同步的难点在于：**网络包有延迟，当客户端收到“开始跑步”的指令时，模型其实已经在往前平移了，这会导致模型像在冰面上滑行一样（脚不动但人往前飘）。**

`NetAnimatorView.cs` 完美解决了这个问题：

### 3.1 防滑冰补偿 (Anti-IceSkating)
- 框架允许你配置 `AntiIceTargetStates`（如 Idle）和 `AntiIceSourceStates`（如 Walk）。
- 当服务端下发状态从 Walk 变为 Idle 时，客户端会检查：**如果模型在空间上还在往前飘（速度 > 0 或 距离目标点还有一段距离），那么强制把动画状态锁死在 Walk！** 直到模型真正停下来，才允许播放 Idle 动画。

### 3.2 动态播放速度 (PlaybackSpeed)
- 录像回放时，如果你按下了“2倍速”，`NetAnimatorView` 会自动将 `Animator.speed` 设为 2，并且将 BlendTree 的参数平滑时间（`ParamSmoothTime`）等比缩短，保证动画表现与空间位移完美匹配。

## 4. 服务端的心跳驱动：ITickableComponent

在服务端，并不是所有的逻辑都是靠“收到客户端网络包”来触发的。很多时候，我们需要服务端自己主动进行运算（比如：怪物巡逻、毒圈缩小、定时广播同步包）。

### 4.1 什么是 ITickableComponent？
只要你的 `ServerRoomComponent` 实现了 `ITickableComponent` 接口，框架的底层引擎就会在每一帧自动调用它的 `OnTick()` 方法。

```csharp
using StellarNet.Lite.Server.Core;

[RoomComponent(999, "MyCustomTick", "自定义心跳组件")]
public sealed class ServerMyCustomComponent : ServerRoomComponent, ITickableComponent
{
    public void OnTick()
    {
        // 这里的代码会以 Config.TickRate 的频率（默认 60Hz）被自动调用
        if (Room.State != RoomState.Playing) return;
        
        // 执行你的定时逻辑，例如：
        // 1. 怪物 AI 寻路
        // 2. 毒圈收缩计算
        // 3. 定时广播全服消息
    }
}
```

### 4.2 它是怎么运转的？
`StellarNetMirrorManager.FixedUpdate` -> `ServerApp.Tick()` -> `Room.Tick()` -> 遍历所有实现了该接口的组件并调用 `OnTick()`。

**注意**：`OnTick` 的执行频率严格受 `NetConfig.TickRate` 控制。在编写逻辑时，请务必使用 `1f / _app.Config.TickRate` 来获取真实的 DeltaTime，而不要依赖 Unity 的 `Time.deltaTime`。
