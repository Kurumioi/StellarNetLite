# 06. StellarNet Lite 零修改框架实现原功能拓展指南

在商业化网络游戏开发中，框架的底层代码（如登录、房间生命周期、状态同步机制）通常经过了严格的测试与边界处理。**直接修改框架源码（`Runtime/` 目录下的内容）会带来极大的维护灾难**（例如：无法平滑升级框架、容易破坏原有的状态机、引发未知的并发 Bug）。

StellarNet Lite 严格遵循 **开闭原则 (OCP)** 与 **MSV 架构**。本文档将通过四个典型的实战案例，向开发者展示如何利用 **“协议拦截与路由转发”** 以及 **“事件生命周期钩子”** 的思想，在**不修改任何一行框架核心源码**的前提下，自由拓展框架的业务能力。

> **💡 核心认知界限**：
> - **不可修改区**：`Assets/StellarNetLite/Runtime/` （框架核心逻辑、底层协议）。
> - **自由修改区**：`Assets/StellarNetLite/GameDemo/` （业务协议、UI 面板、表现层视图、自定义服务端脚本）。

---

## 核心思想：为什么要“拦截与转发”？

框架自带的协议（如 `C2S_Login`, `C2S_SetReady`）就像是“自动闸机”。它们执行标准、底层的核心逻辑（如顶号处理、会话绑定、大厅广播）。
当我们想要在这些标准流程中夹带“私货”（如密码校验、金币扣除）时，我们不应该去拆解“闸机”，而是应该在“闸机”前设立一个**“自定义安检亭”**：

1. **定义新协议**：客户端不再发送框架底层的协议，而是发送带有额外参数的自定义协议。
2. **服务端拦截**：服务端新建一个模块/组件接收该自定义协议，执行自定义的业务逻辑（如查数据库、扣金币）。
3. **伪造与转发**：业务逻辑通过后，服务端在内存中构造一个框架底层的标准协议包，并直接塞给框架的 `Dispatcher`（分发器）。
4. **无缝衔接**：框架底层模块收到转发来的标准包，按常规流程处理并下发标准结果，客户端原有的 UI 逻辑完美闭环。

## 案例一：全局模块拦截 —— 扩展登录参数（密码、地理位置）

**需求**：框架的 `C2S_Login` 仅包含 `AccountId`。我们需要在登录时增加 `Password` 和 `GeoLocation`（地理位置）参数，并在服务端进行校验与存储。

### 1. 定义扩展协议 (Shared)
在您的业务协议目录下，新建一个全局协议，包含所需的所有参数。
```csharp
using System.IO;
using StellarNet.Lite.Shared.Core;
using StellarNet.Lite.Shared.Infrastructure;

namespace StellarNet.Lite.Game.Shared.Protocol
{
    /// <summary>
    /// 自定义高级登录协议。
    /// 设计意图：替代框架底层的 C2S_Login，用于在客户端发起连接时携带额外的业务验证参数。
    /// 采用 NetScope.Global 确保在未进入房间的大厅阶段即可被全局路由解析。
    /// </summary>
    [NetMsg(9001, NetScope.Global, NetDir.C2S)]
    public sealed class C2S_AdvancedLogin : ILiteNetSerializable
    {
        // 基础参数：保留底层所需的必要字段以供后续转发使用
        public string AccountId;
        public string ClientVersion;
        
        // 扩展参数：业务层新增的自定义字段
        public string Password;
        public string GeoLocation; 

        /// <summary>
        /// 实现二进制序列化接口。
        /// 采用显式写入以降低 GC 压力，避免使用反射序列化。
        /// </summary>
        public void Serialize(BinaryWriter writer)
        {
            // 前置拦截空引用，确保网络流写入安全
            writer.Write(AccountId ?? string.Empty);
            writer.Write(ClientVersion ?? string.Empty);
            writer.Write(Password ?? string.Empty);
            writer.Write(GeoLocation ?? string.Empty);
        }
        
        /// <summary>
        /// 实现二进制反序列化接口。
        /// 必须与 Serialize 的写入顺序严格保持一致。
        /// </summary>
        public void Deserialize(BinaryReader reader)
        {
            AccountId = reader.ReadString();
            ClientVersion = reader.ReadString();
            Password = reader.ReadString();
            GeoLocation = reader.ReadString();
        }
    }
}
```

### 2. 服务端拦截与转发 (Server)
新建一个全局模块，拦截请求，校验密码，存储地理位置，随后转发给框架底层的 `ServerUserModule`。
```csharp
using System.Buffers;
using StellarNet.Lite.Server.Core;
using StellarNet.Lite.Shared.Core;
using StellarNet.Lite.Shared.Infrastructure;
using StellarNet.Lite.Shared.Protocol;

namespace StellarNet.Lite.Game.Server.Modules
{
    /// <summary>
    /// 高级鉴权拦截器模块。
    /// 设计意图：作为 C2S_AdvancedLogin 的唯一处理入口。
    /// 负责执行业务层面的密码校验与数据缓存，校验通过后通过内存伪造底层协议触发框架原生登录流程。
    /// </summary>
    [ServerModule("ServerAdvancedAuthModule", "高级鉴权拦截器")]
    public sealed class ServerAdvancedAuthModule
    {
        private readonly ServerApp _app;
        private readonly INetSerializer _serializer;

        public ServerAdvancedAuthModule(ServerApp app)
        {
            _app = app;
            // 实例化轻量级序列化器，专用于内部协议转发时的封包构建
            _serializer = new LiteNetSerializer(); 
        }

        /// <summary>
        /// 拦截客户端发来的高级登录请求。
        /// </summary>
        [NetHandler]
        public void OnC2S_AdvancedLogin(Session session, C2S_AdvancedLogin msg)
        {
            // 1. 业务前置校验：执行密码匹配
            // 此处采用 Early Return 模式，拒绝非法请求
            if (msg.Password != "MySecretPassword")
            {
                // 校验失败时，直接复用框架底层的 S2C_LoginResult 协议下发错误信息
                // 这样客户端无需修改原有的错误处理 UI 逻辑
                _app.SendMessageToSession(session, new S2C_LoginResult { Success = false, Reason = "密码错误或账号不存在" });
                return;
            }

            // 2. 数据缓存：将验证通过的扩展数据存入业务层数据中心
            // 确保后续其他模块（如同城大厅）可以随时获取该玩家的地理位置
            MyCustomPlayerDataCenter.SavePlayerCity(msg.AccountId, msg.GeoLocation);

            // 3. 协议降级与伪造：构造框架底层的标准登录协议
            var baseLoginMsg = new C2S_Login { AccountId = msg.AccountId, ClientVersion = msg.ClientVersion };
            
            // 4. 路由转发魔法：将伪造的协议直接塞入框架全局分发器
            // 设计意图：通过这种方式触发底层的 ServerUserModule，从而无缝复用框架的顶号、Session 绑定等核心机制
            if (NetMessageMapper.TryGetMeta(typeof(C2S_Login), out NetMessageMeta meta))
            {
                // 从对象池租用内存，严格控制 GC 分配
                byte[] buffer = ArrayPool<byte>.Shared.Rent(1024);
                try
                {
                    // 将底层协议序列化为二进制流
                    int len = _serializer.Serialize(baseLoginMsg, buffer);
                    // 构造标准数据包
                    Packet packet = new Packet(0, meta.Id, NetScope.Global, string.Empty, buffer, 0, len);
                    // 核心调用：派发给全局路由
                    _app.GlobalDispatcher.Dispatch(session, packet);
                }
                finally
                {
                    // 确保内存归还，防止内存泄漏
                    ArrayPool<byte>.Shared.Return(buffer);
                }
            }

            // 5. 事件驱动：主动触发同城大厅的推送逻辑
            // 此时底层登录已完成，Session 状态已更新，可以安全地进行业务广播
            ServerRegionalLobbyModule.PushFullList(session);
            ServerRegionalLobbyModule.BroadcastStateChange(session, false);
        }
    }
    
    /// <summary>
    /// 业务层自定义玩家数据中心。
    /// 设计意图：横向扩展玩家状态存储，与框架底层的 Session 数据物理隔离。
    /// </summary>
    public static class MyCustomPlayerDataCenter
    {
        // 采用静态字典缓存玩家账号与城市的映射关系
        private static readonly System.Collections.Generic.Dictionary<string, string> _playerCityMap = new();
        
        public static void SavePlayerCity(string accountId, string city) => _playerCityMap[accountId] = city;
        
        public static string GetPlayerCity(string accountId) => _playerCityMap.TryGetValue(accountId, out var city) ? city : "未知";
    }
}
```

## 案例二：大厅数据隔离 —— 纯事件驱动的同城玩家推送

**需求**：框架默认会广播全服玩家。我们需要基于案例一的 `GeoLocation`，让服务端**自动**向客户端推送同城玩家的全量与增量状态（登录、离线、进出房间），且**不使用轮询，完全复用框架的事件生命周期**。

**思路**：
1. 客户端 UI 忽略框架底层的全服数据，转而监听自定义的同城协议。
2. **登录钩子**：在 `OnC2S_AdvancedLogin` 转发完成后直接调用推送。
3. **断线钩子**：订阅框架暴露的 `StellarNetMirrorManager.OnServerClientDisconnectedEvent`。
4. **房间钩子**：新建一个业务 `RoomComponent` 并配置到房间模板中，利用其生命周期捕获进出房间事件。

### 1. 定义同城玩家同步协议 (Shared)
```csharp
namespace StellarNet.Lite.Game.Shared.Protocol
{
    /// <summary>
    /// 同城玩家基础信息结构体。
    /// </summary>
    public sealed class RegionalPlayerInfo
    {
        public string AccountId;
        public string DisplayName;
        public string City;
        public bool IsOnline;
        public bool IsInRoom;
    }

    /// <summary>
    /// 同城玩家全量列表同步协议。
    /// 设计意图：仅在玩家初次登录成功时下发一次，用于初始化大厅界面。
    /// </summary>
    [NetMsg(9002, NetScope.Global, NetDir.S2C)]
    public sealed class S2C_RegionalPlayerListSync 
    { 
        public RegionalPlayerInfo[] Players; 
    }

    /// <summary>
    /// 同城玩家状态增量同步协议。
    /// 设计意图：当同城内有其他玩家状态发生变化（上线、离线、进出房间）时，定向广播此协议，降低带宽消耗。
    /// </summary>
    [NetMsg(9003, NetScope.Global, NetDir.S2C)]
    public sealed class S2C_RegionalPlayerIncrementalSync 
    { 
        // 标记该玩家是否已彻底断线，供客户端从列表中移除
        public bool IsRemoved; 
        public RegionalPlayerInfo Player; 
    }
}
```

### 2. 服务端同城广播中心与断线钩子 (Server)
新建 `ServerRegionalLobbyModule`，它既是广播中心，也负责监听物理断线事件。
```csharp
using System.Collections.Generic;
using StellarNet.Lite.Server.Core;
using StellarNet.Lite.Shared.Core;
using StellarNet.Lite.Shared.Infrastructure;
using StellarNet.Lite.Game.Shared.Protocol;

namespace StellarNet.Lite.Game.Server.Modules
{
    /// <summary>
    /// 同城大厅事件驱动模块。
    /// 设计意图：作为同城数据的广播中心，并负责挂载底层断线事件钩子。
    /// 彻底抛弃轮询机制，实现纯事件驱动的精准推送。
    /// </summary>
    [ServerModule("ServerRegionalLobbyModule", "同城大厅事件驱动模块")]
    public sealed class ServerRegionalLobbyModule
    {
        // 缓存 ServerApp 实例，供静态广播方法使用
        private static ServerApp _appInstance;

        public ServerRegionalLobbyModule(ServerApp app)
        {
            _appInstance = app;
            // 挂载物理层断线事件钩子
            // 采用先减后加的防御性编程方式，防止在编辑器热重载时发生重复订阅
            StellarNetMirrorManager.OnServerClientDisconnectedEvent -= OnClientDisconnected;
            StellarNetMirrorManager.OnServerClientDisconnectedEvent += OnClientDisconnected;
        }

        /// <summary>
        /// 物理层断线事件回调。
        /// 设计意图：当底层 Transport 检测到连接断开时触发，在此处进行业务层的离线广播。
        /// </summary>
        private void OnClientDisconnected(int connId)
        {
            if (_appInstance == null) return;
            
            // 遍历当前所有 Session，寻找匹配该连接 ID 的玩家
            foreach (var kvp in _appInstance.Sessions)
            {
                if (kvp.Value.ConnectionId == connId)
                {
                    // 找到目标玩家，触发同城离线增量广播
                    BroadcastStateChange(kvp.Value, true); 
                    break;
                }
            }
        }

        /// <summary>
        /// 推送全量同城玩家列表。
        /// 设计意图：由登录拦截器在玩家验证通过后主动调用。
        /// </summary>
        public static void PushFullList(Session targetSession)
        {
            if (_appInstance == null) return;
            
            // 获取目标玩家所在城市
            string myCity = MyCustomPlayerDataCenter.GetPlayerCity(targetSession.AccountId);
            var list = new List<RegionalPlayerInfo>();

            // 遍历全服在线玩家，筛选同城目标
            foreach (var kvp in _appInstance.Sessions)
            {
                if (kvp.Value.IsOnline && MyCustomPlayerDataCenter.GetPlayerCity(kvp.Value.AccountId) == myCity)
                {
                    list.Add(BuildInfo(kvp.Value, myCity));
                }
            }
            
            // 仅向该目标玩家下发全量数据
            _appInstance.SendMessageToSession(targetSession, new S2C_RegionalPlayerListSync { Players = list.ToArray() });
        }

        /// <summary>
        /// 广播玩家状态增量变化。
        /// 设计意图：当玩家上线、离线或进出房间时调用，向同城其他在线玩家推送该玩家的最新状态。
        /// </summary>
        public static void BroadcastStateChange(Session changedSession, bool isRemoved)
        {
            if (_appInstance == null) return;
            
            string city = MyCustomPlayerDataCenter.GetPlayerCity(changedSession.AccountId);
            var syncMsg = new S2C_RegionalPlayerIncrementalSync
            {
                IsRemoved = isRemoved,
                Player = BuildInfo(changedSession, city)
            };

            // 遍历全服，向同一城市的其他在线玩家广播
            foreach (var kvp in _appInstance.Sessions)
            {
                Session s = kvp.Value;
                // 排除自己，且确保目标在线并同城
                if (s.IsOnline && s.SessionId != changedSession.SessionId && MyCustomPlayerDataCenter.GetPlayerCity(s.AccountId) == city)
                {
                    _appInstance.SendMessageToSession(s, syncMsg);
                }
            }
        }

        /// <summary>
        /// 构造同城玩家信息实体。
        /// </summary>
        private static RegionalPlayerInfo BuildInfo(Session session, string city)
        {
            return new RegionalPlayerInfo
            {
                AccountId = session.AccountId,
                DisplayName = string.IsNullOrEmpty(session.Uid) ? "Unknown" : session.Uid,
                City = city,
                IsOnline = session.IsOnline,
                // 根据底层 Session 的 CurrentRoomId 判断是否在房间内
                IsInRoom = !string.IsNullOrEmpty(session.CurrentRoomId)
            };
        }
    }
}
```

### 3. 房间进出钩子 (Server Component)
新建一个业务组件，并在 `GameDemo/Shared/Binders/RoomTypeTemplateRegistry.cs` 中将其 ID 添加到您的房间模板中。

using StellarNet.Lite.Server.Core;
using StellarNet.Lite.Shared.Core;

namespace StellarNet.Lite.Game.Server.Components

    /// <summary>

    }

## 案例三：房间内逻辑拦截 —— 扩展准备校验（检查金币）

**需求**：玩家在房间内点击“准备”时，必须检查其是否拥有足够的入场费（金币）。如果不足，拒绝准备；如果充足，则允许准备。

### 1. 定义扩展协议 (Shared)
```csharp
namespace StellarNet.Lite.Game.Shared.Protocol
{
    /// <summary>
    /// 自定义房间准备请求协议。
    /// 设计意图：替代底层的 C2S_SetReady，用于在房间内触发带有业务校验的准备流程。
    /// 采用 NetScope.Room 确保该协议仅在房间内生效。
    /// </summary>
    [NetMsg(9004, NetScope.Room, NetDir.C2S)]
    public sealed class C2S_CustomSetReady
    {
        public bool IsReady;
    }
}
```

### 2. 服务端房间组件拦截 (Server)
新建一个自定义的房间组件，拦截该请求。
```csharp
using System.Buffers;
using StellarNet.Lite.Server.Components;
using StellarNet.Lite.Server.Core;
using StellarNet.Lite.Shared.Core;
using StellarNet.Lite.Shared.Protocol;

namespace StellarNet.Lite.Game.Server.Components
{
    /// <summary>
    /// 自定义准备校验组件。
    /// 设计意图：拦截玩家的准备请求，执行金币扣除等业务逻辑，
    /// 校验通过后再将请求转发给底层的 ServerRoomSettingsComponent。
    /// </summary>
    [RoomComponent(900, "CustomReadyCheck", "自定义准备校验组件")]
    public sealed class ServerCustomReadyComponent : RoomComponent
    {
        private readonly ServerApp _app;
        private readonly INetSerializer _serializer;

        public ServerCustomReadyComponent(ServerApp app)
        {
            _app = app;
            _serializer = new LiteNetSerializer();
        }

        /// <summary>
        /// 拦截自定义准备请求。
        /// </summary>
        [NetHandler]
        public void OnC2S_CustomSetReady(Session session, C2S_CustomSetReady msg)
        {
            // 1. 状态放行：如果是取消准备，无需校验金币，直接放行
            if (!msg.IsReady)
            {
                ForwardToFrameworkReady(session, false);
                return;
            }

            // 2. 业务校验：执行金币检查逻辑
            int playerGold = GetPlayerGoldFromDatabase(session.AccountId);
            if (playerGold < 100)
            {
                // 校验失败：单独向该玩家下发系统提示协议，阻断准备流程
                Room.SendMessageTo(session, new S2C_SystemPrompt { Message = "金币不足，无法准备！" });
                return;
            }

            // 3. 校验通过：转发给框架底层的准备处理逻辑
            ForwardToFrameworkReady(session, true);
        }

        /// <summary>
        /// 协议转发逻辑。
        /// 设计意图：构造底层 C2S_SetReady 协议并派发给当前房间的 Dispatcher。
        /// </summary>
        private void ForwardToFrameworkReady(Session session, bool isReady)
        {
            var baseMsg = new C2S_SetReady { IsReady = isReady };
            if (NetMessageMapper.TryGetMeta(typeof(C2S_SetReady), out NetMessageMeta meta))
            {
                byte[] buffer = ArrayPool<byte>.Shared.Rent(256);
                try
                {
                    int len = _serializer.Serialize(baseMsg, buffer);
                    // 注意：此处 Scope 必须为 Room，且需传入当前 RoomId
                    Packet packet = new Packet(0, meta.Id, NetScope.Room, Room.RoomId, buffer, 0, len);
                    // 核心调用：塞给当前房间的分发器 (Room.Dispatcher)，而非全局分发器
                    Room.Dispatcher.Dispatch(session, packet);
                }
                finally
                {
                    ArrayPool<byte>.Shared.Return(buffer);
                }
            }
        }

        /// <summary>
        /// 模拟数据库查询接口。
        /// </summary>
        private int GetPlayerGoldFromDatabase(string accountId) => 50; 
    }
}
```

## 案例四：实体状态横向扩展 —— 增加血量与装备同步

**需求**：框架的 `ObjectSyncState` 只同步了 Transform 和 Animator。我们需要同步玩家的血量（HP）和当前手持武器的 ID。
**原则**：**绝对不要去修改底层的 `ObjectSyncState` 结构体！** 状态同步是超高频（20Hz-60Hz）广播，塞入低频业务数据会造成极大的带宽浪费。

### 1. 定义独立的业务同步协议 (Shared)
```csharp
namespace StellarNet.Lite.Game.Shared.Protocol
{
    /// <summary>
    /// 实体战斗数据同步协议。
    /// 设计意图：独立于底层的空间状态同步，专门用于低频的业务属性（如血量、装备）广播。
    /// </summary>
    [NetMsg(9005, NetScope.Room, NetDir.S2C)]
    public sealed class S2C_EntityCombatSync
    {
        // 必须包含 NetId 以便客户端寻址目标实体
        public int NetId;
        public int CurrentHp;
        public int WeaponId;
    }
}
```

### 2. 服务端横向组件 (Server)
新建 `ServerCombatComponent`，独立管理血量，仅在**数值发生变化时**进行单次广播（事件驱动）。
```csharp
using System.Collections.Generic;
using StellarNet.Lite.Server.Core;
using StellarNet.Lite.Shared.Core;

namespace StellarNet.Lite.Game.Server.Components
{
    /// <summary>
    /// 服务端战斗与血量管理组件。
    /// 设计意图：横向扩展房间能力，集中管理房间内所有实体的战斗属性。
    /// 严格遵循事件驱动原则，拒绝在 Update 中轮询广播。
    /// </summary>
    [RoomComponent(901, "Combat", "战斗与血量管理")]
    public sealed class ServerCombatComponent : RoomComponent
    {
        // 独立维护业务数据字典，键为实体的 NetId，值为当前血量
        private readonly Dictionary<int, int> _entityHp = new Dictionary<int, int>();

        /// <summary>
        /// 实体受击处理接口。
        /// </summary>
        public void TakeDamage(int netId, int damage)
        {
            // 容错处理：初始化默认血量
            if (!_entityHp.ContainsKey(netId)) _entityHp[netId] = 100;
            
            // 扣除血量
            _entityHp[netId] -= damage;
            
            // 核心逻辑：仅在数值发生实质性变化（受伤）时，进行单次房间内广播。
            // 这种设计彻底避免了占用底层 Transform 同步的宝贵带宽。
            Room.BroadcastMessage(new S2C_EntityCombatSync 
            { 
                NetId = netId, 
                CurrentHp = _entityHp[netId],
                WeaponId = 1001 // 示例武器 ID
            });
        }
    }
}
```

### 3. 客户端横向组件与表现层解耦 (Client)
客户端新建 `ClientCombatComponent` 接收协议，并通过 `Room.NetEventSystem` 抛出本地事件。UI 脚本或挂载在模型头顶的血条脚本监听该事件并刷新表现。
```csharp
using StellarNet.Lite.Client.Core;
using StellarNet.Lite.Shared.Core;

namespace StellarNet.Lite.Game.Client.Components
{
    /// <summary>
    /// 客户端战斗数据接收组件。
    /// 设计意图：作为网络层与表现层（View）的解耦桥梁。
    /// 本组件仅负责接收网络协议，严禁在此处直接操作 UI 或模型对象。
    /// </summary>
    [RoomComponent(901, "Combat", "客户端战斗数据接收")]
    public sealed class ClientCombatComponent : ClientRoomComponent
    {
        /// <summary>
        /// 拦截服务端下发的战斗同步协议。
        /// </summary>
        [NetHandler]
        public void OnS2C_EntityCombatSync(S2C_EntityCombatSync msg)
        {
            // 核心解耦逻辑：将收到的网络数据结构直接抛给房间级的事件总线。
            // 表现层（如挂载在角色头顶的 UI_HealthBar.cs）只需订阅该事件即可完成刷新。
            // 完美遵循 Model-Service-View 架构规范。
            Room.NetEventSystem.Broadcast(msg);
        }
    }
}
```

## 总结

通过以上四个案例，我们可以得出 StellarNet Lite 框架的高级扩展范式：

1. **修改请求参数/流程**：使用**拦截器模式 (Interceptor)**。新建协议 -> 自定义模块拦截 -> 内存伪造底层包 -> 转发给 `GlobalDispatcher` 或 `Room.Dispatcher`。
2. **替换/过滤下发数据（被动）**：使用**表现层替换模式**。服务端新建协议下发自定义数据 -> 客户端 UI 脚本取消订阅底层协议 -> 改为订阅自定义协议并刷新界面。
3. **自动化推送（主动）**：使用**事件生命周期钩子 (Event Hooks)**。利用拦截器触发登录事件，订阅 `StellarNetMirrorManager` 捕获断线事件，利用自定义 `RoomComponent` 捕获房间进出事件，实现纯事件驱动的自动化推送。
4. **增加实体属性**：使用**横向扩展模式 (Horizontal Extension)**。新建独立的 `RoomComponent` -> 维护独立的数据字典 -> 定义独立的低频同步协议 -> 客户端通过 `NetEventSystem` 解耦刷新 UI。

遵循这些范式，您的游戏业务代码将与框架底层代码彻底物理隔离。您可以随时无痛升级框架版本，同时保持业务代码的绝对安全与整洁。
