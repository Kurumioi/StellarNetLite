# 06. StellarNet Lite 回放重连与框架自定义指南

> 本文对应当前仓库的真实实现。  
> 当前目录分层下，回放相关官方实现位于 `Extensions/Replay`，默认登录/重连流程位于 `Extensions/DefaultGameFlow`。  
> 重点说明三件事：
> 1. 录像回放为什么能 Seek
> 2. 断线重连为什么能恢复房间
> 3. 自定义组件怎样接入快照链路

---

## 1. 回放系统的核心不是“从头重播”

当前仓库的回放系统不是单纯把所有历史消息从第 0 Tick 重播一遍。

它真正依赖的是两类帧：

- 普通消息帧
- 快照帧

快照帧类型定义在：

- `Extensions/Replay/Shared/Replay/ReplaySnapshotDefines.cs`

其中当前使用的快照帧类型是：

```csharp
ReplayFrameKind.ObjectSnapshot
```

注意：

- 名字虽然还是 `ObjectSnapshot`
- 但当前实现已经不是“只给对象同步用”
- 它实际承载的是**通用组件快照帧**

---

## 2. 当前录像快照结构长什么样

最关键的两个类型是：

### 2.1 `ComponentSnapshotData`

```csharp
public struct ComponentSnapshotData
{
    public int ComponentId;
    public byte[] Payload;
}
```

含义：

- `ComponentId`：哪个房间组件导出的快照
- `Payload`：这个组件自己的序列化数据

### 2.2 `ReplaySnapshotFrame`

```csharp
public sealed class ReplaySnapshotFrame : ILiteNetSerializable
{
    public int Tick;
    public ComponentSnapshotData[] ComponentSnapshots;
}
```

含义：

- `Tick`：这个快照对应的录像时间点
- `ComponentSnapshots`：这个时间点所有组件的快照数据块

所以当前录像不是“单一对象快照”，而是：

- 某个 Tick 下的多组件快照集合

---

## 3. 回放 Seek 为什么能快

当前播放端核心在：

- `Extensions/Replay/Client/Core/ClientReplayPlayer.cs`

它启动回放时会做两件关键事：

1. 解析录像头部
2. 为消息帧和快照帧建立索引

当前索引主要包括：

- `_messageFrameIndex`
- `_snapshotFrameIndex`

这意味着 Seek 时不是从头暴力回放，而是：

1. 找到离目标 Tick 最近的快照帧
2. 先恢复该快照帧
3. 再补播后面的少量消息帧

这就是当前回放能支持中段跳转的根本原因。

---

## 4. 回放启动后客户端做了什么

当前 `ClientReplayPlayer.StartReplay()` 大致会做：

1. 校验文件存在
2. 校验当前客户端状态必须是 `InLobby`
3. 初始化录像流，并读取录像头里的原始 `TickRate`
4. 建立索引
5. 重建回放沙盒房间

这里的“沙盒房间”本质上还是一个 `ClientRoom`，只是它的状态是：

- `ClientAppState.ReplayRoom`

所以回放不是“另起一套系统”，而是：

- 沿用客户端房间体系
- 只是消息来源从在线网络改成录像文件
- 播放节奏也按录像记录时的 `TickRate` 驱动，而不是写死某个固定帧率

---

## 5. 断线重连链路当前怎么工作

当前重连逻辑主要涉及：

- `Runtime/StellarNetAppManager.cs`
- `Extensions/DefaultGameFlow/Client/Modules/ClientUserModule.cs`
- `Extensions/DefaultGameFlow/Server/Modules/ServerUserModule.cs`

---

## 6. 客户端断线后不会立刻硬退

如果客户端当前处于：

- `ClientAppState.OnlineRoom`

然后物理连接断开，当前逻辑不会直接清光，而是：

1. 进入 `ConnectionSuspended`
2. 启动自动重试协程
3. 在 15 秒窗口内持续尝试重连

对应代码在：

- `StellarNetAppManager.HandleClientDisconnected()`
- `StellarNetAppManager.ReconnectionRoutine()`

---

## 7. 客户端物理重连成功后会做什么

当底层重新连上后，`StellarNetAppManager.HandleClientConnected()` 会检查：

- 当前是否处于重连态
- `AccountId` 是否存在

如果满足条件，会自动再次发送：

```csharp
new C2S_Login
{
    AccountId = ClientApp.Session.AccountId,
    ClientVersion = Application.version
}
```

也就是说，当前重连不是发一个完全新的专用登录协议，而是：

- 复用现有登录链

---

## 8. 服务端怎么识别这是重连而不是新登录

当前服务端在：

- `ServerUserModule.OnC2S_Login(...)`

会通过 `AccountId` 查旧会话：

```csharp
Session oldSession = _app.GetSessionByAccountId(accountId);
```

如果找到了旧会话，会做下面这些事：

1. 必要时踢掉旧连接
2. 把新连接绑定回旧会话
3. 保留旧 `SessionId`
4. 判断旧房间是否还存在
5. 返回 `HasReconnectRoom`

所以当前重连恢复的关键不在客户端，而在服务端保留了：

- `AccountId -> Session`
- `Session -> CurrentRoomId`

补充：

- 当前 `HasReconnectRoom` 不是只看 `CurrentRoomId` 有没有值。
- 还要同时满足：房间仍然存在、房间未结束、该 `Session` 仍然是房间成员。
- 如果恢复上下文已经失效，服务端会先清理失效状态，而不是继续把它当作可恢复房间。

---

## 9. 客户端拿到 `HasReconnectRoom` 后怎么走

当前客户端在：

- `ClientUserModule.OnS2C_LoginResult(...)`

会这样处理：

### 9.1 如果服务端说有可恢复房间

- 若当前状态是 `ConnectionSuspended`
  直接发送 `C2S_ConfirmReconnect { Accept = true }`

### 9.2 如果服务端说没有可恢复房间

- 清理恢复上下文
- 离开旧房间
- 返回大厅

---

## 10. `C2S_ConfirmReconnect` 和 `C2S_ReconnectReady` 分别做什么

### 10.1 `C2S_ConfirmReconnect`

作用：

- 告诉服务端我是否接受恢复这次房间

服务端处理在：

- `ServerUserModule.OnC2S_ConfirmReconnect(...)`

如果接受，服务端返回：

- `S2C_ReconnectResult`

里面包含：

- `RoomId`
- `ComponentIds`

### 10.2 `C2S_ReconnectReady`

作用：

- 告诉服务端：客户端房间和组件已经重建好了，可以开始补快照了

服务端处理在：

- `ServerUserModule.OnC2S_ReconnectReady(...)`

补充：

- 当前 `C2S_ReconnectReady` 不再只是单向通知。
- 服务端会先返回 `S2C_RoomSetupResult` 作为最终确认。
- 只有在最终确认成功后，才会触发后续快照补发。

它会调用：

```csharp
room.TriggerReconnectSnapshot(session);
```

这一步就是当前重连恢复真正的快照触发点。

## 11. 快照接口是整个回放与重连体系的连接点

当前快照接口定义在：

- `IReplaySnapshotProvider`
- `IReplaySnapshotConsumer`

### 11.1 服务端接口

```csharp
public interface IReplaySnapshotProvider
{
    int SnapshotComponentId { get; }
    byte[] ExportSnapshot();
}
```

作用：

- 导出当前组件的快照字节流

### 11.2 客户端接口

```csharp
public interface IReplaySnapshotConsumer
{
    int SnapshotComponentId { get; }
    void ApplySnapshot(byte[] payload);
}
```

作用：

- 接收服务端或录像文件还原出来的快照

这两个接口是当前框架里最重要的高级扩展点之一。

---

## 12. 默认 `ObjectSync` 是怎么接入快照链路的

默认对象同步组件已经完整接入了这套机制。

服务端：

```csharp
ServerObjectSyncComponent : IReplaySnapshotProvider
```

客户端：

```csharp
ClientObjectSyncComponent : IReplaySnapshotConsumer
```

所以它天然支持：

- 录像 Seek
- 断线重连恢复
- 中途进房的快照补发

补充：

- 当前 `ClientObjectSyncComponent.ApplyReplaySnapshot(...)` 不再只是“补不存在的、删多余的”。
- 对于快照里仍然存在的实体，也会刷新状态。
- 这能保证回放 Seek 和断线重连恢复后，已有对象也会被校正到快照时刻。

---

## 13. 你自己的组件怎么接入快照

只要满足下面两件事就行：

1. 服务端组件实现 `IReplaySnapshotProvider`
2. 客户端组件实现 `IReplaySnapshotConsumer`

同时要保证：

- 两边 `SnapshotComponentId` 一致

### 13.1 服务端示例

```csharp
public sealed class ServerCombatComponent : ServerRoomComponent, IReplaySnapshotProvider
{
    public int SnapshotComponentId => 902;

    public byte[] ExportSnapshot()
    {
        using (var ms = new MemoryStream())
        using (var writer = new BinaryWriter(ms))
        {
            writer.Write(1);
            writer.Write(100);
            return ms.ToArray();
        }
    }
}
```

### 13.2 客户端示例

```csharp
public sealed class ClientCombatComponent : ClientRoomComponent, IReplaySnapshotConsumer
{
    public int SnapshotComponentId => 902;

    public void ApplySnapshot(byte[] payload)
    {
        using (var ms = new MemoryStream(payload))
        using (var reader = new BinaryReader(ms))
        {
            int count = reader.ReadInt32();
            int hp = reader.ReadInt32();
        }
    }
}
```

## 14. 自定义同步系统怎么兼容回放和重连

如果你要完全换掉默认 `ObjectSync`，当前推荐做法不是改底层回放系统，而是：

1. 自己做新的房间组件
2. 自己做新的同步协议
3. 自己实现快照接口

这样你仍然可以复用：

- 房间主链
- 回放主链
- 重连主链

你改的是组件，不是改整套录像系统。

---

## 15. 初学者最容易误解的点

### 15.1 回放不是在线包直接重放

当前实现依赖：

- 索引
- 快照
- 补播

### 15.2 重连不是“继续复用原 socket”

当前是：

- 新物理连接重新建立
- 再绑定回旧会话

### 15.3 只有实现快照接口的组件才能完整参与 Seek / 重连恢复

如果你的组件不实现快照接口，那么：

- 在线时能工作
- 录像 Seek 和断线重连时就无法自动恢复其状态

---

## 16. 一句话总结

当前回放和重连的真正核心不是某个单独模块，而是：

- 房间组件快照接口
- 快照帧索引
- 房间沙盒重建

只要你的组件接入这三件事，它就能进入当前框架的高级恢复体系。
