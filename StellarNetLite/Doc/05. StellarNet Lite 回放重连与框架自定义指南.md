## 1. 为什么回放系统能做到“中段跳转 (Seek)”？

在很多简陋的网络框架中，回放就是“把历史收到的网络包，从第 0 秒开始重新播一遍”。
这种做法有一个致命缺陷：**如果你想快进到第 10 分钟，系统必须在后台把前 10 分钟的包疯狂运算一遍，极度卡顿，且容易出错。**

### StellarNet Lite 的解法：对象关键帧 (Snapshot)
StellarNet Lite 的录像文件（`.replay`）里，不仅存了普通的“消息帧”（比如谁发了表情，谁扣了血），还定期（比如每 150 帧）存了一个 **对象关键帧（ReplayObjectSnapshotFrame）**。

这个关键帧里，保存了当前场景里所有实体的**完整生成态**（位置、血量、动画状态、预制体是谁）。

**当你拖动进度条 Seek 到第 5 分钟时：**
1. 播放器（`ClientReplayPlayer`）会在录像文件里，往回找离第 5 分钟最近的一个“关键帧”。
2. 播放器把当前场景里的怪物全部清空。
3. 播放器根据关键帧，把第 5 分钟时的怪物瞬间全部实例化出来，摆到正确的位置。
4. 然后再从关键帧开始，补播几秒钟的普通消息帧，直到精确对齐你拖动的时间点。

**这就是为什么 StellarNet Lite 的回放跳转如此丝滑且稳定的底层机密！**

## 2. 断线重连是怎么无缝恢复的？

手机玩游戏，进电梯断网了，出来后怎么恢复？

### 2.1 软挂起 (ConnectionSuspended)
当底层网络库检测到断开时，`ClientApp` 不会立刻把玩家踢回登录界面，而是进入一个特殊状态：`ClientAppState.ConnectionSuspended`。
此时，UI 路由会弹出一个半透明的遮罩：“正在尝试恢复连接...”。

### 2.2 顶号与快照定向下发
1. 客户端在后台疯狂重试连接。
2. 连上后，客户端重新发送 `C2S_Login`（带上之前的 AccountId）。
3. 服务端 `ServerUserModule` 发现：“咦，这个账号之前在房间里啊！” 于是把旧的僵尸连接踢掉，把新连接绑定到旧的 `Session` 上。
4. 服务端返回 `HasReconnectRoom = true`。
5. 客户端收到后，发送 `C2S_ConfirmReconnect`（确认重连）。
6. 客户端本地把房间组件重新装配一遍，发送 `C2S_ReconnectReady`。
7. **最关键的一步**：服务端收到 Ready 后，会遍历房间里所有的 `RoomComponent`，调用它们的 `OnSendSnapshot(session)` 方法！

### 2.3 开发者需要做什么？
作为业务开发者，你只需要在你的 `RoomComponent` 里，把 `OnSendSnapshot` 写好就行了！

```csharp
// 服务端组件
public override void OnSendSnapshot(Session session)
{
    // 把当前房间的计分板、怪物血量，打包发给这个刚刚重连上来的玩家
    Room.SendMessageTo(session, new S2C_ScoreSnapshot { TeamAScore = 10, TeamBScore = 5 });
}
```
客户端收到这个快照包，更新本地 UI，玩家就无缝复活了！

## 3. 框架自定义：如何替换底层网络库？

StellarNet Lite 默认使用 Mirror 作为底层传输库。但如果你的主程说：“Mirror 性能不行，我们要换成 KCP 或者 ENet！”
没问题，StellarNet Lite 的核心逻辑与 Mirror 是**完全解耦**的。

### 替换步骤

#### 第一步：实现 `INetworkTransport` 接口
新建一个脚本，接入你喜欢的网络库（比如 LiteNetLib 或 KCP），并实现框架要求的 6 个方法：

```csharp
using StellarNet.Lite.Shared.Core;
using System;

public class MyCustomTransport : INetworkTransport
{
    public void SendToServer(Packet packet) {
        // 把 packet.Payload 发给服务端
    }

    public void SendToClient(int connectionId, Packet packet) {
        // 把 packet.Payload 发给指定的客户端
    }

    public void DisconnectClient(int connectionId) {
        // 踢掉某个客户端
    }

    public void StopServer() { /* 关服 */ }
    public void StopClient() { /* 关客户端 */ }
    public float GetRTT() { return 0.05f; /* 返回延迟 */ }
}
```

#### 第二步：在收到数据时，喂给框架
当你的网络库收到字节流时，组装成 `Packet`，交给框架的 `App` 处理：

```csharp
// 客户端收到数据时
Packet p = new Packet(seq, msgId, scope, roomId, bytes, length);
ClientApp.OnReceivePacket(p);

// 服务端收到数据时
Packet p = new Packet(seq, msgId, scope, roomId, bytes, length);
ServerApp.OnReceivePacket(connectionId, p);
```

#### 第三步：替换启动入口
在你的 `GameLauncher` 中，不要再使用 `StellarNetMirrorManager`，而是实例化你自己的 `MyCustomTransport`，并在创建 `ServerApp` 和 `ClientApp` 时把它传进去。

**搞定！整个 StellarNet Lite 的房间管理、回放系统、UI 路由、对象同步，将完美运行在你的新网络库之上！**

## 4. 框架自定义：如何替换 JSON 序列化？

框架默认对普通协议使用 JSON（Newtonsoft），对高频协议使用二进制（`ILiteNetSerializable`）。
如果你想全局使用 Protobuf 或 MemoryPack 以追求极致性能：

实现 `INetSerializer` 接口：
```csharp
public class MyProtobufSerializer : INetSerializer
{
    public byte[] Serialize(object obj) {
        // 用 Protobuf 序列化
    }

    public int Serialize(object obj, byte[] buffer) {
        // 用 Protobuf 序列化到内存池 buffer 中
    }

    public object Deserialize(byte[] data, int offset, int length, Type type) {
        // 用 Protobuf 反序列化
    }
}
```
然后在 `ServerApp` 和 `ClientApp` 初始化时，把你的 Serializer 传进去即可。
