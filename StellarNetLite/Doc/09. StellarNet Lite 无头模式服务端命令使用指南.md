# 09. StellarNet Lite 无头模式服务端命令使用指南

本文档用于说明 StellarNet Lite 在服务端无头模式下的控制台命令、日志策略、常见排查方法与使用建议。

---

## 1. 适用范围

本文对应的无头服务端控制台能力由以下脚本提供：

- `Runtime/Server/Infrastructure/HeadlessServerLogger.cs`

适用场景：

- Windows 无头服务端
- Linux 无头服务端
- 云服务器、物理机、容器内的纯服务端运行
- 不依赖 Unity 图形界面的命令行排障

---

## 2. 什么是无头模式

无头模式指服务端在没有图形渲染设备，或以批处理模式启动时运行。

当前脚本会在以下条件成立时自动启用命令行监听：

- `SystemInfo.graphicsDeviceType == Null`
- `Application.isBatchMode == true`

启用后，服务端会：

- 启动标准输入命令监听
- 接收你在控制台输入的命令
- 用 ASCII 表格输出状态、房间、会话和日志结果

---

## 3. 日志策略

当前无头服务端日志策略是“只记录报错类日志”。

会被记录到 `ServerLog_*.txt` 的类型：

- `Error`
- `Assert`
- `Exception`

不会写入日志文件的内容：

- 普通 `Info`
- 普通 `Warning`
- 无头控制台命令本身输出的表格内容

日志目录：

- `ServerLogs/`

日志文件命名格式：

- `ServerLog_yyyyMMdd_HHmmss.txt`

保留策略：

- 启动时自动清理 30 天以前的旧日志文件
- 只保留近 30 天内的日志文件

内存缓存策略：

- 仅缓存最近 `10000` 条错误日志
- 用于 `logs` 和 `findlog` 命令快速查询

注意：

- 如果某个日志文件是空的，通常说明该服务端进程启动后没有产生错误日志
- 这属于当前设计预期，不是异常

---

## 4. 命令总览

以下命令均在无头服务端控制台中直接输入。

| 命令 | 用途 |
| --- | --- |
| `help` | 查看全部命令说明 |
| `status` | 查看服务端运行状态总览 |
| `rooms` | 查看当前所有房间列表 |
| `room <roomId>` | 查看指定房间详细信息 |
| `sessions` | 查看当前所有会话列表 |
| `session <sessionId\|accountId>` | 查看指定会话详细信息 |
| `kick <sessionId\|accountId> [reason]` | 踢掉指定会话 |
| `logs [count]` | 查看最近若干条错误日志 |
| `findlog <keyword> [limit]` | 按关键字搜索近 30 天错误日志 |
| `logfiles` | 查看近 30 天错误日志文件清单 |
| `gc` | 手动触发一次服务端 GC Tick |
| `exit` | 逐步清理服务端状态并退出进程 |

---

## 5. 命令详细说明

### 5.1 `help`

作用：

- 打印所有可用命令和简要说明

示例：

```bash
help
```

建议使用时机：

- 刚连上服务器控制台时
- 忘记某条命令参数格式时

---

### 5.2 `status`

作用：

- 查看当前服务端整体运行状态

典型输出内容：

- 启动时间
- 运行时长
- 当前传输层类型
- 端口
- TickRate
- 房间数
- 会话数
- 在线/离线会话数
- 当前日志文件路径
- 近 30 天日志文件数量
- 内存错误日志缓存占用情况

示例：

```bash
status
```

建议使用时机：

- 判断服务端是否真正启动完成
- 快速确认当前在线规模
- 确认日志文件是否已创建

---

### 5.3 `rooms`

作用：

- 列出所有当前活动房间

典型字段：

- `RoomId`
- `Name`
- `State`
- `Members`
- `Online`
- `Tick`
- `AgeMin`
- `Components`

示例：

```bash
rooms
```

建议使用时机：

- 查看是否存在僵尸房间
- 判断某房间是否正在运行
- 检查房间组件是否装配正确

---

### 5.4 `room <roomId>`

作用：

- 查看单个房间的详细信息

会输出三部分内容：

- 房间摘要信息
- 组件清单
- 房间成员清单

示例：

```bash
room 8a4fc2d1
```

适合排查：

- 某个房间为何无法进入
- 房间成员是否在线
- 房间是否处于 `Waiting / Playing / Finished`
- 房间组件 Id 与展示名是否匹配

---

### 5.5 `sessions`

作用：

- 查看当前所有会话

典型字段：

- `SessionId`
- `AccountId`
- `ConnId`
- `Online`
- `RoomId`
- `Ready`
- `LastSeq`
- `ActiveAgoSec`
- `OfflineMin`

示例：

```bash
sessions
```

适合排查：

- 在线人数不对
- 玩家掉线后是否仍残留会话
- 某玩家是否卡在未 Ready 状态

---

### 5.6 `session <sessionId|accountId>`

作用：

- 查看单个会话详情

支持两种查询方式：

- 直接传 `SessionId`
- 直接传 `AccountId`

示例：

```bash
session 2fd1f6f4d6b34e8c9f30f3f7fbe8c521
```

```bash
session player_10001
```

输出重点：

- 当前连接号
- 在线状态
- 当前房间
- 已授权目标房间
- 房间 Ready 状态
- 最近业务活跃时间
- 最近房间业务活跃时间

如果出现多条模糊匹配结果，控制台会列出候选项，要求你改用更精确的值。

---

### 5.7 `kick <sessionId|accountId> [reason]`

作用：

- 强制断开指定会话
- 如果该会话当前在房间中，会先从房间移除
- 如房间因此为空，房间会被销毁
- 在线会话会收到 `S2C_KickOut`

示例：

```bash
kick player_10001
```

```bash
kick player_10001 Duplicate login
```

```bash
kick 2fd1f6f4d6b34e8c9f30f3f7fbe8c521 Manual admin cleanup
```

建议：

- 优先使用精确 `SessionId`
- 如果使用 `AccountId`，确保当前只有一个明确目标

---

### 5.8 `logs [count]`

作用：

- 查看最近缓存的错误日志

参数：

- `count` 可选，表示显示多少条
- 默认值为 `20`

示例：

```bash
logs
```

```bash
logs 50
```

说明：

- 这里只显示当前进程内存里缓存的错误日志
- 如果服务端刚重启，内存缓存不会自动带上旧进程的历史错误

---

### 5.9 `findlog <keyword> [limit]`

作用：

- 在“当前进程内存错误日志 + 近 30 天日志文件”中搜索关键字

参数：

- `keyword` 必填，关键字
- `limit` 可选，返回条数上限，默认 `20`

示例：

```bash
findlog timeout
```

```bash
findlog SessionId:abc123 30
```

```bash
findlog "room not found" 10
```

返回结果：

- 先输出摘要表格
- 再依次输出每条命中的完整错误内容

适合排查：

- 某个玩家相关报错
- 某个 `RoomId` 相关异常
- 某个协议或模块名相关异常

---

### 5.10 `logfiles`

作用：

- 查看近 30 天保留的错误日志文件清单

输出内容：

- 文件名
- 最后更新时间
- 文件年龄
- 文件大小

示例：

```bash
logfiles
```

适合排查：

- 判断日志是否持续写入
- 判断最近几天是否频繁生成错误文件
- 快速定位应该打开哪个日志文件

---

### 5.11 `gc`

作用：

- 手动执行一次服务端 `Tick()`，让房间/会话 GC 逻辑立即跑一轮

示例：

```bash
gc
```

注意：

- 这个命令不是 .NET 垃圾回收
- 它触发的是 `ServerApp` 内部的房间与会话生命周期清理逻辑

适合排查：

- 某个离线会话是否应该被清掉
- 某个空房间是否应该被回收

---

### 5.12 `exit`

作用：

- 发起服务端优雅停机
- 结束进行中的房间对局
- 向在线会话发送停服通知
- 停止传输层与服务端逻辑宿主
- 最终退出服务端进程

示例：

```bash
exit
```

当前执行顺序：

1. 锁定停机流程，避免重复执行
2. 汇总当前房间数、对局中房间数、会话数、在线会话数
3. 对所有 `Playing` 状态房间执行结束对局
4. 向所有在线会话发送 `S2C_KickOut`，原因是 `Server shutting down.`
5. 等待极短时间让停服通知尽量发出
6. 调用 `StopServer()`
7. 等待 `ServerApp.Dispose()` 清理完成
8. 退出进程

注意：

- 这是“优雅退出”，不是立即强制杀进程
- 如果服务端当前没有启动 `ServerApp`，则会直接退出应用
- 如果你重复输入 `exit`，后续输入会被忽略

适合使用场景：

- 运维主动停服
- 发布新版本前的正常关服
- 需要在退出前尽量完成房间结束与资源清理

---

## 6. 参数输入规则

命令参数支持空格分隔。

如果参数中本身包含空格，请使用双引号包裹：

```bash
findlog "serialization failed"
```

```bash
kick player_10001 "Manual cleanup by operator"
```

---

## 7. 推荐使用流程

### 场景一：先判断服务端是否正常

建议顺序：

1. `status`
2. `rooms`
3. `sessions`

目的：

- 看服务端是否存活
- 看房间/会话数量是否符合预期

---

### 场景二：某玩家进不去房间

建议顺序：

1. `session <accountId>`
2. `room <roomId>`
3. `findlog <accountId>`
4. `findlog <roomId>`

目的：

- 看会话是否在线
- 看房间是否存在
- 看是否有明确报错

---

### 场景三：怀疑存在僵尸房间或残留会话

建议顺序：

1. `rooms`
2. `sessions`
3. `gc`
4. 再执行一次 `rooms`
5. 再执行一次 `sessions`

目的：

- 观察 GC 前后状态是否变化

---

### 场景四：排查近期错误

建议顺序：

1. `logs 50`
2. `findlog <关键词>`
3. `logfiles`

目的：

- 先看最近错误
- 再做关键字检索
- 最后定位具体文件

---

## 8. 常见问题

### 8.1 为什么 `ServerLog_*.txt` 是空的

常见原因：

- 当前进程运行期间没有产生错误日志
- 当前问题只产生了普通信息输出，不属于错误

这在当前“只记录错误日志”的策略下是正常现象。

---

### 8.2 为什么控制台里能看到表格，但 txt 里没有

因为控制台命令输出使用的是 `Console.WriteLine`，它用于人机交互，不属于错误日志持久化内容。

---

### 8.3 为什么搜不到旧日志

原因通常有两类：

- 该日志超过 30 天，已经被清理
- 搜索关键字和实际错误内容不一致

建议：

- 使用更稳定的关键字，如 `RoomId`、`SessionId`、模块名、协议名、异常短语

---

### 8.4 为什么 `session accountId` 会提示多个匹配

这是因为当前会话模糊匹配到了多条结果。

解决方式：

- 改用完整 `SessionId`
- 或输入更完整的 `AccountId`

---

## 9. 维护建议

建议运营或运维同学至少掌握以下命令：

- `status`
- `rooms`
- `sessions`
- `logs 50`
- `findlog`
- `logfiles`

建议研发排查线上问题时优先使用以下组合：

- `status` + `sessions`
- `room <roomId>`
- `session <sessionId|accountId>`
- `findlog <keyword>`

---

## 10. 结论

当前无头模式服务端已经具备以下能力：

- 通过控制台直接查看服务端运行状态
- 快速查看房间与会话明细
- 手动执行会话踢除和生命周期 GC
- 只持久化错误类日志
- 支持近 30 天错误日志留存与关键字排查

如果后续还要继续扩展无头命令，建议保持以下原则：

- 命令名短且稳定
- 输出以表格为主
- 默认优先服务于排障
- 对危险操作给出明确参数格式与结果反馈
