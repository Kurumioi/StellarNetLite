# 12. StellarNet Lite 无头模式服务端命令使用指南

本文档用于说明 StellarNet Lite 在服务端无头模式下的控制台命令、日志策略、异步持久化观察能力、常见排查方法与使用建议。

---

## 1. 适用范围

本文对应的无头服务端控制台能力由以下脚本提供：

- `Runtime/Server/Infrastructure/HeadlessServerLogger.cs`

适用场景：

- Windows 无头服务端
- Linux 无头服务端
- 云服务器、物理机、容器内的纯服务端运行
- 不依赖 Unity 图形界面的命令行排障

---

## 2. 什么是无头模式

无头模式指服务端在没有图形渲染设备，或以批处理模式启动时运行。

当前脚本会在以下条件成立时自动启用命令行监听：

- `SystemInfo.graphicsDeviceType == Null`
- `Application.isBatchMode == true`

启用后，服务端会：

- 启动标准输入命令监听
- 接收你在控制台输入的命令
- 用 ASCII 表格输出状态、房间、会话、日志和持久化任务结果

补充：

- 当前控制台输入读取仍然是后台阻塞式 `Console.ReadLine`
- 但命令执行本身已经统一投递到 `UnityPlayerLoopDispatcher`
- 不再依赖 `HeadlessServerLogger.Update()` 轮询命令队列

---

## 3. 日志策略

当前无头服务端日志策略是“只记录报错类日志”。

会被记录到 `ServerLog_*.txt` 的类型：

- `Error`
- `Assert`
- `Exception`

不会写入日志文件的内容：

- 普通 `Info`
- 普通 `Warning`
- 无头控制台命令本身输出的表格内容

日志目录：

- `ServerLogs/`

日志文件命名格式：

- `ServerLog_yyyyMMdd_HHmmss.txt`

保留策略：

- 启动时自动清理 30 天以前的旧日志文件
- 只保留近 30 天内的日志文件

内存缓存策略：

- 仅缓存最近 `10000` 条错误日志
- 用于 `logs` 和 `findlog` 命令快速查询

注意：

- 如果某个日志文件是空的，通常说明该服务端进程启动后没有产生错误日志
- 这属于当前设计预期，不是异常

---

## 4. 命令总览

以下命令均在无头服务端控制台中直接输入。

| 命令 | 用途 |
| --- | --- |
| `help` | 查看全部命令说明 |
| `status` | 查看服务端运行状态总览 |
| `rooms` | 查看当前所有房间列表 |
| `room <roomId>` | 查看指定房间详细信息 |
| `sessions` | 查看当前所有会话列表 |
| `session <sessionId\|accountId>` | 查看指定会话详细信息 |
| `kick <sessionId\|accountId> [reason]` | 踢掉指定会话 |
| `logs [count]` | 查看最近若干条错误日志 |
| `findlog <keyword> [limit]` | 按关键字搜索近 30 天错误日志 |
| `logfiles` | 查看近 30 天错误日志文件清单 |
| `persist` | 查看当前已登记的异步持久化任务 |
| `pending` | `persist` 的别名 |
| `gc` | 手动触发一次服务端 GC Tick |
| `exit` / `quit` | 等待持久化任务、清理服务端状态并退出进程 |

---

## 5. 命令详细说明

### 5.1 `help`

作用：

- 打印所有可用命令和简要说明

示例：

```bash
help
```

---

### 5.2 `status`

作用：

- 查看当前服务端整体运行状态

典型输出内容：

- 启动时间
- 运行时长
- 当前传输层类型
- 端口
- TickRate
- 房间数
- 会话数
- 在线/离线会话数
- 当前持久化待处理任务数量
- 当前 `JoinOnShutdown` 待处理任务数量
- 当前是否已经进入持久化 shutdown 模式
- 当前日志文件路径
- 近 30 天日志文件数量
- 内存错误日志缓存占用情况

示例：

```bash
status
```

---

### 5.3 `rooms`

作用：

- 列出所有当前活动房间

典型字段：

- `RoomId`
- `Name`
- `State`
- `Members`
- `Online`
- `Tick`
- `AgeMin`
- `Components`

示例：

```bash
rooms
```

---

### 5.4 `room <roomId>`

作用：

- 查看单个房间的详细信息

会输出三部分内容：

- 房间摘要信息
- 组件清单
- 房间成员清单

示例：

```bash
room 8a4fc2d1
```

---

### 5.5 `sessions`

作用：

- 查看当前所有会话

典型字段：

- `SessionId`
- `AccountId`
- `ConnId`
- `Online`
- `RoomId`
- `Ready`
- `LastSeq`
- `ActiveAgoSec`
- `OfflineMin`

示例：

```bash
sessions
```

---

### 5.6 `session <sessionId|accountId>`

作用：

- 查看单个会话详情

支持两种查询方式：

- 直接传 `SessionId`
- 直接传 `AccountId`

示例：

```bash
session 2fd1f6f4d6b34e8c9f30f3f7fbe8c521
```

```bash
session player_10001
```

---

### 5.7 `kick <sessionId|accountId> [reason]`

作用：

- 强制断开指定会话
- 如果该会话当前在房间中，会先从房间移除
- 如房间因此为空，房间会被销毁
- 在线会话会收到 `S2C_KickOut`

示例：

```bash
kick player_10001
```

```bash
kick player_10001 Duplicate login
```

---

### 5.8 `logs [count]`

作用：

- 查看最近缓存的错误日志

参数：

- `count` 可选，表示显示多少条
- 默认值为 `20`

示例：

```bash
logs
```

```bash
logs 50
```

说明：

- 这里只显示当前进程内存里缓存的错误日志
- 如果服务端刚重启，内存缓存不会自动带上旧进程的历史错误

---

### 5.9 `findlog <keyword> [limit]`

作用：

- 在“当前进程内存错误日志 + 近 30 天日志文件”中搜索关键字

参数：

- `keyword` 必填，关键字
- `limit` 可选，返回条数上限，默认 `20`

示例：

```bash
findlog timeout
```

```bash
findlog "room not found" 10
```

返回结果：

- 先输出摘要表格
- 再依次输出每条命中的完整错误内容

---

### 5.10 `logfiles`

作用：

- 查看近 30 天保留的错误日志文件清单

输出内容：

- 文件名
- 最后更新时间
- 文件年龄
- 文件大小

示例：

```bash
logfiles
```

---

### 5.11 `persist`

作用：

- 查看当前所有已登记的异步持久化任务
- 观察每个任务是否会在 `exit` 时阻塞停机

典型字段：

- `TaskId`
- `Owner`
- `Operation`
- `JoinOnShutdown`
- `StartUtc`
- `ElapsedSec`

示例：

```bash
persist
```

```bash
pending
```

说明：

- 只有通过 `ServerApp.PersistenceRuntime.TrackAsync(...)` 登记的任务才会出现在这里
- 没有登记的异步操作，不会被无头服务端感知，也不会在 `exit` 时等待

---

### 5.12 `gc`

作用：

- 手动执行一次服务端 `Tick()`，让房间/会话 GC 逻辑立即跑一轮

示例：

```bash
gc
```

注意：

- 这个命令不是 .NET 垃圾回收
- 它触发的是 `ServerApp` 内部的房间与会话生命周期清理逻辑

---

### 5.13 `exit` / `quit`

作用：

- 发起服务端优雅停机
- 先等待已登记的异步持久化任务
- 再结束进行中的房间对局
- 再向在线会话发送停服通知
- 最终停止服务端并退出进程

示例：

```bash
exit
```

当前执行顺序：

1. 锁定停机流程，避免重复执行。
2. 调用 `ServerApp.PersistenceRuntime.BeginShutdown()`，进入持久化 shutdown 模式。
3. 默认阻止新的普通持久化任务启动。
4. 如果存在 `JoinOnShutdown = true` 的待处理任务，则循环等待，并按表格输出当前仍在执行的任务。
5. 持久化任务排空后，统计当前房间数、对局中房间数、会话数、在线会话数。
6. 对所有 `Playing` 状态房间执行结束对局。
7. 向所有在线会话发送 `S2C_KickOut`。
8. 调用 `StopServer()` 和 `StopClient()`。
9. 等待 `ServerApp.Dispose()` 清理完成。
10. 退出进程。

注意：

- `exit` 等待的是“已登记且 `JoinOnShutdown = true`”的任务
- 如果开发者显式使用 `allowStartDuringShutdown = true`，那么停机阶段仍可继续启动新的持久化任务
- 这类任务只要还处于 pending 状态，就会持续显示在等待表格中

---

## 6. 参数输入规则

命令参数支持空格分隔。

如果参数中本身包含空格，请使用双引号包裹：

```bash
findlog "serialization failed"
```

```bash
kick player_10001 "Manual cleanup by operator"
```

## 7. 推荐使用流程

### 场景一：先判断服务端是否正常

建议顺序：

1. `status`
2. `rooms`
3. `sessions`

目的：

- 看服务端是否存活
- 看房间和会话数量是否符合预期

---

### 场景二：排查某个玩家进不去房间

建议顺序：

1. `session <accountId>`
2. `room <roomId>`
3. `findlog <accountId>`
4. `findlog <roomId>`

目的：

- 看会话是否在线
- 看房间是否存在
- 看是否有明确报错

---

### 场景三：排查持久化任务是否还在执行

建议顺序：

1. `status`
2. `persist`

目的：

- 先看总体 pending 数量
- 再看具体是哪一个模块、哪一个操作还没完成

---

### 场景四：停服前确认是否会被异步任务阻塞

建议顺序：

1. `persist`
2. `exit`

目的：

- 先看当前是否已有待落盘、待加载、待上报任务
- 再执行优雅停机

---

### 场景五：排查近期错误

建议顺序：

1. `logs 50`
2. `findlog <关键词>`
3. `logfiles`

目的：

- 先看最近错误
- 再做关键字检索
- 最后定位具体文件

---

## 8. 常见问题

### 8.1 为什么 `ServerLog_*.txt` 是空的

常见原因：

- 当前进程运行期间没有产生错误日志
- 当前问题只产生了普通信息输出，不属于错误

这在当前“只记录错误日志”的策略下是正常现象。

---

### 8.2 为什么控制台里能看到表格，但 txt 里没有

因为控制台命令输出使用的是 `Console.WriteLine`，它用于人机交互，不属于错误日志持久化内容。

---

### 8.3 为什么 `persist` 里是空的

常见原因：

- 当前没有正在执行的持久化任务
- 业务代码虽然做了异步操作，但没有通过 `ServerApp.PersistenceRuntime.TrackAsync(...)` 登记

---

### 8.4 为什么 `exit` 会卡在等待阶段

原因通常是：

- 仍有 `JoinOnShutdown = true` 的任务未完成
- 某个开发者自定义的 JSON / MySQL / WebAPI 保存逻辑本身还在运行

此时请先看控制台等待表格，确认是哪个 `Owner` 和 `Operation` 卡住了。

---

## 9. 维护建议

建议运营或运维同学至少掌握以下命令：

- `status`
- `rooms`
- `sessions`
- `logs 50`
- `findlog`
- `persist`
- `exit`

建议研发排查线上问题时优先使用以下组合：

- `status` + `persist`
- `room <roomId>`
- `session <sessionId|accountId>`
- `findlog <keyword>`

---

## 10. 结论

当前无头模式服务端已经具备以下能力：

- 通过控制台直接查看服务端运行状态
- 快速查看房间与会话明细
- 手动执行会话踢除和生命周期 GC
- 只持久化错误类日志
- 支持近 30 天错误日志留存与关键字排查
- 支持查看当前异步持久化 pending 任务
- 支持在 `exit` 时等待已登记的持久化任务完成

如果后续还要继续扩展无头命令，建议保持以下原则：

- 命令名短且稳定
- 输出以表格为主
- 默认优先服务于排障
- 对停机相关操作明确说明等待行为和阻塞来源
