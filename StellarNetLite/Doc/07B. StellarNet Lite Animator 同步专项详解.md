# 07B. StellarNet Lite Animator 同步专项详解

本文专门讲 `ObjectSync` 里的 `Animator` 同步。

如果你已经知道：

1. 实体怎么出生
2. `NetIdentity` 是什么
3. `NetTransformView` 是怎么工作的

那就可以继续看这篇。

这篇会重点讲清楚：

1. 服务端到底要不要写动画逻辑
2. `NetAnimatorView` Inspector 每个参数是做什么的
3. 为什么现在改成了“可视化映射 + 动态参数列表”
4. BlendTree、多参数、回放、防滑步是怎么协同工作的
5. 新手如何从 0 配好一个同步动画实体

## 1. Animator 同步到底在同步什么

Animator 同步不是“把 Unity Animator 整个复制到网络里”。

框架真正同步的是三类信息：

1. 当前逻辑动画状态
   - 例如 `Idle`
   - `Walk`
   - `Run`
   - `Attack`
2. 当前动画播放进度
   - `AnimNormalizedTime`
3. 当前动画 float 参数
   - 例如 `MoveSpeed`
   - `MoveX`
   - `MoveY`
   - `UpperBodyBlend`

你可以把它理解成：

服务端只同步“足够还原表现”的最小动画事实，而不是同步整个 Animator 控制器内部运行状态。

---

## 2. 这套 Animator 同步的设计原则

### 2.1 服务端权威

决定实体当前应该播什么动画的，是服务端业务逻辑。

不是客户端自己想播什么就播什么。

### 2.2 客户端负责表现

客户端的 `NetAnimatorView` 负责：

1. 把服务端逻辑状态映射到 Unity `Animator` 状态
2. 把服务端逻辑参数映射到 Unity float 参数
3. 做平滑过渡
4. 做防滑步
5. 做回放时的时间补偿

### 2.3 逻辑名与 Animator 名解耦

服务端写的是逻辑状态名和逻辑参数名，例如：

```csharp
entity.SetAnimState("Move");
entity.SetAnimParam("MoveSpeed", speed);
```

客户端可以把它映射到任意 Animator 资源命名：

1. 逻辑状态名 `Move`
   映射到 Animator 状态 `Locomotion`
2. 逻辑参数名 `MoveSpeed`
   映射到 Animator float 参数 `Speed`

这就是新方案最重要的升级点之一。

## 3. 从 0 做一个可同步动画实体

这部分按新手实操来讲。

### 3.1 先准备一个能正常本地播放的 Animator

先不要急着接网络。

先保证你本地已有：

1. `Animator` 组件
2. 一个可工作的 `Animator Controller`
3. 里面有状态机、BlendTree、float 参数

必须先满足：

1. 不接网络时，本地手动调参数也能正常切状态
2. 状态名字、参数名字明确

因为 `NetAnimatorView` 只是“驱动已有 Animator”，不是替你生成 Animator 资源。

### 3.2 在 Prefab 上挂组件

动画同步实体常见最小配置：

1. `NetIdentity`
2. `NetTransformView`
3. `NetAnimatorView`
4. `Animator`

一般情况：

1. `NetAnimatorView` 挂在根节点
2. `TargetAnimator` 指向角色模型上的 `Animator`

如果你不手动指定 `TargetAnimator`，脚本会尝试：

`GetComponentInChildren<Animator>()`

自动查找。

### 3.3 配置 `SourceAnimatorController`

`NetAnimatorView` Inspector 里有：

`SourceAnimatorController`

作用：

1. 给编辑器扫描用
2. 自动读取所有 Animator 状态名
3. 自动读取所有 float 参数名
4. 做可视化映射和校验

如果你没手动指定，而 `TargetAnimator` 已经有 Controller，脚本会自动从：

`TargetAnimator.runtimeAnimatorController`

补进来。

### 3.4 用 Inspector 映射状态

现在的设计不是手敲一个字符串数组，而是可视化列表。

你会看到：

`状态同步映射`

其中每一项都有：

1. 是否启用
2. 逻辑状态名
3. 对应 Animator 状态

最简单配置方式是：

1. `LogicStateName = Idle`
   `AnimatorStateName = Idle`
2. `LogicStateName = Walk`
   `AnimatorStateName = Walk`
3. `LogicStateName = Run`
   `AnimatorStateName = Run`

如果逻辑名与 Animator 状态名不同，也完全可以：

1. `LogicStateName = Move`
2. `AnimatorStateName = Locomotion`

### 3.5 用 Inspector 映射 float 参数

你会看到：

`Float 参数同步映射`

每一项也有：

1. 是否启用
2. 逻辑参数名
3. 对应 Animator float 参数名

例如：

1. `LogicParamName = MoveSpeed`
   `AnimatorParamName = Speed`
2. `LogicParamName = MoveX`
   `AnimatorParamName = MoveX`
3. `LogicParamName = MoveY`
   `AnimatorParamName = MoveY`

这就是对 BlendTree 最友好的新方案。

因为不再限制只有 3 个固定槽位了。

### 3.6 服务端写动画逻辑

服务端不是写 `animator.Play(...)`。

服务端是写：

```csharp
entity.SetAnimState("Idle");
entity.SetAnimState("Walk");
entity.SetAnimParam("MoveSpeed", speed);
```

这一步必不可少。

因为客户端 `NetAnimatorView` 只是“接收并表现”。

如果服务端不写逻辑动画状态和参数，客户端没有任何可同步内容。

## 4. `EntitySyncMask.Animator` 有什么作用

定义文件：

`ObjectSyncProtocols.cs`

它表示：

“这个实体具备动画同步能力。”

如果实体需要动画同步，生成时必须带上：

```csharp
EntitySyncMask.Animator
```

或者直接用：

```csharp
EntitySyncMask.All
```

如果没开这个位：

1. 服务端即使写了 `AnimStateHash`
2. 客户端也不会按 Animator 同步链路去处理

## 5. 协议层里 Animator 同步了哪些字段

### 5.1 `ObjectSpawnState`

定义文件：

`Extensions/ObjectSync/Shared/Protocol/Components/ObjectSync/ObjectSpawnState.cs`

Animator 相关字段有：

#### `AnimStateHash`

当前逻辑动画状态哈希。

#### `AnimNormalizedTime`

当前动画归一化播放时间。

#### `AnimParamCount`

当前同步参数数量。

#### `AnimParams`

动态参数数组。

数组元素类型是：

`AnimatorParamValue`

每一项包含：

1. `ParamHash`
2. `Value`

### 5.2 `ObjectSyncState`

定义文件：

`Extensions/ObjectSync/Shared/Protocol/Components/ObjectSync/ObjectSyncProtocols.cs`

Animator 增量同步相关字段同样是：

1. `AnimStateHash`
2. `AnimNormalizedTime`
3. `AnimParamCount`
4. `AnimParams`

### 5.3 `ObjectSyncDirtyMask` 里 Animator 相关位

#### `AnimState`

当前动画状态变了。

#### `AnimNormalizedTime`

当前动画时间进度变了。

#### `AnimParams`

当前同步参数列表变了。

#### `AllAnimator`

组合位：

```csharp
AnimState | AnimNormalizedTime | AnimParams
```

作用：

1. 出生全量恢复
2. 周期性全量重同步
3. 回放快照恢复

## 6. 服务端动画同步到底要写什么

这是最常被误解的地方。

答案是：

要写，而且必须写。

但写的是“逻辑动画状态”和“逻辑参数”，不是写 Unity `Animator` API。

## 7. `ServerSyncEntity` 和服务端动画写法

核心文件：

`Extensions/ObjectSync/Server/Components/ServerObjectSyncComponent.cs`

### 7.1 `ServerSyncEntity` 里和动画有关的字段

#### `AnimStateHash`

当前逻辑动画状态哈希。

#### `AnimNormalizedTime`

当前逻辑动画时间。

#### `AnimParams`

当前逻辑参数数组。

#### `AnimParamCount`

当前逻辑参数数量。

### 7.2 `SetAnimState(string logicStateName, float normalizedTime = 0f)`

作用：

1. 把逻辑状态名哈希成稳定哈希
2. 写入 `AnimStateHash`
3. 写入 `AnimNormalizedTime`

新手可以把它理解成：

“告诉客户端，接下来该播哪个逻辑状态。”

### 7.3 `SetAnimParam(string logicParamName, float value)`

作用：

1. 把逻辑参数名转成稳定哈希
2. 写入动态参数数组

### 7.4 `SetAnimParam(int logicParamHash, float value)`

作用：

1. 用哈希形式直接写参数
2. 会在数组中按哈希升序插入

这里的“按哈希排序插入”非常重要。

它解决的问题是：

如果同一组参数只是写入顺序不同，而值本身没变，旧方案可能会被误判成“参数变了”，导致：

1. 不必要的脏数据
2. 高频重复广播
3. 回放记录冗余

现在按哈希稳定排序后：

1. 相同参数集合有稳定顺序
2. `HasAnimParamMeaningfulChange(...)` 才能做正确比较

### 7.5 `ClearAnimParams()`

作用：

1. 把当前逻辑参数集合清空
2. `AnimParamCount = 0`

它很重要，因为有些状态切换后，原来那组 float 参数已经不应该继续生效。

例如：

1. 从 `Move` 进入 `Attack`
2. 之前的 `MoveSpeed`、`MoveX`、`MoveY` 不该永远残留

服务端可以根据业务选择：

1. 继续保留某些参数
2. 或明确 `ClearAnimParams()`

## 8. 一个实用的服务端动画模板

下面给你一个适合大多数新手起步的模板思路。

### 8.1 纯状态切换型

适合：

1. `Idle`
2. `Walk`
3. `Attack`
4. `Hit`
5. `Death`

服务端逻辑伪代码：

```csharp
if (isDead)
{
    entity.ClearAnimParams();
    entity.SetAnimState("Death");
}
else if (isHit)
{
    entity.ClearAnimParams();
    entity.SetAnimState("Hit");
}
else if (isAttacking)
{
    entity.ClearAnimParams();
    entity.SetAnimState("Attack");
}
else if (speed > 0.01f)
{
    entity.ClearAnimParams();
    entity.SetAnimState("Walk");
}
else
{
    entity.ClearAnimParams();
    entity.SetAnimState("Idle");
}
```

### 8.2 BlendTree 驱动型

适合：

1. `Idle/Walk/Run` 混合
2. 八方向移动
3. 上下半身混合权重

服务端逻辑伪代码：

```csharp
entity.SetAnimState("Move");
entity.SetAnimParam("MoveSpeed", speed);
entity.SetAnimParam("MoveX", inputX);
entity.SetAnimParam("MoveY", inputY);
```

如果进入非移动状态，例如攻击：

```csharp
entity.ClearAnimParams();
entity.SetAnimState("Attack");
```

### 8.3 什么时候该 `ClearAnimParams()`

建议原则：

1. 进入新的大状态，而且旧参数已无意义时，就清
2. 继续使用同一套 BlendTree 时，不要清

典型需要清的场景：

1. `Move -> Attack`
2. `Move -> Hit`
3. `Move -> Death`

典型不需要清的场景：

1. 同一个 `Move` BlendTree 内继续更新速度方向
2. 同一 locomotion 状态下只改变参数值

## 9. `NetAnimatorView` Inspector 每个参数是什么意思

核心文件：

`Extensions/ObjectSync/Client/Views/NetAnimatorView.cs`

### `TargetAnimator`

真正要被驱动的 Unity `Animator`。

如果为空，脚本会尝试自动找子节点 Animator。

建议：

1. 角色根节点与模型 Animator 分离时，手动拖进去
2. 结构简单时也可以用自动查找

### `SourceAnimatorController`

编辑器扫描源。

作用：

1. 自动列出状态名
2. 自动列出 float 参数名
3. 做校验
4. 支持 OverrideController 的基础控制器扫描

### `ParamSmoothTime`

参数平滑时间。

默认：

`0.05`

作用：

1. float 参数不是每次硬改
2. 用 `SmoothDamp` 逐步逼近目标值

这个值越小：

1. 参数越跟手
2. 但也越容易显得生硬

这个值越大：

1. 参数越柔和
2. 但切换速度会拖

### `AnimCrossFadeTime`

状态切换 CrossFade 时间。

默认：

`0.15`

作用：

1. 服务端状态变了以后
2. 客户端不是硬切，而是做固定时长的 CrossFade

### `EnableAntiIceSkating`

是否启用防滑步。

建议：

大多数角色开启。

### `AntiIceTargetStates`

防滑步目标状态列表。

通常填写“停步类状态”。

例如：

1. `Idle`

### `AntiIceSourceStates`

防滑步来源状态列表。

通常填写“仍可能有位移惯性的移动状态”。

例如：

1. `Walk`
2. `Run`

### `SyncedStates`

状态映射列表。

每项类型是：

`AnimatorStateSyncEntry`

字段：

1. `Enabled`
   这条映射是否参与同步
2. `LogicStateName`
   服务端使用的逻辑状态名
3. `AnimatorStateName`
   客户端 Animator 里的真实状态名

### `SyncedFloatParams`

参数映射列表。

每项类型是：

`AnimatorParamSyncEntry`

字段：

1. `Enabled`
   这条映射是否参与同步
2. `LogicParamName`
   服务端使用的逻辑参数名
3. `AnimatorParamName`
   客户端 Animator 里的真实 float 参数名

## 10. 为什么 BlendTree 不再只有 3 个参数限制

以前的固定 `FloatParam1/2/3` 方案问题很明显：

1. 不够表达复杂 BlendTree
2. 参数名全靠手敲，容易拼错
3. 服务端和客户端字段耦合太死
4. 多状态多参数场景扩展性很差

现在改成：

`AnimatorParamValue[] + AnimParamCount + 可视化映射`

就解决了这些问题。

你现在可以同步任意数量的 float 参数，只要：

1. 服务端有写
2. 客户端有映射

所以多个 BlendTree 不再是问题。

因为本质上它们都只是：

“不同状态下使用的不同 float 参数集合”

## 11. `NetAnimatorViewEditor` 现在做了什么

核心文件：

`Editor/NetAnimatorViewEditor.cs`

这是本次易用性升级里非常重要的一层。

### 11.1 自动扫描

编辑器会从 `SourceAnimatorController` 或 `TargetAnimator` 推导出控制器后，自动扫描：

1. 所有 Animator 状态名
2. 所有 float 参数名

### 11.2 可视化列表

它会把扫描结果直接渲染成：

1. `状态同步映射`
2. `Float 参数同步映射`

你不需要再自己盲敲列表。

### 11.3 刷新按钮

Inspector 提供：

1. `刷新状态映射`
2. `刷新参数映射`
3. `全部刷新`
4. `清理无效映射`

适合在以下场景使用：

1. Animator Controller 改名后
2. 新增状态后
3. 新增 float 参数后
4. 清理废弃配置时

### 11.4 校验会提示什么

当前校验会重点抓这些问题：

1. 逻辑状态名重复
2. 逻辑参数名重复
3. 启用的状态映射缺失逻辑名
4. Animator 里不存在这个状态
5. Animator 里不存在这个 float 参数
6. 防滑步列表里填了未知状态
7. 一个状态都没启用，导致服务端状态无法分发

这正是你之前担心的那类问题：

1. 参数名拼错
2. 状态名拼错
3. 服务端发过来但客户端接不住

现在大部分都能在 Inspector 层先暴露出来。

## 12. `NetAnimatorView` 的关键缓存字段都在干什么

这部分偏源码理解，但我还是写给新手看。

### `_logicToAnimatorStateHashes`

逻辑状态哈希 -> Animator 状态哈希 映射表。

作用：

1. 服务端发来逻辑状态哈希
2. 客户端靠它找到真正要播的 Animator 状态

### `_logicToAnimatorParamHashes`

逻辑参数哈希 -> Animator 参数哈希 映射表。

作用：

1. 服务端发来逻辑参数哈希
2. 客户端找到真正要写入的 Animator float 参数

### `_animatorFloatParamHashes`

当前 Animator 里全部 float 参数哈希集合。

作用：

1. 校验映射目标是否真的是一个存在的 float 参数
2. 避免错误写入

### `_antiIceTargetHashes`

停步类状态哈希集合。

### `_antiIceSourceHashes`

移动类状态哈希集合。

这两个集合配合起来判断：

“当前是否应该拦住一次过早的停步切换。”

### `_receivedAnimatorParamHashes`

当前这一帧从服务端实际收到的参数集合。

作用：

1. 帮助判断哪些参数本帧没有发
2. 没发到的参数可能要回到初始值

### `_initialAnimatorParamValues`

记录 Animator 初始 float 默认值。

作用：

1. 服务端不再发送某个参数时
2. 客户端知道应该往哪个默认值回退

### `_currentParamValues`

客户端当前正在平滑中的参数值缓存。

### `_paramVelocities`

每个参数做 `SmoothDamp` 时的速度缓存。

作用：

1. 保证参数平滑连续
2. 避免每帧都像第一次插值那样抖

### `_unknownSyncedStateHashes`

记录那些收到了但没映射成功的状态哈希。

作用：

1. 只打一次警告
2. 避免控制台每帧刷屏

### `_unknownSyncedParamHashes`

同理，用于未映射参数哈希。

## 13. `NetAnimatorView` 每个关键函数在做什么

### `Awake()`

作用：

1. 缓存 `NetIdentity`
2. 自动补 `TargetAnimator`
3. 做旧配置迁移
4. 初始化运行时映射缓存

### `Start()`

再次确保运行时初始化完成。

作用是兜底，防止某些引用在 `Awake` 后才准备好。

### `OnValidate()`

编辑器下改 Inspector 时触发。

作用：

1. 自动补引用
2. 执行旧配置迁移
3. 清掉运行时缓存，让映射重新构建

### `HardSetInitialState(int animHash, float normalizedTime, AnimatorParamValue[] animParams, int animParamCount)`

实体刚出生时使用。

作用：

1. 先把服务端动画状态哈希映射成 Animator 状态哈希
2. 立即 `Play(...)`
3. 把参数立刻应用到 Animator

为什么要“立刻”：

因为第一次生成时，正确性比平滑更重要。

### `Update()`

每帧入口。

流程：

1. 检查引用
2. 如果是在线房间本地拥有实体，则跳过远端覆盖
3. 从 `ClientObjectSyncComponent` 取 `PredictedAnimatorData`
4. 调用 `ProcessAnimatorSync(...)`

### `ProcessAnimatorSync(ref PredictedAnimatorData syncData)`

这是动画同步核心。

它做三件事：

1. 处理状态切换
2. 处理参数写入
3. 处理 Animator 播放速度

### `ApplyAnimatorPlaybackSpeed(float playbackSpeed)`

作用：

1. 根据当前播放速度设置 `TargetAnimator.speed`
2. 如果 Transform 还在轻微追赶，会临时加一点速度

这里的思想是：

如果物体位置还在追赶服务器，而动画已经早早停住，就会显得脚下打滑。

所以这里会做一点配套补偿。

### `ShouldBlockTransitionIntoStopState(int targetHash)`

这是防滑步的关键判断。

逻辑大意：

1. 如果目标状态属于停步类状态
2. 当前状态属于移动类状态
3. 并且 Transform 层判断物体还明显在移动或还没对齐
4. 那就先别切到停步状态

这样可以避免：

1. 人物看起来还在往前滑
2. 动画却已经提前 Idle 了

### `ShouldTransitionTo(int targetHash)`

作用：

1. 判断这次是不是确实需要切状态
2. 避免重复对同一状态反复 CrossFade

### `ApplyAnimatorParams(AnimatorParamValue[] animParams, int animParamCount, bool instantApply, float playbackSpeed)`

作用：

1. 把服务端参数逐个解析映射
2. 即时应用或平滑应用
3. 记录本帧收到过哪些参数

### `ResetMissingAnimatorParams(bool instantApply, float effectiveSmoothTime)`

这是这次方案里很重要的稳定性修复点。

它解决的问题是：

如果服务端之前发过某个参数，后来不发了，客户端不能让旧值永久残留。

例如：

1. 之前 `MoveSpeed = 1`
2. 现在进入攻击状态，不再发 `MoveSpeed`
3. 如果客户端不回退，这个值会卡在旧值
4. BlendTree 可能继续维持错误姿态

现在的做法是：

1. 本帧没收到的已映射参数
2. 回退到 Animator 初始默认值
3. 可以瞬时回退，也可以平滑回退

这就是为什么它对 BlendTree 非常关键。

### `EnsureRuntimeInitialized()`

作用：

1. 只初始化一次运行时映射
2. 构建状态映射、参数映射、防滑步哈希集合等

### `BuildStateMappings()`

根据 `SyncedStates` 建立：

逻辑状态哈希 -> Animator 状态哈希

### `BuildAnimatorFloatParamSet()`

扫描 Animator 里的 float 参数，并记录：

1. 参数哈希
2. 初始默认值

### `BuildParamMappings()`

根据 `SyncedFloatParams` 建立：

逻辑参数哈希 -> Animator 参数哈希

### `BuildAntiIceHashes()`

把防滑步的状态名列表转换成哈希集合。

### `ResolveAnimatorStateHash(int serverOrAnimatorHash)`

作用：

1. 先判断收到的哈希是否已经是 Animator 状态哈希
2. 如果不是，再去逻辑映射表里找
3. 找不到就打一条未映射警告

### `ResolveAnimatorParamHash(int logicParamHash, out int animatorParamHash)`

作用：

1. 把逻辑参数哈希映射到实际 Animator float 参数
2. 顺便校验该参数在 Animator 中确实存在

### `AutoAssignAnimatorReferences()`

作用：

1. 自动补 `TargetAnimator`
2. 自动补 `SourceAnimatorController`

### `MigrateLegacyConfigIfNeeded()`

作用：

1. 把老版本的 `SyncedStateNames`
2. 以及旧的 `FloatParam1Name/2Name/3Name`
3. 自动迁移成新的可视化映射结构

这对旧项目升级非常重要。

### `TryAddLegacyParamMapping(string legacyParamName)`

把旧单个参数名补进新的动态参数映射表里，并避免重复。

### `ResetRuntimeCache()`

作用：

1. Inspector 配置变化后清空旧缓存
2. 让下次重新构建映射

## 14. 动画不通过连线怎么控制切换和过渡

这个问题要分清两个层面。

### 14.1 逻辑切换由服务端决定

服务端决定：

1. 现在是 `Idle`
2. 还是 `Walk`
3. 还是 `Attack`

### 14.2 过渡表现由客户端 Animator 决定

客户端用：

```csharp
TargetAnimator.CrossFadeInFixedTime(...)
```

来做表现层过渡。

也就是说：

1. 服务端决定“目标状态”
2. 客户端决定“怎么平滑过去”

所以即使你不完全依赖 Animator 图里的复杂连线逻辑，也可以正常工作。

但建议是：

1. Animator Controller 里仍然保留合理状态结构
2. `NetAnimatorView` 负责网络层驱动
3. 不要把所有状态切换判断都塞到客户端本地条件里

否则服务端权威就会被削弱。

## 15. 防滑步到底在防什么

“滑步”指的是：

1. 角色位置还在明显移动
2. 动画却已经提前停到了 `Idle`

这样看起来就像脚在地上滑。

当前方案里，防滑步由三部分共同完成：

1. `NetTransformView` 负责位置追赶
2. `NetAnimatorView.ShouldBlockTransitionIntoStopState(...)`
   负责延后停步状态
3. `ApplyAnimatorPlaybackSpeed(...)`
   在轻微追赶期间给动画速度一点补偿

这三层配合起来，目的就是：

1. 保持动画与位移一致
2. 避免远端角色突然“脚停了但身子还飘”

## 16. 为什么这套 Animator 同步对回放也友好

因为同步的是：

1. 状态
2. 参数
3. 归一化时间

而不是同步一堆临时 Unity 本地内部状态。

这带来几个好处：

1. 快照可恢复
2. 时间轴可重放
3. 回放倍速时仍可控制
4. 客户端和回放播放器用的是同一套表现逻辑

## 17. 参数应该怎么配

### 17.1 普通角色推荐起点

1. `ParamSmoothTime = 0.03 ~ 0.08`
2. `AnimCrossFadeTime = 0.1 ~ 0.18`
3. `EnableAntiIceSkating = true`
4. `AntiIceTargetStates = Idle`
5. `AntiIceSourceStates = Walk, Run`

### 17.2 动作游戏偏灵敏风格

1. `ParamSmoothTime` 更小
2. `AnimCrossFadeTime` 更短

### 17.3 社交 / 展示型风格

1. `ParamSmoothTime` 可略大
2. `AnimCrossFadeTime` 可略大

## 18. 新手最常见错误与排查

### 18.1 服务端写了状态，客户端不切动画

优先检查：

1. 实体 `Mask` 是否包含 `Animator`
2. `SyncedStates` 是否启用了对应逻辑状态名
3. `AnimatorStateName` 在 Controller 里是否真实存在
4. 控制台是否出现 unmapped state hash 警告

### 18.2 BlendTree 参数没反应

优先检查：

1. 服务端是否真的调用了 `SetAnimParam(...)`
2. `SyncedFloatParams` 是否映射了这个逻辑参数
3. 对应 Animator 参数是否是 `Float`
4. 参数名是否拼错

### 18.3 攻击后角色一直保持旧移动姿态

优先检查：

1. 是否应该在切攻击时 `ClearAnimParams()`
2. 是否开启了 `ResetMissingAnimatorParams(...)` 这条逻辑
3. Animator 默认参数值是否本身就不合理

### 18.4 控制台不断报未映射参数或状态

优先检查：

1. 服务端逻辑名是否和客户端映射表一致
2. 是否改了 Animator 状态名却没刷新映射
3. 是否删了参数但映射表还保留着旧配置

### 18.5 停下后人物还是有点滑

优先检查：

1. `EnableAntiIceSkating` 是否开启
2. `AntiIceTargetStates` 是否正确包含停步状态
3. `AntiIceSourceStates` 是否正确包含移动状态
4. 服务端位移停止时是否真的把 `Velocity` 归零

### 18.6 本地玩家看起来没被网络动画驱动

这是当前设计使然。

在线房间里，本地拥有实体会跳过远端覆盖，以保证本地控制响应。

这意味着：

1. 本地玩家的最终表现，通常应由本地控制逻辑直接驱动
2. 服务端同步主要用于远端观察者、权威状态和回放

## 19. 你现在应该记住的几件事

1. 服务端必须写动画逻辑状态和参数，只是写法不是 Unity `Animator` API，而是 `SetAnimState / SetAnimParam`。
2. 客户端 `NetAnimatorView` 只是一个表现层映射器，它不替代服务端决策。
3. 新的可视化映射方案已经支持任意数量 float 参数，BlendTree 不再被 3 个固定槽位限制。
4. `ResetMissingAnimatorParams(...)` 和参数有序插入，是这次稳定性和生产可用性的关键修复点。
5. 动画同步要想自然，必须和 `Transform` 同步一起看，所以本文最好配合 `07A` 一起掌握。
