# 11. StellarNet Lite 服务端异步持久化工具使用指南

本文档用于说明 StellarNet Lite 新增的“服务端异步持久化工具”应该怎么理解、怎么接入，以及开发者如何基于它自由实现 JSON、MySQL、WebAPI 或其他存储策略。

---

## 1. 工具定位

这次新增的不是一套完整持久化框架，而是一个最小基础工具。

代码位置：

- `Runtime/Server/Core/ServerApp.cs`
- `Runtime/Server/Infrastructure/ServerPersistenceRuntime.cs`
- `Runtime/Server/Infrastructure/HeadlessServerLogger.cs`

它只负责三件事：

- 跟踪当前有哪些异步持久化任务正在执行
- 提供 pending 快照，方便无头模式和研发排查
- 在无头模式执行 `exit` 时，等待已登记的任务先完成

它不负责：

- 不定义玩家金币、地图、角色、背包等业务结构
- 不规定你必须用 JSON、MySQL、Redis、WebAPI 还是别的方案
- 不规定存档路径、加载路径、分表方式、分文件方式
- 不规定你一定要在什么业务时机触发保存或加载

也就是说，框架只给“工具”，业务策略完全由开发者自己定。

---

## 2. 设计思路

如果把持久化做成一整套统一接口，框架会很快变重。

当前版本故意保持轻量：

- 全局模块想持久化，就自己调用工具
- 房间业务组件想持久化，也自己调用工具
- 加载什么、保存什么、存到哪、怎么序列化，都由业务模块自己决定

这个设计最适合当前 StellarNet Lite 的定位：

- 保持 Runtime 纯净
- 不强绑某种数据库
- 不破坏已有模块和房间组件的开发方式

---

## 3. 接入点

现在每个 `ServerApp` 都自带一个持久化运行时：

```csharp
public ServerPersistenceRuntime PersistenceRuntime { get; } = new ServerPersistenceRuntime();
```

所以不管你写的是全局模块还是房间组件，只要能拿到 `ServerApp`，就能直接用：

```csharp
_app.PersistenceRuntime.TrackAsync(...);
```

无头模式下还新增了两个配套能力：

- `status` 会显示当前 pending 数量和 shutdown 状态
- `persist` / `pending` 会列出当前所有已登记的异步任务
- `exit` 会先等待 `JoinOnShutdown = true` 的任务完成，再继续停服

---

## 4. 核心 API 说明

### 4.1 `TrackAsync(...)`

最常用方法：

```csharp
Task TrackAsync(
    string owner,
    string operation,
    Func<Task> work,
    bool joinOnShutdown = true,
    bool allowStartDuringShutdown = false)
```

含义：

- `owner`：任务归属者，建议写模块名或组件名，例如 `GoldModule`
- `operation`：任务名，建议写成 `save-gold:player_10001`
- `work`：真正的异步加载或保存逻辑
- `joinOnShutdown`：是否在 `exit` 时阻塞停机等待，默认 `true`
- `allowStartDuringShutdown`：服务端已经进入 shutdown 模式时，是否仍允许启动该任务，默认 `false`

如果你需要接收 `CancellationToken`，也可以使用带 token 的重载。

---

### 4.2 `SnapshotPending()`

用于查看当前还没结束的任务：

```csharp
IReadOnlyList<PersistencePendingInfo> pending = _app.PersistenceRuntime.SnapshotPending();
```

每条 `PersistencePendingInfo` 里包含：

- `Id`
- `Owner`
- `Operation`
- `StartUtc`
- `JoinOnShutdown`
- `Elapsed`

这个能力适合：

- 无头控制台查看
- 调试界面查看
- 自定义后台监控输出

---

### 4.3 `BeginShutdown()`

进入 shutdown 模式：

```csharp
_app.PersistenceRuntime.BeginShutdown();
```

进入后：

- 默认不允许新的普通持久化任务再启动
- 已经在跑的任务会继续执行
- 如果业务代码明确设置 `allowStartDuringShutdown = true`，仍可以启动新任务

当前框架中，这一步由无头模式 `exit` 自动调用。

---

### 4.4 `WaitForIdleAsync(...)`

用于等待需要参与停机排空的任务结束：

```csharp
PersistenceDrainResult result = await _app.PersistenceRuntime.WaitForIdleAsync(TimeSpan.FromSeconds(5));
```

返回值里包含：

- `Completed`
- `Pending`
- `PendingCount`

注意：

- 它等待的是 `JoinOnShutdown = true` 的任务
- `JoinOnShutdown = false` 的任务不会阻塞停机

---

## 5. 推荐使用原则

建议按下面思路接入：

- 只要某个异步保存或加载需要被框架感知，就用 `TrackAsync(...)`
- 需要在 `exit` 时确保落盘完成的任务，保持默认 `joinOnShutdown = true`
- 不想阻塞停机的后台任务，可以显式传 `joinOnShutdown = false`
- 不要让业务代码偷偷开异步任务却不登记，否则 `exit` 无法等待

## 6. 完整案例：金币全局模块使用 JSON 持久化

下面给一个完整案例。

场景：

- 维护一个“账号 -> 金币数量”的字典
- 每个账号一个 JSON 文件
- 模块初始化或玩家上线时按路径加载
- 金币变更后按路径异步保存
- 这些加载和保存都通过 `PersistenceRuntime` 登记

这只是示例。

你完全可以把里面的 JSON 文件替换成：

- 单个大 JSON
- MySQL
- Redis
- WebAPI
- 云存档服务

---

### 6.1 示例数据结构

```csharp
[Serializable]
public sealed class GoldArchive
{
    public int Amount;
}
```

---

### 6.2 示例模块骨架

```csharp
using System;
using System.Collections.Generic;
using System.IO;
using System.Text;
using System.Threading.Tasks;
using StellarNet.Lite.Server.Core;
using UnityEngine;

public sealed class ServerGoldModule
{
    private readonly ServerApp _app;
    private readonly Dictionary<string, int> _goldByAccount = new Dictionary<string, int>();
    private readonly string _goldRootPath;

    public ServerGoldModule(ServerApp app, string goldRootPath)
    {
        _app = app;
        _goldRootPath = goldRootPath;
    }

    public int GetGold(string accountId)
    {
        if (string.IsNullOrWhiteSpace(accountId))
        {
            return 0;
        }

        return _goldByAccount.TryGetValue(accountId, out int value) ? value : 0;
    }

    public async Task LoadGoldAsync(string accountId)
    {
        if (string.IsNullOrWhiteSpace(accountId))
        {
            return;
        }

        await _app.PersistenceRuntime.TrackAsync(
            "GoldModule",
            $"load-gold:{accountId}",
            async () =>
            {
                string path = BuildAccountPath(accountId);
                if (!File.Exists(path))
                {
                    _goldByAccount[accountId] = 0;
                    return;
                }

                using (var stream = new FileStream(path, FileMode.Open, FileAccess.Read, FileShare.Read, 4096, true))
                using (var reader = new StreamReader(stream, Encoding.UTF8))
                {
                    string json = await reader.ReadToEndAsync();
                    GoldArchive archive = JsonUtility.FromJson<GoldArchive>(json);
                    _goldByAccount[accountId] = archive != null ? archive.Amount : 0;
                }
            });
    }

    public async Task SaveGoldAsync(string accountId)
    {
        if (string.IsNullOrWhiteSpace(accountId))
        {
            return;
        }

        int amount = GetGold(accountId);
        await _app.PersistenceRuntime.TrackAsync(
            "GoldModule",
            $"save-gold:{accountId}",
            async () =>
            {
                string path = BuildAccountPath(accountId);
                string directory = Path.GetDirectoryName(path);
                if (!string.IsNullOrEmpty(directory) && !Directory.Exists(directory))
                {
                    Directory.CreateDirectory(directory);
                }

                string json = JsonUtility.ToJson(new GoldArchive { Amount = amount }, true);
                byte[] bytes = Encoding.UTF8.GetBytes(json);
                using (var stream = new FileStream(path, FileMode.Create, FileAccess.Write, FileShare.None, 4096, true))
                {
                    await stream.WriteAsync(bytes, 0, bytes.Length);
                }
            });
    }

    public async Task AddGoldAsync(string accountId, int delta)
    {
        if (!_goldByAccount.ContainsKey(accountId))
        {
            await LoadGoldAsync(accountId);
        }

        _goldByAccount[accountId] = GetGold(accountId) + delta;
        await SaveGoldAsync(accountId);
    }

    private string BuildAccountPath(string accountId)
    {
        return Path.Combine(_goldRootPath, $"{accountId}.json");
    }
}
```

---

### 6.3 这个案例体现了什么

这个案例里，框架只提供了以下基础能力：

- 记录 `load-gold:*`
- 记录 `save-gold:*`
- 让 `persist` 命令能看到它们
- 让 `exit` 能等待它们

但这些内容都还是开发者自己决定的：

- 路径怎么拼
- 一个账号一个文件，还是全部写到一个文件
- 文件格式是不是 JSON
- 是加载单人档，还是加载整张表
- 金币何时保存，是即时保存、延迟保存还是定时保存

这正是当前工具的目标。

---

### 6.4 如果换成 MySQL 或 WebAPI

你只需要替换 `TrackAsync(...)` 里的异步主体。

比如：

- JSON 版本里，主体是 `FileStream` 读写
- MySQL 版本里，主体可以是 `ExecuteNonQueryAsync`
- WebAPI 版本里，主体可以是 `HttpClient.PostAsync`

外层调用方式不变：

```csharp
await _app.PersistenceRuntime.TrackAsync(
    "GoldModule",
    $"save-gold:{accountId}",
    async () =>
    {
        await UploadToWebApiAsync(accountId, amount);
    });
```

所以真正的“策略模式”仍然掌握在开发者自己手里。

## 7. 房间业务组件怎么用

房间组件也是同样思路。

例如你有一个“地图存档组件”，想在房间结束前异步保存地图状态：

```csharp
await _app.PersistenceRuntime.TrackAsync(
    "MapArchiveComponent",
    $"save-room-map:{Room.RoomId}",
    async () =>
    {
        string path = BuildRoomMapPath(Room.RoomId);
        await SaveRoomMapJsonAsync(path);
    });
```

如果你有一个“角色组件”，想在玩家进入房间时加载角色存档：

```csharp
await _app.PersistenceRuntime.TrackAsync(
    "RoleComponent",
    $"load-role:{accountId}",
    async () =>
    {
        await LoadRoleFromDatabaseAsync(accountId);
    });
```

你会发现：

- 不需要框架额外定义房间组件专用持久化接口
- 不需要框架规定房间组件必须实现某个生命周期函数
- 只要能拿到 `ServerApp`，就能使用同一个工具

## 8. `persist` 和 `exit` 的实际效果

当业务代码使用 `TrackAsync(...)` 后：

- `persist` 会显示当前任务列表
- `status` 会显示 pending 统计
- `exit` 会先调用 `BeginShutdown()`
- 然后等待所有 `JoinOnShutdown = true` 的任务结束

如果任务一直没结束，控制台会继续打印等待中的任务表格。

这能直接回答两个问题：

- 现在到底有没有保存任务还没完成
- 是哪个模块、哪个操作在阻塞停机

---

## 9. 常见使用建议

建议：

- `owner` 尽量写稳定的模块名或组件名
- `operation` 尽量带上关键业务标识，例如账号、房间号、地图号
- 真正需要落盘保证的任务，保留默认 `joinOnShutdown = true`
- 不重要的后台上报任务，可以显式设为 `joinOnShutdown = false`

不建议：

- 自己偷偷 `Task.Run(...)` 然后不登记
- 在 shutdown 之后还无限制地继续创建新持久化任务
- 把特别宽泛的字符串作为 `operation`，导致排查时看不出具体业务

---

## 10. 总结

这套工具的核心价值不是替你决定“怎么存”，而是替框架解决两个基础问题：

- 框架怎么知道现在有哪些异步持久化任务还没结束
- 服务端停机时怎么优雅地等这些任务先完成

因此更准确地说，它是：

- 一个异步持久化任务托管器
- 一个停机等待器
- 一个无头模式可观察性工具

至于 JSON、MySQL、WebAPI、分文件、分表、分片、定时刷盘、脏标记批量落盘，这些都应该继续留给开发者按业务自由实现。
