# 10. StellarNet Lite 零修改框架实现原功能拓展指南

> 本文只讲一个主题：  
> **尽量不改 `Runtime/` 核心源码，也能把业务做出来。**

当前目录分层下，框架骨架保留在 `Runtime`，官方默认流程与可复用能力放在 `Extensions`，示例业务放在 `Samples`。

当前仓库最推荐的扩展方式有三种：

1. 拦截并转发已有流程
2. 横向新增独立模块或房间组件
3. 替换默认组件，但保留框架主链

---

## 1. 为什么尽量不要直接改 Runtime

`Runtime/` 目录承担的是框架骨架：

- 协议路由
- 会话管理
- 房间管理
- `ClientApp / ServerApp / Factory`
- 传输层与消息分发基类

而当前官方默认流程、房间设置、对象同步、回放、网络监控已经收敛到了 `Extensions/`。

直接改这里的代价通常很大：

1. 后续升级框架很难合并
2. 容易破坏已有状态机
3. Bug 会落在全局公共链路上
4. 业务代码和框架代码会越缠越死

所以更推荐的方式是：

- 协议自己扩
- 模块自己加
- 组件自己挂
- 必要时再把请求转回框架已有处理器

---

## 2. 方式一：拦截并转发已有流程

适用场景：

- 登录前加密码校验
- 进房前加门票校验
- 准备前加金币或体力检查
- 对默认流程加前置风控

核心思路：

1. 定义一个新的业务协议
2. 用你自己的模块或组件接住它
3. 先做业务校验
4. 校验通过后，构造框架原生协议
5. 手动把原生协议交回 Dispatcher

这样底层主链还是原来的，只是在入口前多了一层业务门禁。

---

## 3. 示例一：给登录流程加密码校验

### 3.1 定义新协议

```csharp
using System.IO;
using StellarNet.Lite.Shared.Core;
using StellarNet.Lite.Shared.Infrastructure;

[NetMsg(9001, NetScope.Global, NetDir.C2S, AllowUnauthenticated = true)]
public sealed class C2S_AdvancedLogin : ILiteNetSerializable
{
    public string AccountId;
    public string ClientVersion;
    public string Password;
    public string DeviceModel;

    public void Serialize(BinaryWriter writer)
    {
        writer.Write(AccountId ?? string.Empty);
        writer.Write(ClientVersion ?? string.Empty);
        writer.Write(Password ?? string.Empty);
        writer.Write(DeviceModel ?? string.Empty);
    }

    public void Deserialize(BinaryReader reader)
    {
        AccountId = reader.ReadString();
        ClientVersion = reader.ReadString();
        Password = reader.ReadString();
        DeviceModel = reader.ReadString();
    }
}
```

### 3.2 新建一个全局模块接住它

```csharp
using System.Buffers;
using StellarNet.Lite.Server.Core;
using StellarNet.Lite.Shared.Core;
using StellarNet.Lite.Shared.Infrastructure;
using StellarNet.Lite.Shared.Protocol;

[ServerModule("ServerAdvancedAuthModule", "高级登录拦截")]
public sealed class ServerAdvancedAuthModule
{
    private readonly ServerApp _app;
    private readonly INetSerializer _serializer = new LiteNetSerializer();

    public ServerAdvancedAuthModule(ServerApp app)
    {
        _app = app;
    }

    [NetHandler]
    public void OnC2S_AdvancedLogin(Session session, C2S_AdvancedLogin msg)
    {
        if (session == null || msg == null) return;

        if (msg.Password != "MySecretPassword")
        {
            _app.SendMessageToSession(session, new S2C_LoginResult
            {
                Success = false,
                Reason = "密码错误"
            });
            return;
        }

        var baseMsg = new C2S_Login
        {
            AccountId = msg.AccountId,
            ClientVersion = msg.ClientVersion
        };

        if (!NetMessageMapper.TryGetMeta(typeof(C2S_Login), out NetMessageMeta meta))
        {
            return;
        }

        byte[] buffer = ArrayPool<byte>.Shared.Rent(1024);
        try
        {
            int len = _serializer.Serialize(baseMsg, buffer);
            if (len <= 0) return;

            Packet packet = new Packet(0, meta.Id, NetScope.Global, string.Empty, buffer, 0, len);
            _app.GlobalDispatcher.Dispatch(session, packet);
        }
        finally
        {
            ArrayPool<byte>.Shared.Return(buffer);
        }
    }
}
```

这段代码真正做的事情只有两步：

1. 先做你自己的登录前校验
2. 再把请求转回框架已有的 `C2S_Login` 处理链

补充：

- 这类“登录前拦截协议”必须声明 `AllowUnauthenticated = true`
- 修改后要重新运行 `LiteProtocolScanner`，让未鉴权白名单重新生成

## 4. 示例二：给房间内准备流程加门槛

当前官方 `RoomFlow` 扩展里自带的准备流程是：

- 协议：`C2S_SetReady`
- 服务端处理：`ServerRoomSettingsComponent`

如果你想在准备前加业务门槛，例如：

- 金币不足不允许准备
- 装备评分不足不允许准备

不要直接改 `ServerRoomSettingsComponent`，更推荐这样做。

### 4.1 定义自己的请求和结果协议

```csharp
[NetMsg(9002, NetScope.Room, NetDir.C2S)]
public sealed class C2S_CustomSetReady
{
    public bool IsReady;
}

[NetMsg(9003, NetScope.Room, NetDir.S2C)]
public sealed class S2C_CustomReadyRejected
{
    public string Reason;
}
```

### 4.2 新建房间组件拦截它

```csharp
using System.Buffers;
using StellarNet.Lite.Server.Core;
using StellarNet.Lite.Shared.Core;
using StellarNet.Lite.Shared.Infrastructure;
using StellarNet.Lite.Shared.Protocol;

[RoomComponent(901, "CustomReadyCheck", "准备前校验组件")]
public sealed class ServerCustomReadyCheckComponent : ServerRoomComponent
{
    private readonly INetSerializer _serializer = new LiteNetSerializer();

    [NetHandler]
    public void OnC2S_CustomSetReady(Session session, C2S_CustomSetReady msg)
    {
        if (session == null || msg == null) return;

        if (msg.IsReady)
        {
            int playerGold = QueryPlayerGold(session.AccountId);
            if (playerGold < 100)
            {
                Room.SendMessageTo(session, new S2C_CustomReadyRejected
                {
                    Reason = "金币不足"
                });
                return;
            }
        }

        var baseMsg = new C2S_SetReady
        {
            IsReady = msg.IsReady
        };

        if (!NetMessageMapper.TryGetMeta(typeof(C2S_SetReady), out NetMessageMeta meta))
        {
            return;
        }

        byte[] buffer = ArrayPool<byte>.Shared.Rent(256);
        try
        {
            int len = _serializer.Serialize(baseMsg, buffer);
            if (len <= 0) return;

            Packet packet = new Packet(0, meta.Id, NetScope.Room, Room.RoomId, buffer, 0, len);
            Room.Dispatcher.Dispatch(session, packet);
        }
        finally
        {
            ArrayPool<byte>.Shared.Return(buffer);
        }
    }

    private int QueryPlayerGold(string accountId)
    {
        return 50;
    }
}
```

这套写法的重点不是“自己重写准备流程”，而是：

- 前面拦一下
- 合法后继续走框架原生 `C2S_SetReady`

## 5. 方式二：横向新增独立组件，不污染已有组件

适用场景：

- 新增战斗系统
- 新增房间聊天系统
- 新增任务系统
- 新增排行榜同步

这类需求最容易犯的错误是：

- 直接把业务字段塞进默认同步结构
- 直接把业务逻辑写进底层组件

例如不推荐这样做：

- 往 `ObjectSyncState` 里继续塞血量、武器、Buff

原因很简单：

- `ObjectSyncState` 是高频同步结构
- 低频业务数据混进去会浪费带宽
- 默认对象同步组件会越来越臃肿

## 6. 示例三：新增独立战斗组件

### 6.1 定义自己的业务协议

```csharp
[NetMsg(9101, NetScope.Room, NetDir.S2C)]
public sealed class S2C_EntityCombatSync
{
    public int NetId;
    public int CurrentHp;
    public int WeaponId;
}
```

### 6.2 服务端独立维护战斗状态

```csharp
using System.Collections.Generic;
using StellarNet.Lite.Server.Core;

[RoomComponent(902, "Combat", "战斗数据组件")]
public sealed class ServerCombatComponent : ServerRoomComponent
{
    private readonly Dictionary<int, int> _entityHp = new Dictionary<int, int>();

    public void TakeDamage(int netId, int damage)
    {
        if (!_entityHp.ContainsKey(netId))
        {
            _entityHp[netId] = 100;
        }

        _entityHp[netId] -= damage;

        Room.BroadcastMessage(new S2C_EntityCombatSync
        {
            NetId = netId,
            CurrentHp = _entityHp[netId],
            WeaponId = 1001
        });
    }
}
```

### 6.3 客户端独立接收，再抛给表现层

```csharp
using StellarNet.Lite.Client.Core;

[RoomComponent(902, "Combat", "战斗数据组件")]
public sealed class ClientCombatComponent : ClientRoomComponent
{
    [NetHandler]
    public void OnS2C_EntityCombatSync(S2C_EntityCombatSync msg)
    {
        if (msg == null || Room == null) return;
        Room.NetEventSystem.Broadcast(msg);
    }
}
```

这样做的好处是：

1. 不改默认 `ObjectSync`
2. 不改默认 `RoomSettings`
3. 新业务能自己独立维护
4. UI 或 View 只订阅自己的本地事件

## 7. 方式三：替换默认组件，但不拆框架主链

如果你不是“新增一层功能”，而是想直接替换默认能力，也不一定要改 Runtime。

例如：

- 不想用默认 `ObjectSync`
- 想换成自己的 2D 同步
- 想换成回合制状态同步

当前更推荐的思路是：

1. 建房时不再挂 `ComponentIdConst.ObjectSync`
2. 改成挂你自己的组件 Id
3. 让你自己的组件负责同步和快照

这样你替换的是“房间里的某个能力”，不是：

- 会话系统
- 房间系统
- 回放主链
- 重连主链

所以框架骨架仍然能复用。

## 8. 这套扩展方式如何接入自动装配

新增完协议、模块、组件后，不会自动生效。

因为当前框架运行时不做反射扫描，而是依赖 Editor 生成的静态装配代码。

你必须重新执行：

- `StellarNetLite/重新生成协议与组件常量表`
- `StellarNetLite/强制重新生成协议与组件常量表`

当前生成物包括：

- `MsgIdConst`
- `ComponentIdConst`
- `AutoMessageMetaRegistry`
- `AutoRegistry`
- 各特性分片 Binder

如果你新增代码后没有生效，优先排查的通常不是 Runtime 逻辑，而是生成步骤没跑。

## 9. 当前仓库里的真实扩展示例

当前 Demo 本身就展示了“横向扩展”的做法。

关键文件：

- `Samples/SocialDemo/Server/Components/ServerSocialRoomComponent.cs`
- `Samples/SocialDemo/Client/Components/ClientSocialRoomComponent.cs`
- `Samples/SocialDemo/Shared/Protocol/SocialRoomProtocols.cs`

它们并没有改写底层的：

- `ServerRoomSettingsComponent`
- `ServerObjectSyncComponent`

而是作为独立房间组件接入。

当前默认房间模板配置在：

- `Samples/SocialDemo/Shared/Binders/RoomTypeTemplateRegistry.cs`

当前默认模板是：

- `SocialRoom`
- `RoomSettings`
- `ObjectSync`

这就是当前仓库最典型的扩展方式：

- 保留框架主链
- 只加业务层

## 10. 实战时先问自己三个问题

每次要扩新功能前，先问：

### 10.1 这件事发生在大厅还是房间

- 大厅功能：全局模块
- 房间功能：房间组件

### 10.2 这是修改原流程，还是新增独立能力

- 原流程前加门禁：拦截并转发
- 新增独立能力：横向扩展

### 10.3 这是高频状态还是低频业务数据

- 高频：独立高频同步结构
- 低频：事件驱动单次广播

如果按这三个问题来拆，后续维护成本会低很多。
