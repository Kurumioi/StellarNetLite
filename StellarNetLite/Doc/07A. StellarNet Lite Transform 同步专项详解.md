# 07A. StellarNet Lite Transform 同步专项详解

本文专门讲 `ObjectSync` 里的 `Transform` 同步。

如果你是第一次做网络实体，不要跳着看。建议顺序是：

1. 先看 `07. StellarNet Lite 实体同步总览与 ObjectSync 入门`
2. 再看本文
3. 最后看 `07B. StellarNet Lite Animator 同步专项详解`

这篇文档会尽量回答新手最常见的几个问题：

1. 到底要挂哪些组件
2. 服务端要写什么
3. 客户端为什么不会一直抖、一直跳、一直穿插
4. 每个字段应该怎么调
5. 预测、平滑、Snap、追赶、回放分别是什么

---

## 1. Transform 同步到底在同步什么

在 `StellarNet Lite` 里，`Transform` 同步不是只同步一个 `position`。

它实际同步的是这一组空间状态：

1. 位置
   - `PosX`
   - `PosY`
   - `PosZ`
2. 旋转
   - `RotX`
   - `RotY`
   - `RotZ`
3. 速度
   - `VelX`
   - `VelY`
   - `VelZ`
4. 缩放
   - `ScaleX`
   - `ScaleY`
   - `ScaleZ`

其中最容易忽略的是“速度”。

框架不是只把“服务器当前位置”直接塞给客户端，而是把“当前状态 + 速度趋势”一起同步给客户端。客户端再基于这套数据做：

1. 远端实体预测
2. 平滑追赶
3. 停止时快速归位
4. 回放时按时间缩放继续播放

所以 `Transform` 同步的核心目标不是“收到包就立刻跳过去”，而是“在服务端权威前提下，让客户端尽量连续、平滑、可回放”。

## 2. 整体链路是怎么跑起来的

理解链路后，你会知道每个脚本分别负责什么。

### 2.1 服务端负责什么

服务端由 `ServerObjectSyncComponent` 管理权威实体。

业务层通常会拿到一个 `ServerSyncEntity`，然后持续写入它的状态：

```csharp
ServerSyncEntity entity = _syncService.SpawnObject(
    prefabHash,
    EntitySyncMask.Transform,
    spawnPos,
    spawnRotEuler,
    initialVelocity,
    ownerSessionId);

entity.Position = newPos;
entity.Rotation = new Vector3(0f, yaw, 0f);
entity.Velocity = velocity;
entity.Scale = Vector3.one;
```

服务端不会帮你“自动读取 Unity 场景里的某个 Transform 再同步”。

要点是：

1. 你自己决定这个实体的权威位置
2. 你自己决定这个实体当前速度
3. `ServerObjectSyncComponent` 负责比较脏数据、打包、广播

### 2.2 协议层负责什么

第一次出生时，走 `ObjectSpawnState`。

后续增量同步时，走 `ObjectSyncState`。

两者区别：

1. `ObjectSpawnState`
   一次发完整状态，保证客户端第一次生成实体时就能摆到正确位置
2. `ObjectSyncState`
   配合 `DirtyMask` 只发变化的字段，减少带宽

### 2.3 客户端负责什么

客户端收到包后，不会立刻每次都生硬覆盖 Unity `Transform`。

客户端分成两层：

1. `ClientObjectSyncComponent`
   缓存原始同步数据，并给视图层提供“预测后的数据”
2. `NetTransformView`
   每帧取预测结果，再决定是：
   - 平滑移动
   - 快速追赶
   - 直接 Snap
   - 停止归位

所以真正改 Unity 场景物体位置的是 `NetTransformView`。

## 3. 从 0 做一个可同步的 Transform 实体

这一节按“实操步骤”来写。

### 3.1 准备客户端 Prefab

准备一个普通 Unity Prefab，比如：

1. 玩家角色
2. NPC
3. 箱子
4. 可移动投射物

如果这个物体只需要位置同步，不需要动画，那么最小配置通常是：

1. 根节点有可见模型
2. 根节点挂 `NetIdentity`
3. 根节点挂 `NetTransformView`

如果之后还要动画同步，再额外挂 `NetAnimatorView`。

### 3.2 放到可被框架解析的位置

默认 `ObjectSyncClientRuntime.ResourcesNetPrefabResolver` 会根据 `PrefabHash` 去 `Resources` 找预制体。

所以通常要放到：

`Client/Resources/NetPrefabs/`

然后重新生成 `NetPrefabConsts.cs`，让 `PrefabHash -> 路径` 的映射可用。

如果这里没做对，客户端会出现：

1. 收到 `S2C_ObjectSpawn`
2. 但是找不到对应 Prefab
3. 实体无法实例化

### 3.3 挂载组件

根节点至少要有：

1. `NetIdentity`
2. `NetTransformView`

注意：

1. `NetIdentity` 负责记录 `NetId`
2. `NetTransformView` 负责把同步结果应用到 Unity 物体

如果你忘记手动挂，默认 `ObjectSyncClientRuntime.DefaultNetViewBinder` 也会尝试自动补上这两个组件。

但文档仍然建议你在 Prefab 上手动挂好，因为：

1. 便于在 Inspector 里直接调参数
2. 便于团队成员知道这个物体是网络实体
3. 避免运行时才发现组件补挂后参数还是默认值

### 3.4 服务端生成实体

服务端常见写法：

```csharp
ServerSyncEntity entity = _syncService.SpawnObject(
    PlayerPrefabHash,
    EntitySyncMask.Transform,
    spawnPosition,
    Vector3.zero,
    Vector3.zero,
    session.SessionId);
```

如果是“玩家角色 + 后续还会同步动画”，就直接用：

```csharp
EntitySyncMask.All
```

### 3.5 服务端持续更新状态

只出生一次还不够，你还要在 Tick 或业务消息里持续写权威状态：

```csharp
playerSync.Position = newPos;
playerSync.Rotation = new Vector3(0f, yaw, 0f);
playerSync.Velocity = currentVelocity;
```

这里最关键的是：

`Velocity` 不要忘。

因为客户端预测、追赶、停止判断都依赖速度。如果位置在变，但速度长期不写，客户端视图层就会缺少趋势信息。

## 4. `EntitySyncMask.Transform` 是什么作用

定义在：

`ObjectSyncProtocols.cs`

它的作用是告诉框架：

“这个实体需要同步空间状态。”

常见值：

1. `EntitySyncMask.Transform`
   只同步空间
2. `EntitySyncMask.All`
   同步空间和动画

如果你没带上 `Transform` 位：

1. 服务端就算改了 `Position`
2. 客户端也不会把它当成 Transform 同步实体处理

所以一个“需要移动”的物体，最基本一定要包含 `EntitySyncMask.Transform`。

## 5. `ObjectSpawnState` 里和 Transform 有关的字段

定义文件：

`Extensions/ObjectSync/Shared/Protocol/Components/ObjectSync/ObjectSpawnState.cs`

它表示“实体第一次出生时的完整快照”。

### 5.1 字段说明

#### `NetId`

网络实体唯一编号。

作用：

1. 客户端后续靠它把增量包分发给正确的物体
2. `NetIdentity` 会记录它

#### `PrefabHash`

预制体哈希。

作用：

1. 客户端根据它找到对应的 Prefab
2. 决定实例化哪个实体外观

#### `Mask`

同步能力位掩码。

作用：

1. 指示这个实体是否开启 `Transform`
2. 指示是否还开启 `Animator`

#### `PosX/PosY/PosZ`

出生位置。

作用：

1. 客户端实例化时把实体摆到正确地点
2. `NetTransformView.HardSetInitialState(...)` 初始落位时使用

#### `RotX/RotY/RotZ`

出生欧拉旋转。

作用：

1. 客户端实例化时设置初始朝向
2. 作为后续平滑旋转的起点

#### `DirX/DirY/DirZ`

当前代码里它实际承载的是“出生时速度方向/速度值”。

你可以把它理解成：

“第一次出生时给客户端的初始速度信息。”

作用：

1. 帮助客户端建立初始运动趋势
2. 在快照恢复时带回运动方向

虽然字段名叫 `Dir`，但当前运行语义更接近 `Velocity`。

#### `ScaleX/ScaleY/ScaleZ`

出生缩放。

作用：

1. 让客户端生成出来就有正确体积
2. 回放恢复时保持一致

### 5.2 关键函数

#### `Serialize(BinaryWriter writer)`

作用：

1. 把完整出生状态写入二进制流
2. 发给客户端或导出快照时使用

#### `Deserialize(BinaryReader reader)`

作用：

1. 客户端把收到的二进制流还原成 `ObjectSpawnState`
2. 回放恢复时也会使用

#### `CreateDefault()`

作用：

1. 给结构体提供安全默认值
2. 默认把缩放设为 `1`
3. 默认把 `AnimParams` 设为空数组，避免空引用

## 6. `ObjectSyncState` 里和 Transform 有关的字段

定义文件：

`Extensions/ObjectSync/Shared/Protocol/Components/ObjectSync/ObjectSyncProtocols.cs`

它表示“后续每一帧或每几帧的增量同步状态”。

### 6.1 Transform 相关字段

#### `NetId`

告诉客户端这条同步数据属于哪一个实体。

#### `Mask`

说明这个实体本来具备哪些同步能力。

#### `DirtyMask`

最关键的字段之一。

它说明这次包里到底哪些子字段发生了变化。

#### `PosX/PosY/PosZ`

只有当 `DirtyMask` 包含 `Position` 时才会参与序列化。

#### `RotX/RotY/RotZ`

只有当 `DirtyMask` 包含 `Rotation` 时才会参与序列化。

#### `VelX/VelY/VelZ`

只有当 `DirtyMask` 包含 `Velocity` 时才会参与序列化。

#### `ScaleX/ScaleY/ScaleZ`

只有当 `DirtyMask` 包含 `Scale` 时才会参与序列化。

### 6.2 `ObjectSyncDirtyMask` 里和 Transform 相关的位

定义同样在 `ObjectSyncProtocols.cs`。

#### `Position`

表示位置变化了。

#### `Rotation`

表示旋转变化了。

#### `Velocity`

表示速度变化了。

#### `Scale`

表示缩放变化了。

#### `AllTransform`

这是组合位：

```csharp
Position | Rotation | Velocity | Scale
```

作用：

1. 首次全量同步时快速标记全部空间字段
2. 周期性全量重同步时直接使用

### 6.3 为什么要做 DirtyMask

因为不是每次都要发所有字段。

例如：

1. 角色位置变了，但缩放没变
2. 投射物只在飞，不会改缩放
3. 某些静止物体只是偶尔旋转

如果每个 Tick 都发全量：

1. 带宽浪费
2. 房间大了以后放大得很快
3. 回放记录体积也会增长

所以 `DirtyMask` 是高负载下非常重要的优化点。

## 7. `ServerObjectSyncComponent` 如何判断 Transform 该不该发

核心文件：

`Extensions/ObjectSync/Server/Components/ServerObjectSyncComponent.cs`

### 7.1 这个组件做什么

它是服务端权威同步管理器，负责：

1. 创建实体
2. 销毁实体
3. 周期性比较当前状态和上次已发送状态
4. 生成 `ObjectSyncState`
5. 广播给在线客户端
6. 导出给回放系统

### 7.2 `ServerSyncEntity` 里和 Transform 相关的字段

#### `Position`

当前权威位置。

#### `Rotation`

当前权威欧拉旋转。

#### `Velocity`

当前权威速度。

#### `Scale`

当前权威缩放。

### 7.3 `SpawnObject(...)`

作用：

1. 分配 `NetId`
2. 记录实体初始状态
3. 生成出生消息 `S2C_ObjectSpawn`
4. 广播给房间成员

### 7.4 `BuildSpawnState(ServerSyncEntity entity)`

作用：

1. 把服务端实体转成 `ObjectSpawnState`
2. 第一次出生和快照导出都依赖它

你会看到：

1. `Position` 被写进 `PosX/Y/Z`
2. `Rotation` 被写进 `RotX/Y/Z`
3. `Velocity` 被写进 `DirX/Y/Z`
4. `Scale` 被写进 `ScaleX/Y/Z`

### 7.5 `HasMeaningfulChange(ServerSyncEntity entity)`

作用：

1. 判断和“上一次发送快照”相比有没有足够明显的变化
2. 决定是否值得继续构建增量包

Transform 侧会检查：

1. 位置平方差是否超过 `PositionDirtyThresholdSqr`
2. 速度平方差是否超过 `VelocityDirtyThresholdSqr`
3. 缩放平方差是否超过 `ScaleDirtyThresholdSqr`
4. 欧拉角各轴差值是否超过 `RotationDirtyThreshold`

这一步是第一层优化。

如果完全没明显变化，就直接不发。

### 7.6 `BuildDirtyMask(ServerSyncEntity entity, bool forceFull)`

作用：

1. 逐项计算哪些字段真的脏了
2. 给 `ObjectSyncState` 写入正确位标记

`forceFull == true` 时，会直接走 `BuildFullDirtyMask(entity.Mask)`。

这通常用于：

1. 周期性全量重同步
2. 防止长时间只靠增量导致状态漂移

### 7.7 `BuildFullDirtyMask(byte entityMask)`

作用：

1. 如果实体有 `Transform` 能力，就打上 `AllTransform`
2. 如果实体有 `Animator` 能力，再打上 `AllAnimator`

### 7.8 为什么服务端要保存 `_lastSentSnapshots`

因为没有上一次基线，就不知道这次变没变。

它的作用是：

1. 给脏位计算提供比较基准
2. 防止每 Tick 无脑发全量
3. 支撑中大型房间里的同步成本控制

## 8. `ClientObjectSyncComponent` 如何把 Transform 变成“可预测数据”

核心文件：

`Extensions/ObjectSync/Client/Components/ClientObjectSyncComponent.cs`

### 8.1 这个组件做什么

它不是直接操作 Unity 物体的脚本。

它负责：

1. 接收 `S2C_ObjectSpawn`
2. 接收 `S2C_ObjectSync`
3. 更新客户端缓存
4. 向视图层提供预测后的 Transform 数据
5. 向回放系统提供快照恢复能力

### 8.2 `SyncEntityData` 是什么

这是客户端对单个实体的缓存结构。

Transform 相关字段有：

1. `RawPos`
2. `RawRot`
3. `RawVel`
4. `RawScale`
5. `LocalReceiveTime`
6. `ServerTime`

可以把它理解成：

“从服务端收到的最近一份原始状态。”

### 8.3 `OnS2C_ObjectSpawn(S2C_ObjectSpawn msg)`

作用：

1. 把出生状态写进缓存
2. 广播给视图层，让 `ObjectSpawnerView` 去生成实体

### 8.4 `OnS2C_ObjectSync(S2C_ObjectSync msg)`

作用：

1. 批量消费增量同步包
2. 调用 `ApplySyncState(...)` 把脏字段写回缓存
3. 记录最新服务器时间和本地接收时间

### 8.5 `TryGetTransformData(int netId, out PredictedTransformData result)`

这是 `NetTransformView` 最依赖的方法。

它负责输出：

1. `Position`
2. `Rotation`
3. `Velocity`
4. `Scale`
5. `TimeSinceLastSync`
6. `PlaybackSpeed`

你可以把它理解成：

“把原始同步数据，整理成当前这一帧应该如何播放的预测结果。”

### 8.6 `PredictedTransformData` 每个字段的作用

#### `Position`

预测后目标位置。

#### `Rotation`

预测后目标旋转。

#### `Velocity`

当前运动趋势。

#### `Scale`

当前目标缩放。

#### `TimeSinceLastSync`

距离最近一次收到同步包，已经过去了多久。

作用：

1. 计算预测窗口
2. 控制回放和在线房间中的推进程度

#### `PlaybackSpeed`

这不是“移动速度”，而是“当前这一帧客户端应按多快的时间倍率追赶/播放”。

作用：

1. 在线房间下做轻微追赶
2. 回放时跟随时间缩放
3. 让视图层知道该平滑得快一点还是慢一点

`Extensions/ObjectSync/Client/Views/NetTransformView.cs`

这是你在 Inspector 上最常调的脚本。

## 10. `NetTransformView` 每个字段是什么意思

### `PosSmoothTime`

位置平滑时间。

默认值：

`0.1`

作用：

1. 控制远端实体正常追踪目标点时的平滑程度
2. 数值越小，追得越快
3. 数值越大，看起来越柔和，但也更容易拖尾

适合调大：

1. 慢速角色
2. 对平滑观感要求高的 NPC

适合调小：

1. 快速移动物体
2. 对跟手性要求更高的对象

### `RotSmoothSpeed`

旋转平滑速度。

默认值：

`15`

作用：

1. 控制 `Quaternion.Slerp` 的追赶速度
2. 数值越大，朝向越快对齐服务器

### `SnapThreshold`

强制 Snap 阈值。

默认值：

`3.0`

作用：

1. 如果客户端当前位置离目标位置太远
2. 不再慢慢追，直接瞬移到目标点

它是“防止越追越离谱”的安全闸。

### `CatchUpThreshold`

加速追赶阈值。

默认值：

`1.5`

作用：

1. 目标点比平常偏远，但又没远到必须 Snap
2. 这时把 `effectiveSmoothTime` 降低一半
3. 让客户端追得更快

### `StopThreshold`

停止归位阈值。

默认值：

`0.05`

作用：

1. 如果服务器速度几乎为 0
2. 同时客户端已经离目标非常近
3. 就直接贴回目标点并清空平滑速度

这样可以减少物体停下来时的轻微漂移和抖动。

### `LocalSnapThreshold`

本地软校正时的强制纠正阈值。

默认值：

`2.0`

作用：

1. 当物体属于本地玩家
2. 客户端本地控制和服务器结果差距太大
3. 就不要再柔性纠正，直接拉回服务器位置

### `LocalSoftCorrectionSpeed`

本地软校正速度。

默认值：

`5`

作用：

1. 本地玩家和服务器有轻微偏差时
2. 用 `Lerp` 慢慢纠回
3. 避免完全硬拉导致手感突兀

### `LocalIgnoreThreshold`

本地忽略阈值。

默认值：

`0.05`

作用：

1. 如果本地预测位置和服务器位置差得极小
2. 就直接忽略，不做校正

这个值过小会导致频繁微抖，过大会导致小误差堆积更久。

### `IsLocalPlayer`

本地拥有者标记。

它不是手动长期配置项，而是运行时状态。

默认 `DefaultNetViewBinder` 会根据：

`state.OwnerSessionId == NetClient.Session.SessionId`

自动设置。

作用：

1. 本地玩家和远端玩家走不同处理分支
2. 避免在线房间里把本地控制对象每帧都被远端同步强行覆盖

## 11. `NetTransformView` 每个关键函数在做什么

### `Awake()`

作用：

1. 缓存 `NetIdentity`
2. 缓存 `CharacterController`

为什么要缓存 `CharacterController`：

因为直接改带 `CharacterController` 的物体位置时，某些情况下会与控制器碰撞更新产生冲突。

### `HardSetInitialState(Vector3 pos, Quaternion rot, Vector3 scale)`

作用：

1. 实体初次生成时立刻设置到正确位置
2. 设置旋转和缩放
3. 清空平滑速度 `_currentVelocity`

使用时机：

1. 客户端首次生成实体
2. 回放恢复初始状态

### `Update()`

这是每帧驱动入口。

流程是：

1. 检查 `NetIdentity` 和 `SyncService`
2. 从 `ClientObjectSyncComponent` 取 `PredictedTransformData`
3. 如果是在线房间下的本地玩家，直接返回，不覆盖本地控制
4. 否则走 `ProcessRemoteTransformSync(...)`

注意：

当前代码里 `ProcessLocalReconciliation(...)` 已经具备能力，但 `Update()` 里在线房间本地玩家目前是直接跳过远端覆盖。

这意味着当前设计更偏向：

1. 在线房间本地玩家自己驱动表现
2. 服务端同步更多用于远端观察者和兜底校正数据源

### `ProcessLocalReconciliation(ref PredictedTransformData syncData)`

作用：

1. 处理本地玩家的服务器纠偏
2. 小误差忽略
3. 中误差软校正
4. 大误差直接拉回

虽然当前 `Update()` 没有在在线房间里主动调用它，但这套逻辑本身已经具备，后续如果你要把本地纠错做得更强，可以基于它继续演进。

### `ProcessRemoteTransformSync(ref PredictedTransformData syncData)`

这是远端实体表现的核心。

内部主要分几种情况：

1. `PlaybackSpeed > 5f` 或距离超出 `SnapThreshold`
   - 直接 Snap
2. 速度几乎为 0 且距离很近
   - 直接归位并清零速度
3. 正常追赶
   - 用 `SmoothDamp`
4. 超过 `CatchUpThreshold`
   - 缩短平滑时间，让追赶更积极

旋转和缩放也会根据 `PlaybackSpeed` 决定是：

1. 直接设置
2. 还是平滑插值

### `ApplyPosition(Vector3 pos)`

作用：

1. 统一封装位置写入
2. 对带 `CharacterController` 的对象，先暂时禁用控制器，再改位置，再启用

这是兼容性细节，非常重要。

如果不这么做，某些角色控制器对象会出现：

1. 碰撞体推挤异常
2. 改位置失败
3. 地形贴地表现异常

## 12. 平滑、追赶、Snap 的原理是什么

这部分很多人只会调参数，但不知道为什么要这么设计。

### 12.1 为什么不能每次收到包都直接改位置

因为网络包不是每帧等间隔到达的。

如果你直接硬改：

1. 角色会一卡一卡跳
2. 高延迟下更明显
3. 录像回放也会很生硬

### 12.2 为什么要速度

因为只有位置，没有趋势，客户端只能傻等下一次包。

有了速度后，客户端可以在两次包之间继续推演：

1. 角色大概率还在这个方向移动
2. 下一帧先往前走一点
3. 收到新包后再修正

这就是远端预测的基础。

### 12.3 为什么还要 CatchUp

因为网络抖动时，客户端有时会明显落后于服务器。

如果仍然用平常的慢速平滑：

1. 看起来就像人物被橡皮筋拖着
2. 永远追不上真实位置

所以距离一旦超过 `CatchUpThreshold`，就临时加快追赶速度。

### 12.4 为什么还要 Snap

如果误差已经非常大，再继续插值反而会更假。

常见场景：

1. 刚进入房间
2. 丢包后恢复
3. 回放跳时间点
4. 实体瞬移

这时候直接 Snap 反而是正确行为。

### 12.5 为什么停止时要快速归位

一个移动中的平滑算法在接近目标点时，可能会残留一点点微漂移。

如果服务器已经明确表示：

1. 速度几乎为 0
2. 且客户端已经非常接近目标

那最合理的做法就是：

1. 直接对齐
2. 清掉 `_currentVelocity`

这样人物停下来会更稳。

## 13. 回放为什么也能用这套 Transform 同步

因为 `ObjectSync` 从设计上就区分了：

1. 原始同步状态
2. 视图层表现

服务端可以导出：

1. `ExportSpawnStates()`
2. `ExportSnapshot()`

客户端可以消费：

1. `ApplySnapshot(byte[] payload)`
2. `ApplyReplaySnapshot(ObjectSpawnState[] states)`

而 `NetTransformView` 并不知道“这是在线实时包还是回放包”，它只知道：

1. 现在拿到一个预测后的目标
2. 再根据 `PlaybackSpeed` 去决定怎么播放

这就是为什么这套设计天然回放友好。

## 14. 新手最常用的服务端移动模板

### 14.1 玩家输入驱动型

适合：

1. 社交房间
2. 轻动作房间
3. 玩家自己上报移动，服务端做轻校验

示例思路：

```csharp
Vector3 newPos = new Vector3(msg.PosX, msg.PosY, msg.PosZ);
Vector3 newVel = new Vector3(msg.VelX, msg.VelY, msg.VelZ);

playerSync.Position = newPos;
playerSync.Velocity = newVel;
playerSync.Rotation = new Vector3(0f, msg.RotY, 0f);
```

关键点：

1. 位置要写
2. 速度要写
3. 朝向最好也同步更新

### 14.2 服务端 Tick 驱动型

适合：

1. AI
2. 巡逻 NPC
3. 纯服务端权威物体

示例思路：

```csharp
entity.Velocity = moveDir * speed;
entity.Position += entity.Velocity * deltaTime;
entity.Rotation = new Vector3(0f, targetYaw, 0f);
```

关键点：

1. 服务端自己推进位置
2. 不依赖客户端上报结果
3. 客户端只负责表现

## 15. 参数怎么调比较稳

没有一套参数能通吃所有物体，但有比较稳的起点。

### 15.1 角色 / NPC 的推荐起点

1. `PosSmoothTime = 0.08 ~ 0.12`
2. `RotSmoothSpeed = 12 ~ 18`
3. `CatchUpThreshold = 1.0 ~ 1.8`
4. `SnapThreshold = 2.5 ~ 4.0`
5. `StopThreshold = 0.03 ~ 0.08`

### 15.2 快速物体的推荐起点

1. `PosSmoothTime` 更小
2. `SnapThreshold` 更大一些
3. `CatchUpThreshold` 更小一些，让它更快追

### 15.3 本地玩家如果未来启用软校正

1. `LocalIgnoreThreshold` 不要太小
2. `LocalSoftCorrectionSpeed` 不要太激进
3. `LocalSnapThreshold` 要明显大于忽略阈值

## 16. 常见错误与排查

### 16.1 客户端不生成实体

优先检查：

1. `PrefabHash` 是否正确
2. Prefab 是否放进可解析目录
3. `NetPrefabConsts.cs` 是否已重新生成
4. 客户端是否收到 `S2C_ObjectSpawn`

### 16.2 物体生成了但不动

优先检查：

1. 服务端是否持续写 `Position`
2. 服务端是否持续写 `Velocity`
3. `Mask` 是否包含 `EntitySyncMask.Transform`
4. 客户端实例上是否真的有 `NetTransformView`

### 16.3 物体一直瞬移

优先检查：

1. `SnapThreshold` 是否设太小
2. 服务端是否偶尔给出超大位置差
3. `PlaybackSpeed` 是否异常过高
4. 是否经常触发全量重同步

### 16.4 物体像在拖尾

优先检查：

1. `PosSmoothTime` 是否过大
2. `CatchUpThreshold` 是否过大
3. 服务端同步频率是否太低
4. 速度写入是否不准确

### 16.5 物体停下来后还在轻微抖

优先检查：

1. `StopThreshold` 是否过小
2. 服务端静止时是否真的把 `Velocity` 归零
3. 业务层是否来回写入很小的误差位置

### 16.6 带 CharacterController 的对象改位置异常

优先检查：

1. 是否走了 `NetTransformView.ApplyPosition(...)`
2. 是否自己又在别的脚本里强改了位置
3. 是否本地控制脚本和网络脚本同时争夺位置

---

## 17. 你现在应该记住的几件事

1. `Transform` 同步的核心不是“收到包就硬改位置”，而是“服务端权威 + 客户端预测平滑”。
2. 服务端一定要维护 `Position`、`Rotation`、`Velocity`，否则客户端只能拿到残缺趋势。
3. `NetTransformView` 是真正改 Unity 场景对象的脚本，参数调优基本都在这里。
4. `DirtyMask`、阈值判断、周期性全量同步，是这套系统在高负载下还能稳定运行的重要基础。
5. 如果你后续还要做动画同步，下一篇直接看 `07B`，两者是配套关系。
