# 01. StellarNet Lite 核心概念与全流程指南

> 如果你第一次接触这个框架，先读本文。  
> 本文只做一件事：把当前仓库的主链讲清楚。

---

## 1. 先记住四个目录

当前仓库最重要的四个目录是：

- `Runtime`
- `GameDemo`
- `Editor`
- `Doc`

它们的职责分别是：

### 1.1 `Runtime`

框架核心代码。

这里放的是：

- 会话管理
- 房间管理
- 模块与组件基类
- 协议路由
- 回放
- 重连
- 默认对象同步
- 传输层实现

### 1.2 `GameDemo`

示例业务代码。

这里放的是：

- Demo 房间业务组件
- UI 面板
- 路由器
- Demo 协议
- 示例预制体和场景

### 1.3 `Editor`

编辑器工具链。

这里放的是：

- 协议和组件扫描生成器
- 网络预制体扫描器
- 配置编辑器
- 脚手架工具
- 运行时监控窗口

### 1.4 `Doc`

教学文档。

## 2. 当前框架的最小核心心智

StellarNet Lite 当前是**服务端权威 + 客户端表现层渲染**的结构。

也就是说：

1. 客户端发送请求
2. 服务端计算真实结果
3. 服务端广播结果
4. 客户端只负责接收、缓存、表现

所以不要把客户端理解成“逻辑真相源”。

客户端在这个框架里的职责更准确地说是：

- 输入入口
- 状态缓存
- 事件分发
- 表现层驱动

---

## 3. Global 和 Room 是什么

框架把协议分成两个作用域：

### 3.1 `Global`

不依赖具体房间的流程。

例如：

- 登录
- 拉房间列表
- 大厅聊天
- 下载录像

这类协议通常由**全局模块**处理。

### 3.2 `Room`

必须在房间内发生的流程。

例如：

- 进入房间
- 准备
- 开始游戏
- 对象同步
- 房间内业务协议

这类协议通常由**房间组件**处理。

---

## 4. Session、Room、Module、Component 分别是什么

### 4.1 `Session`

服务端对一个玩家连接的抽象。

当前 `Session` 里最重要的状态包括：

- `SessionId`
- `AccountId`
- `ConnectionId`
- `CurrentRoomId`
- `AuthorizedRoomId`
- `IsOnline`

### 4.2 `Room`

服务端房间实例。

它负责：

- 持有房间成员
- 持有房间组件
- 持有房间消息分发器
- 推进房间 Tick

### 4.3 `Module`

全局域业务处理器。

例如当前内置模块：

- `ServerUserModule / ClientUserModule`
- `ServerRoomModule / ClientRoomModule`
- `ServerLobbyModule / ClientLobbyModule`
- `ServerPingModule / ClientPingModule`
- `ServerReplayModule / ClientReplayModule`

### 4.4 `RoomComponent`

房间域业务处理器。

当前内置房间组件主要有：

- `RoomSettings`
- `ObjectSync`

GameDemo 额外提供：

- `SocialRoom`

## 5. 启动时发生了什么

当前启动入口是：

- `Runtime/StellarNetAppManager.cs`

它做了几件关键事：

1. 找到挂在当前对象上的 `INetworkTransport`
2. 初始化 `LiteNetSerializer`
3. 读取 `NetConfig`
4. 初始化 `NetMessageMapper`
5. 把自动生成的注册器绑定到房间工厂和客户端工厂
6. 监听传输层事件

这意味着当前框架不是运行时反射扫描注册，而是：

- Editor 先生成静态绑定代码
- Runtime 启动时直接使用生成结果

## 6. 一条消息从客户端到服务端怎么走

以登录为例，当前链路是：

### 6.1 UI 发起请求

UI 最终会调用：

```csharp
NetClient.Send(new C2S_Login
{
    AccountId = "Player01",
    ClientVersion = Application.version
});
```

### 6.2 `NetClient` 转交给 `ClientApp`

`NetClient` 本身只是门面。

真正的发送逻辑在：

- `ClientApp.SendMessage(...)`

### 6.3 `ClientApp` 查协议元数据并发包

它会：

1. 用 `NetMessageMapper` 查消息元数据
2. 校验方向是不是 `C2S`
3. 组装 `Packet`
4. 调用 `INetworkTransport.SendToServer`

### 6.4 服务端收到包后交给 `ServerApp`

`ServerApp.OnReceivePacket(...)` 会做：

1. 根据连接查 `Session`
2. 处理匿名会话
3. 处理序号防重放
4. 根据 `Scope` 决定走全局分发还是房间分发

### 6.5 分发到具体模块或组件

例如：

- `GlobalDispatcher.Dispatch(...)`
- `Room.Dispatcher.Dispatch(...)`

最终命中带 `[NetHandler]` 的处理函数。

## 7. 登录、建房、进房三条主链怎么理解

### 7.1 登录

登录主处理在：

- `Runtime/Server/Modules/ServerUserModule.cs`

它负责：

- 校验版本
- 处理重复登录
- 顶掉旧连接
- 生成或复用会话
- 返回 `S2C_LoginResult`

### 7.2 建房 / 加房

主处理在：

- `Runtime/Server/Modules/ServerRoomModule.cs`

它负责：

- 创建房间
- 进入已有房间
- JoinOrCreate 降级建房
- 授权进入房间

注意当前进入房间是两段式：

1. 服务端先授权
2. 客户端初始化房间组件后再回 `C2S_RoomSetupReady`

### 7.3 正式进房

当服务端收到 `C2S_RoomSetupReady` 后：

1. 校验授权房间
2. 把会话正式加入房间
3. 清理授权状态

这一段也是当前房间主链里非常关键的边界保护。

## 8. 客户端房间是怎么构建出来的

当前客户端房间对象是：

- `Runtime/Client/Core/ClientRoom.cs`

它内部持有：

- 房间组件实例
- 房间消息分发器
- 房间事件系统

当客户端进入在线房间或回放房间时，工厂会根据组件 Id 列表构建对应组件。

---

## 9. 当前 Demo 默认房间长什么样

当前 Demo 的默认房间模板在：

- `GameDemo/Shared/Binders/RoomTypeTemplateRegistry.cs`

默认模板只有一个：

- `交友房间`

默认挂载组件是：

- `SocialRoom`
- `RoomSettings`
- `ObjectSync`

这意味着当前 Demo 更接近：

- 社交房间示例

而不是战斗房间示例。

## 10. 生成代码为什么找不到

当前仓库没有提交：

- `Assets/StellarNetLiteGenerated`

所以像下面这些类型：

- `MsgIdConst`
- `ComponentIdConst`
- `AutoMessageMetaRegistry`
- `AutoRegistry`
- `NetPrefabConsts`

并不是不存在，而是需要在本地 Unity Editor 里重新生成。

对应工具链说明见：

- `Doc/08. StellarNet Lite 代码生成与编辑器工具链指南.md`

## 11. 初学者最容易踩的坑

### 11.1 忘了重新生成注册代码

表现通常是：

- 新协议没被路由
- 新组件没被装配
- 新模块不工作

### 11.2 把业务逻辑直接写进 View 或 UI

结果会导致：

- 表现层和网络层耦死
- 回放和在线逻辑混在一起
- 后续状态切换难以维护

### 11.3 把所有字段都塞进默认对象同步

结果会导致：

- 高频包变胖
- 业务边界混乱
- 后续快照恢复也越来越难维护
