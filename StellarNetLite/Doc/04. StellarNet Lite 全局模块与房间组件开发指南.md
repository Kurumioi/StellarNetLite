# 04. StellarNet Lite 全局模块与房间组件开发指南

本文档专门说明一件事：

什么时候写全局模块，什么时候写房间组件，以及两者应该怎样组织代码。

---

## 1. 两类业务入口的边界

当前框架故意把业务入口拆成两类：

- `GlobalModule`
- `RoomComponent`

这是为了把“大厅逻辑”和“房间逻辑”分开。

---

## 2. 全局模块的职责

全局模块适合处理下面这些事情：

- 不依赖具体房间。
- 登录、大厅、房间列表、公告、下载、匹配之类的全局业务。
- 需要直接操作 `ServerApp` 或全局会话索引。

### 2.1 服务端全局模块常见职责

- 按账号查会话。
- 广播大厅消息。
- 分配房间授权。
- 调用房间创建或房间查询接口。

### 2.2 客户端全局模块常见职责

- 接收大厅同步消息。
- 转发全局协议到全局事件总线。
- 更新客户端登录态或大厅态。

---

## 3. 房间组件的职责

房间组件适合处理下面这些事情：

- 必须附着在某个房间实例上。
- 依赖房间成员列表。
- 依赖房间生命周期。
- 依赖房间 Tick。

### 3.1 服务端房间组件常见职责

- 成员加入和离开逻辑。
- 房间内玩法同步。
- 房间内状态机推进。
- 定时广播、定时结算、快照导出。

### 3.2 客户端房间组件常见职责

- 缓存房间域状态。
- 把协议转发给房间事件系统。
- 为表现层提供可查询的数据模型。

---

## 4. 判断规则

遇到新业务时，可以按下面规则判断：

### 4.1 如果答案偏向“大厅”

优先写全局模块：

- 这条消息是否不需要房间上下文。
- 这条消息是否可能在没有房间时使用。
- 这条逻辑是否应该跨房间统一生效。

### 4.2 如果答案偏向“房间”

优先写房间组件：

- 这条消息是否只在房间中有意义。
- 这条逻辑是否需要成员列表或房间状态。
- 这条逻辑是否应该随着房间创建和销毁。

---

## 5. 推荐的代码组织方式

### 5.1 协议放哪

协议统一放 `Shared/Protocol`。

原因：

- 客户端和服务端都要引用。
- 生成器要统一扫描。
- 回放和调试时也会依赖协议元信息。

### 5.2 服务端逻辑放哪

- 全局逻辑放 `Server/Modules`
- 房间逻辑放 `Server/Components`

### 5.3 客户端逻辑放哪

- 全局逻辑放 `Client/Modules`
- 房间逻辑放 `Client/Components`

### 5.4 表现层放哪

不要放进 `Client/Modules` 或 `Client/Components`。

表现层建议单独放在：

- `Client/View`
- `Client/Infrastructure`

这样网络数据模型和 UI 才能解耦。

---

## 6. 全局模块的典型写法

服务端全局模块通常长这样：

```csharp
[ServerModule("ServerExampleModule", "示例全局模块")]
public sealed class ServerExampleModule
{
    private readonly ServerApp _app;

    public ServerExampleModule(ServerApp app)
    {
        _app = app;
    }

    [NetHandler]
    public void OnC2S_Example(Session session, C2S_Example msg)
    {
        _app.SendMessageToSession(session, new S2C_ExampleResult());
    }
}
```

客户端全局模块通常长这样：

```csharp
[ClientModule("ClientExampleModule", "示例全局模块")]
public sealed class ClientExampleModule
{
    [NetHandler]
    public void OnS2C_ExampleResult(S2C_ExampleResult msg)
    {
        GlobalTypeNetEvent.Broadcast(msg);
    }
}
```

---

## 7. 房间组件的典型写法

服务端房间组件通常长这样：

```csharp
[RoomComponent(500, "ExampleRoom", "示例房间组件")]
public sealed class ServerExampleRoomComponent : ServerRoomComponent
{
    [NetHandler]
    public void OnC2S_Example(Session session, C2S_Example msg)
    {
        Room.BroadcastMessage(new S2C_ExampleSync());
    }
}
```

客户端房间组件通常长这样：

```csharp
[RoomComponent(500, "ExampleRoom", "示例房间组件")]
public sealed class ClientExampleRoomComponent : ClientRoomComponent
{
    [NetHandler]
    public void OnS2C_ExampleSync(S2C_ExampleSync msg)
    {
        Room.NetEventSystem.Broadcast(msg);
    }
}
```

---

## 8. 房间组件常见生命周期

服务端房间组件常用生命周期如下：

- `OnInit`
- `OnMemberJoined`
- `OnMemberLeft`
- `OnMemberOffline`
- `OnMemberOnline`
- `OnGameStart`
- `OnGameEnd`
- `OnSendSnapshot`
- `OnDestroy`

如果组件还需要每 Tick 推进，再实现：

- `ITickableComponent`

客户端房间组件常用生命周期如下：

- `OnInit`
- `OnDestroy`

客户端房间内大多数状态变化通常来自收包，而不是主动 Tick。

---

## 9. 推荐的数据流

建议按下面的方式组织客户端房间数据：

1. `ClientRoomComponent` 接收协议。
2. 组件更新自己的缓存状态。
3. 组件通过 `Room.NetEventSystem.Broadcast(...)` 广播房间事件。
4. UI Router 或 View 订阅房间事件并刷新表现。

不要让 UI 直接持有底层网络发包和状态缓存逻辑。

---

## 10. 什么时候不该新增模块

下面几种情况通常不需要新增全局模块：

- 只是某个房间内的小功能。
- 只是某个已有全局模块中的一个简单分支。
- 只是 UI 层的显示需求。

下面几种情况通常不需要新增房间组件：

- 纯大厅逻辑。
- 与房间生命周期无关的全局状态。
- 只在客户端表现层存在的临时视觉数据。

---

## 11. 开发建议

- 一个组件只负责一个明确领域。
- 一个模块只负责一个明确全局功能域。
- 业务边界优先清晰，其次才是“少建文件”。
- 协议命名要能直接看出方向和作用域。
- 组件 Id 和 MsgId 要统一管理，避免临时乱填。

---

## 12. 从 0 开发一个新模块 / 新房间组件的实操步骤

前面讲的是“怎么判断该写哪一类”。这一节补的是“确定类型以后，具体怎么落地”。

### 12.1 开发一个全局模块的最小步骤

按下面顺序做最稳：

1. 先在 `Shared/Protocol` 定义协议。
2. 在 `Server/Modules` 新建服务端模块。
3. 在 `Client/Modules` 新建客户端模块。
4. 给服务端处理函数加 `[NetHandler]`。
5. 给客户端处理函数加 `[NetHandler]`。
6. 执行 `StellarNetLite/强制重新生成协议与组件常量表`。
7. 进运行时验证消息是否真能分发到模块。

最关键的判断点：

- 如果这个功能在“没进房间时也能调用”，它大概率就是全局模块。

### 12.2 开发一个房间组件的最小步骤

按下面顺序做：

1. 先在 `Shared/Protocol` 定义房间域协议。
2. 在服务端创建继承 `ServerRoomComponent` 的组件类。
3. 在客户端创建继承 `ClientRoomComponent` 的组件类。
4. 两端都加相同的 `[RoomComponent(同一个组件Id, 同一个组件名, ...)]`。
5. 服务端房间消息处理函数加 `[NetHandler]`。
6. 客户端房间消息处理函数加 `[NetHandler]`。
7. 如果服务端需要 Tick，再实现 `ITickableComponent`。
8. 把该组件 Id 加入房间模板或建房配置。
9. 执行 `StellarNetLite/强制重新生成协议与组件常量表`。
10. 进房后验证 `Room.GetComponent<你的组件>()` 是否拿得到。

最关键的判断点：

- 如果这条逻辑离开房间上下文就没有意义，它就应该是房间组件。

### 12.3 创建房间组件时最容易漏的 3 件事

1. 两端 `[RoomComponent]` 的组件 Id 不一致。
2. 只写了组件类，但没把组件 Id 加进房间模板。
3. 改完协议和组件后，没有重新生成常量表。

这 3 个问题是最常见的“代码看起来没错，但运行时就是不生效”原因。

### 12.4 建议你每次都做的验证

新增模块或组件后，至少做下面几项：

1. 检查 Unity Console 中没有装配错误。
2. 检查生成后的 `MsgIdConst` 是否包含你的协议。
3. 检查生成后的 `ComponentIdConst` 是否包含你的组件。
4. 发送消息后确认服务端 `[NetHandler]` 是否命中。
5. 房间场景里确认 `Room.GetComponent<你的房间组件>()` 不为 null。

### 12.5 一条很实用的经验

不要一上来就连 UI 一起写。

更推荐的顺序是：

1. 协议先通。
2. 服务端处理先通。
3. 客户端收包和状态更新先通。
4. 最后再接 UI。

这样你排查问题时能明确知道到底是：

- 协议没发出去
- 服务端没处理到
- 客户端没收到
- 还是 UI 根本没订阅到状态变化
