# StellarNet Lite
> 一个面向 Unity 中小型多人联机项目的轻量级房间网络框架  
> 核心理念：**服务端权威、协议事件驱动、房间组件化、回放沙盒化**
---
## 项目简介
**StellarNet Lite** 是一套面向 Unity 的轻量级房间式联网框架，适用于：
- 房间制
- 对局制
- 中小型商业项目
- 需要长期维护的多人联机玩法

它不追求“黑盒自动同步”，而是强调：
- **数据流转清晰可追踪**
- **服务端绝对权威**
- **客户端表现与业务彻底解耦**
- **按房间组件横向扩展玩法**
- **支持回放、重连、GC、配置化等生产级基础能力**

如果你正在开发这类项目：
- 房间对战
- 休闲联机
- 棋牌游戏 / 桌游
- 小型竞技
- 带录像与断线重连的房间游戏
- 需要多人协作长期迭代的中小型联网项目

那么 StellarNet Lite 会比“到处 SyncVar”“逻辑全堆一个房间类里”的方案更适合长期演进。

## 核心特性

### 1. 服务端绝对权威
客户端只负责：
- 采集输入
- 发送请求
- 接收权威同步
- 播放表现结果

所有核心业务状态都由服务端决定，例如：
- 房间状态
- 成员状态
- 血量与结算
- 游戏开始 / 结束
- 行为是否生效
- 是否允许重连
- 回放记录内容

一句话：
> **客户端可以有状态，但不能有权威。**

---
### 2. 显式协议事件驱动
框架拒绝黑盒自动同步，所有状态变化都必须通过清晰协议流转：
客户端请求 -> 服务端校验 -> 服务端修改状态 -> 服务端同步 -> 客户端刷新表现
这意味着：
- 谁发起的变化是清楚的
- 状态从哪里来是清楚的
- 问题排查链路是清楚的
- 更适合回放、重连、多人协作开发

---
### 3. 强类型发送器统一发包
业务层不再手拼 `Packet`，统一通过强类型入口发送：
- `ClientApp.SendMessage<T>()`
- `ServerApp.SendMessageToSession<T>()`
- `Room.BroadcastMessage<T>()`
- `Room.SendMessageTo<T>()`

这套发送链会自动处理：
- `MsgId`
- `Scope`
- `RoomId`
- `Seq`
- 协议方向校验
- 房间上下文校验

所以正式业务开发时，不再建议继续依赖魔数协议号和手工拼包。

---
### 4. 房间组件化扩展
每个房间玩法都可以拆成独立 `Room Component`：
- 房间设置组件
- 战斗组件
- 表情组件
- 聊天组件
- 投票组件
- 其他玩法组件

建房时通过 `ComponentIds` 动态装配。  
新增玩法时，不需要把所有逻辑塞进一个巨型房间类里。
核心收益是：
> **新增一个玩法，优先横向加一个组件，而不是纵向污染已有房间主逻辑。**

---
### 5. 客户端 Service / View 解耦
客户端房间组件只负责：
- 接收网络同步
- 做基础校验
- 转发事件 (无侵入式协议直抛)
- 维护轻量缓存

View 层只负责：
- UI 刷新
- 特效播放
- 插值移动
- 输入采集

这能显著降低：
- UI 与网络层耦合
- 房间切换残留状态
- 回放与在线逻辑串线

---
### 6. 回放沙盒模式
内置 `ReplayRoom` 概念。  
回放不是“在线看房间”，而是客户端本地重建出来的独立沙盒房间：
- 支持录像下载与播放
- 支持暂停、倍速、Seek
- 自动隔离真实在线房间消息
- 回放状态下禁止真实发包
- Seek 倒退采用“销毁重建 + 快进”保证状态纯净

---
### 7. 断线重连与状态恢复
框架支持：
- 登录态恢复
- 房间重连确认
- 客户端先装配组件再接收快照
- 服务端按房间上下文定向恢复

重点不是“自动帮你恢复一切”，而是：
> **重连流程走清晰的握手与快照恢复链路，而不是依赖黑盒状态复制。**

---
### 8. Seq 防重放机制
客户端发包自动附带递增 `Seq`。  
服务端按 Session 执行防重放拦截，避免：
- 重复点击
- 重复请求
- 旧包污染状态机
- 一般性重复消息重入

---
### 9. 两阶段装配与失败回滚
无论服务端还是客户端，房间组件装配都遵循统一原则：
1. 先全量校验
2. 再统一挂载与绑定
3. 最后统一初始化
4. 任一阶段失败整体回滚

这保证不会生成“半残房间实例”。

---
### 10. 生产级治理能力
除了联网基础功能外，框架还包含：
- 协议 ID 冲突扫描
- `NetMessageMapper` 启动期协议报告
- 房间生命周期 GC
- 空房间熔断清理
- 离线 Session GC
- 录像文件滚动清理
- 最低客户端版本拦截
- 编辑器配置窗口
- 业务脚手架生成器

## 设计目标
StellarNet Lite 更适合这些目标场景：
- **100~200 CCU** 量级的中小型多人项目
- 团队规模 **3~8 人**
- 强调 **长期可维护性**
- 希望业务模块可以 **横向独立扩展**
- 不希望项目后期演化成“网络层、房间层、表现层全混在一起”的状态

它不是为了替代大型 MMO 网络方案，也不是为了做超重型分布式后端。  
它的目标很明确：
> **用尽可能清晰、可维护、可扩展的方式，解决 Unity 房间制联网项目的大部分工程问题。**

## 架构概览

### 物理分层
- `Shared`：共享协议、基础结构、绑定器、配置、序列化抽象 (绝对纯净，无表现层事件)
- `Server`：权威逻辑、房间、会话、模块、组件、录制、GC
- `Client`：客户端状态机、房间组件、回放、表现桥接、事件系统 (GlobalTypeEvent / RoomEventSystem)
- `GameDemo`：示例玩法与演示场景

---
### 逻辑分层
#### Global
不依赖房间上下文的业务：
- 登录
- 大厅房间列表
- 建房 / 加房 / 离房
- 录像列表
- 下载录像

#### Room
依赖房间上下文的业务：
- 成员快照
- Ready 状态
- 游戏开始 / 结束
- 战斗同步
- 表情广播
- 局内聊天

---
### 标准数据流
```text
View 输入
-> ClientApp.SendMessage<T>()
-> ServerApp 路由
-> Server Module / RoomComponent 校验并修改权威状态
-> S2C 广播 / 单播
-> Client Module / ClientRoomComponent 收包
-> GlobalTypeEvent / Room.EventSystem 派发协议事件 (无侵入直抛)
-> View 监听协议事件并刷新表现
```

## 快速开始

### 1. 获取项目
```bash
git clone https://github.com/StarrDream/StellarNetLite.git
```

### 2. 用 Unity 打开工程
推荐环境：
- Unity 2021 LTS 或更高版本
- 已安装 Mirror
- 已安装 Newtonsoft.Json

### 3. 打开示例场景
运行带有以下脚本的测试场景：
- `StellarNetMirrorManager`
- `StellarNetDemoUI`
- `DemoGameView`
- `ServerAdminPanel`

### 4. 配置网络参数
打开菜单：
- `StellarNet/Lite 网络配置 (NetConfig)`

### 5. 启动模式
示例支持以下模式：
- `Host`：服务端 + 客户端同进程
- `Server Only`
- `Client Only`

### 6. 协议与组件常量表生成
新增协议或房间组件后，可执行：
- `StellarNet/Lite 强制重新生成协议与组件常量表`

## 典型开发流程

### 扩展一个房间玩法
以“房间发表情”功能为例，标准步骤如下：
1. 在 `Shared/Protocol` 中定义协议 (无需定义 Event 结构体)
2. 在 `Server/Components` 中编写服务端组件
3. 在 `Client/Components` 中编写客户端组件，收到协议后直接 `Room.EventSystem.Broadcast(msg)`
4. 在 `StellarNetMirrorManager` 中注册组件工厂
5. 建房时把对应 `ComponentId` 加入组件清单
6. 在 View 层注册监听 `S2C_` 协议并做表现

这套流程的核心收益是：
> **新玩法可以横向新增，不需要破坏已有房间逻辑结构。**

### 扩展一个大厅功能
例如新增“大厅邮件”或“全局公告”：
1. 在 `Shared/Protocol` 中定义 `Global` 协议
2. 在 `Server/Modules` 中实现对应全局模块
3. 在 `Client/Modules` 中实现接收，直接 `GlobalTypeEvent.Broadcast(msg)`
4. View 层使用链式调用监听协议并刷新 UI

### 使用脚手架生成模板
打开菜单：
- `StellarNet/Lite 业务脚手架 (Scaffold)`
  可以一键生成符合现行架构口径的业务骨架。

## 为什么不用 SyncVar
因为多人项目真正难的不是“同步一个值”，而是：
- 状态是谁改的
- 改动是否合法
- 回放怎么重建
- 重连怎么恢复
- 房间怎么隔离
- 问题出现时能不能追到完整链路

StellarNet Lite 选择显式协议驱动，正是为了保证：
- 权威边界清晰
- 数据流向清晰
- 团队协作清晰
- 回放与重连逻辑可控

---
## 为什么回放必须是沙盒
因为如果回放和在线房间共用一个上下文，会出现非常严重的问题：
- 正在看录像时收到真实房间包
- 回放状态被在线同步覆盖
- 玩家在回放里还能继续操作在线房间
- Seek 倒退后状态无法可靠重建

所以框架明确把回放定义为：
> **客户端本地重建的独立房间沙盒。**

这个项目更适合：
- 注重工程结构的 Unity 开发者
- 想做房间制联网游戏的独立团队
- 希望业务模块横向扩展的主程
- 需要一套可扩展、可维护、可讲清楚的联机底座

## 结语
StellarNet Lite 并不试图成为“最大的 Unity 网络框架”。  
它更关心的是另一件事：
> **当项目从 Demo 进入商业化迭代后，你的网络代码还能不能继续稳定扩展下去。**

如果你也认同这些原则：
- 不信任客户端
- 不迷信黑盒自动同步
- 不接受房间逻辑写成巨石类
- 希望回放、重连、排障都讲得清楚

那么这套框架就是为这种项目准备的。
