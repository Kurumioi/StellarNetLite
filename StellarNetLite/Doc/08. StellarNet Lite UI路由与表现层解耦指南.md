# 08. StellarNet Lite UI路由与表现层解耦指南

本文档说明为什么当前仓库不把业务逻辑直接写进 UI，以及 `SocialDemo` 的表现层是怎样接到底层房间模型上的。

---

## 1. 当前推荐的表现层分层

当前推荐的客户端分层是：

1. `ClientApp`
2. `ClientModule / ClientRoomComponent`
3. `GlobalTypeNetEvent / RoomNetEventSystem`
4. `Router / View / UIPanel`

职责划分如下：

- `ClientApp`
  负责运行时状态和消息收发。
- `ClientModule / ClientRoomComponent`
  负责网络数据缓存和状态更新。
- 事件系统
  负责把底层状态变化广播给上层。
- `Router / View / UIPanel`
  负责场景表现和交互。

---

## 2. 为什么不直接在 UI 里写业务逻辑

如果直接在 UI 中写网络业务逻辑，会带来几个问题：

- UI 和底层协议耦合。
- 回放模式和在线模式难以复用。
- 房间切换时很难保证生命周期干净。
- 调试时分不清是逻辑问题还是表现问题。

当前仓库刻意把这几层拆开，就是为了让业务模型能脱离 UI 独立运行。

---

## 3. 当前使用的两套事件系统

### 3.1 全局事件系统

- `GlobalTypeNetEvent`

适合广播：

- 登录结果
- 大厅列表
- 下载进度
- 系统提示
- 弱网状态

### 3.2 房间事件系统

- `RoomNetEventSystem`

适合广播：

- 房间快照
- 成员加入离开
- 开局结算
- 房间内玩法同步
- 回放快照消费结果

---

## 4. `SocialDemo` 当前的表现层结构

核心入口有两个：

- `GlobalUIRouter.cs`
- `RoomViewManager.cs`

### 4.1 `GlobalUIRouter`

负责全局 UI 状态切换，例如：

- 登录成功后打开大厅。
- 重连成功后关闭大厅。
- 被踢下线后回到登录。
- 录像下载完成后进入回放面板。

### 4.2 `RoomViewManager`

负责根据房间能力动态装配表现层。

它会根据当前房间中是否存在下列组件，决定挂哪些 View 和 Router：

- `ClientObjectSyncComponent`
- `ClientRoomSettingsComponent`
- `ClientSocialRoomComponent`

---

## 5. 在线房间和回放房间的区别

当前表现层明确区分：

- 在线房间
- 回放房间

### 5.1 在线房间

在线房间会装配：

- 输入控制器
- 在线 UI Router
- 在线房间面板

### 5.2 回放房间

回放房间会装配：

- 对象生成器
- 回放 Router
- 回放面板

这样做的意义是：

- 同一套房间模型可以服务在线和回放。
- 表现层根据状态切换，而不是业务层写两套逻辑。

---

## 6. Router 应该做什么

Router 的职责很简单：

- 订阅事件。
- 决定开什么面板、关什么面板。
- 把 UI 事件转成上层操作。

Router 不应该承担：

- 底层网络状态缓存。
- 房间成员状态主存储。
- 对象同步主逻辑。

这些事情应该分别留在：

- `ClientModule`
- `ClientRoomComponent`
- `ClientObjectSyncComponent`

---

## 7. View 应该做什么

View 的职责是：

- 表示单个对象或单块界面。
- 读取模型状态并刷新显示。
- 处理局部交互。

例如：

- 房间成员项
- 聊天气泡
- 角色跟随 UI
- 面板按钮行为

View 不应该做：

- 直接操作 `ServerApp`
- 长期保存业务主状态
- 把复杂协议逻辑直接写进组件里

---

## 8. `Panel` 应该做什么

`UIPanel` 层通常负责：

- 管理面板控件引用。
- 绑定按钮点击事件。
- 订阅全局或房间事件。
- 组合多个 UI Item。

例如：

- `Panel_StellarNetLobby`
- `Panel_StellarNetRoom`
- `Panel_SocialRoomView`
- `Panel_StellarNetReplay`

---

## 9. 当前推荐的数据流

以房间成员列表刷新为例，推荐数据流如下：

1. 服务端房间组件广播房间快照。
2. 客户端房间组件接收房间快照并更新缓存。
3. 客户端房间组件通过 `RoomNetEventSystem` 广播。
4. Router 或 Panel 接收事件并刷新 UI。

这样写的好处是：

- 房间状态可脱离 UI 单独验证。
- UI 重开时可以直接读取最新缓存。
- 回放模式也能复用同一套显示逻辑。

---

## 10. 当前示例里的几个典型分工

### 10.1 `ClientRoomSettingsComponent`

负责：

- 缓存成员列表。
- 缓存房间名、人数、私有状态。
- 广播开局和结算事件。

### 10.2 `ClientSocialRoomComponent`

负责：

- 接收社交房间内的状态同步和气泡同步。
- 把数据喂给 `ClientObjectSyncComponent`。
- 把事件广播给房间表现层。

### 10.3 `SocialRoomInputController`

负责：

- 读取本地输入。
- 驱动本地角色表现。
- 向服务端发送移动、动作和气泡请求。

它是表现层和网络层之间的“输入桥”，不是底层业务状态机。

---

## 11. 开发建议

- UI 先订阅事件，再决定表现。
- 房间组件先维护状态，再广播变化。
- Router 管状态流转，Panel 管界面，Item 管局部显示。
- 在线模式和回放模式优先共用房间模型，不要复制业务逻辑。
