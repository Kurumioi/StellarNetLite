# 08. StellarNet Lite UI路由与表现层解耦指南

本文对应当前仓库中 `Samples/SocialDemo` 的表现层实现，重点说明以下几件事：

1. Runtime 为什么不直接操作 UI。
2. Global UI Router 和 Room UI Router 的职责边界。
3. 在线房间和回放房间为什么要拆成两套路由。
4. Demo 里各个 Panel、Router、View 是如何串起来的。

---

## 1. 为什么 Runtime 不直接开关 UI

当前框架的 Runtime 只负责三件事：

1. 维护连接状态与房间状态。
2. 分发网络协议和本地事件。
3. 驱动房间组件与回放组件。

UI 不属于 Runtime 职责，所以你在 Runtime 中看到的是：

- 全局事件总线：`GlobalTypeNetEvent`
- 房间事件总线：`ClientRoom.NetEventSystem`
- 房间进入优先监听：`S2C_RoomSetupResult`、`S2C_ReconnectResult`
- 房间离开优先监听：`S2C_LeaveRoomResult`，并配合状态跌落检测

而不是直接写：

```csharp
UIKit.OpenPanel<SomePanel>();
```

这样做的直接收益是：

- 网络层不依赖具体面板名。
- 在线房间和回放房间可以复用同一套底层房间组件。
- UI 切换逻辑集中在表现层，定位问题更快。

---

## 2. 当前仓库里的 UI 分层

当前 Demo 的表现层大致分为 4 层：

### 2.1 GlobalUIRouter

文件：

- `Samples/SocialDemo/Client/Infrastructure/GlobalUIRouter.cs`

职责：

- 监听全局事件。
- 决定登录页、大厅页、房间页、回放页、网络监控页的切换。
- 处理状态跌落后的 UI 回退。

当前它直接监听的内容包括：

- `S2C_LoginResult`
- `S2C_ReconnectResult`
- `S2C_KickOut`
- `S2C_DownloadReplayResult`
- `S2C_RoomSetupResult`
- `S2C_ReconnectResult`
- `S2C_LeaveRoomResult`

这意味着：

- 登录成功后进大厅，是 `GlobalUIRouter` 决定的。
- 离开房间后回大厅，也是 `GlobalUIRouter` 决定的。
- 下载回放成功后打开回放页，仍然是 `GlobalUIRouter` 决定的。

### 2.2 RoomViewManager

文件：

- `Samples/SocialDemo/Client/Infrastructure/RoomViewManager.cs`

职责：

- 在收到稳定进房成功信号后，为当前房间装配表现层对象。
- 根据房间组件能力，决定要挂哪些 View 和 Router。

它负责把 Runtime 里的：

- `ClientRoomSettingsComponent`
- `ClientObjectSyncComponent`
- `ClientSocialRoomComponent`

映射成 Demo 表现层里的：

- `ObjectSpawnerView`
- `ClientRoomSettingsOnlineUIRouter`
- `ClientRoomSettingsReplayUIRouter`
- `SocialOnlineUIRouter`
- `SocialReplayUIRouter`

### 2.3 ClientRoomUIRouterBase<T>

文件：

- `Samples/SocialDemo/Client/Infrastructure/Routers/ClientRoomUIRouterBase.cs`

职责：

- 作为“房间组件 -> UI 行为”的桥接基类。
- 绑定某个 `ClientRoomComponent` 后，在 `OnBind` / `OnUnbind` 中控制面板。

这层的价值在于：

- 面板逻辑不直接写进 Runtime 组件。
- 在线和回放可以针对同一组件给出不同 UI 行为。

### 2.4 UIPanelBase 与具体 Panel

文件：

- `Samples/SocialDemo/Client/View/UIKit/UIPanelBase.cs`
- `Samples/SocialDemo/Client/View/UIPanel/...`

职责：

- `UIPanelBase` 提供面板统一生命周期。
- 具体 Panel 只做展示、按钮绑定、输入收集。

例如：

- `Panel_StellarNetLogin` 负责登录输入和重连确认 UI。
- `Panel_StellarNetLobby` 负责大厅房间列表、录像列表、玩家列表、全局聊天。
- `Panel_StellarNetRoom` 负责准备阶段成员列表和开始按钮。
- `Panel_SocialRoomView` 负责对局内聊天气泡和结束按钮。
- `Panel_StellarNetReplay` 负责录像播放控制。

## 3. 全局 UI 路由如何工作

以登录成功为例，当前链路是：

1. `ClientUserModule.OnS2C_LoginResult` 收到协议。
2. 模块把协议抛到 `GlobalTypeNetEvent`。
3. `GlobalUIRouter.OnLoginResult` 监听到后切换面板。
4. `UIKit` 负责真正打开/关闭 Panel。

也就是说：

- 模块只负责“告诉外部发生了什么”。
- Router 负责“界面该怎么切”。
- Panel 负责“打开后怎么显示”。

这个分层要尽量保持，不要把三层职责混在一起。

---

## 4. 房间 UI 路由如何工作

房间路由和全局路由最大的区别是：

- 全局路由关心客户端状态。
- 房间路由关心某个房间组件是否存在，以及该组件在当前房间态下应该打开什么 UI。

当前 Demo 中有两组典型路由。

### 4.1 房间设置路由

文件：

- `Samples/SocialDemo/Client/Infrastructure/Routers/ClientRoomSettingsOnlineUIRouter.cs`
- `Samples/SocialDemo/Client/Infrastructure/Routers/ClientRoomSettingsReplayUIRouter.cs`

含义：

- 在线房间中，`ClientRoomSettingsComponent` 需要驱动准备页和结算页。
- 回放房间中，同一个组件只需要只读状态，不需要准备交互。

### 4.2 Social 房间路由

文件：

- `Samples/SocialDemo/Client/Infrastructure/Routers/SocialOnlineUIRouter.cs`
- `Samples/SocialDemo/Client/Infrastructure/Routers/SocialReplayUIRouter.cs`

含义：

- 在线房间里，游戏开始后要打开 `Panel_SocialRoomView`。
- 回放房间里，不应打开在线输入面板。

这就是“同一底层组件，在线和回放分两套路由”的实际落点。

## 5. 为什么在线和回放必须拆路由

当前代码中，`ClientReplayPlayer` 会创建一个 `ReplayRoom`，然后把录像消息重新分发给客户端房间组件。

这意味着：

- 回放房间底层也会驱动 `ClientRoomSettingsComponent`
- 回放房间底层也会驱动 `ClientSocialRoomComponent`
- 回放房间底层也会驱动 `ClientObjectSyncComponent`

如果你让组件自己直接开 UI，就会出现两个问题：

1. 回放时误打开在线交互按钮。
2. 表现层逻辑越来越难区分“在线态”和“回放态”。

所以当前做法是：

- 在线路由只在 `ClientAppState.OnlineRoom` 下装配。
- 回放路由只在 `ClientAppState.ReplayRoom` 下装配。

这样运行态就不会串线。

## 6. Panel 里允许做什么，不允许做什么

推荐保持下面这个边界：

### 允许

- 绑定按钮事件。
- 读取输入框内容。
- 调用 `NetClient.Send(...)` 发送业务协议。
- 订阅 `GlobalTypeNetEvent` 或 `Room.NetEventSystem`。
- 把协议数据转成文本、列表、条目 UI。

### 不建议

- 在 Panel 里自己维护房间状态机。
- 在 Panel 里直接创建房间组件。
- 在 Panel 里判断在线房间和回放房间的底层构建流程。
- 在多个 Panel 中复制同一套状态切换逻辑。

状态切换优先交给 Router，数据展示交给 Panel。

## 7. 当前仓库里的一个完整示例

以进入在线房间为例：

1. `ClientRoomModule` 收到 `S2C_CreateRoomResult` 或 `S2C_JoinRoomResult`。
2. `ClientApp.EnterOnlineRoom` 创建 `ClientRoom`。
3. `ClientRoomFactory.BuildComponents` 装配客户端房间组件。
4. 广播 `S2C_RoomSetupResult` 成功结果。
5. `RoomViewManager` 根据组件能力装配 View 和 Router。
6. `ClientRoomSettingsOnlineUIRouter` 打开 `Panel_StellarNetRoom`。
7. 如果后续收到 `S2C_GameStarted`，`SocialOnlineUIRouter` 打开 `Panel_SocialRoomView`。

以进入回放为例：

1. `Panel_StellarNetReplay` 创建 `ClientReplayPlayer`。
2. `ClientReplayPlayer.StartReplay` 进入 `ReplayRoom`。
3. 同样构建 `ClientRoom` 和组件。
4. 进入 `ReplayRoom` 后装配回放表现层。
5. `RoomViewManager` 装配回放专用 Router。
6. 回放 UI 只显示只读内容，不打开在线交互页。

## 8. 写新业务时的落地建议

如果你新增一个房间组件，例如 `ClientXXXComponent`，建议按下面顺序扩展：

1. 先补 Runtime 组件和协议。
2. 再决定这个组件是否需要表现层。
3. 如果需要表现层，新增：
   - `XXXOnlineUIRouter`
   - `XXXReplayUIRouter`
4. 最后在 `RoomViewManager` 中装配这两个路由。

这样扩展不会污染 Runtime，也不会把回放链打穿。

---

## 9. 初学者最容易踩的 4 个坑

1. 在 Runtime 模块里直接 `UIKit.OpenPanel`。
2. 在线房间和回放房间复用同一个交互面板。
3. 事件注册后不跟生命周期解绑。
4. 在 Panel 中自己推导复杂状态，而不是监听现成事件。

只要把 Router 作为表现层的唯一入口，这 4 类问题会少很多。
